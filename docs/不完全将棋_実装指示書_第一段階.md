## 0. この文書の使い方

本指示書は、不完全将棋という同時着手型の将棋変種を、複数段階に分けて実装していく計画のうち、**第一段階のみ**を定義する。第一段階のゴールは次の三点に集約される。

1. 言語非依存のルールを、Rust の純粋なライブラリクレートとして実装する。
2. 今日の設計で確定した具体例を、回帰テスト群として写し取り、全て通す。
3. 一人がコマンドラインで両者の着手を入力し、一局を最後まで進められる検証用 CLI を載せる。

第二段階以降（Static Web App、通信対戦、CPU 対戦）は本指示書の範囲外だが、第一段階の設計がそれらに対して手戻りを生まないよう、**前方互換の制約**を随所に明記する。実装しないが、設計上は考慮する、という箇所を取り違えないこと。

## 1. プロジェクト全体像と本段階の位置づけ

### 1.1 不完全将棋とは（要約）

伝統的な将棋を基底とし、**隠すのは盤面ではなく「次の一手」**とする変種。盤上の駒・持ち駒はすべて両者に公開される。各ターン、両者は互いの着手を知らないまま自分の着手を確定し、同時に開示する。手番という概念が無く、両者は毎ターン必ず同時に着手する。これにより先手・後手の非対称が構造的に消える。詳細は仕様書を参照。

### 1.2 段階計画（全体像、参考）

- **第一段階（本指示書）**: Rust によるルールエンジン + 検証用 CLI + テスト群。通信は無し（一人が両者の手を入力する検証モード）。
- 第二段階以降の見通し（実装しない、設計の方向性として記載）:
  - 対人 → CPU の順で対戦相手を拡張。
  - CLI → Static Web App（同一エンジンを `wasm-bindgen` で Wasm 化して呼ぶ）→ スマートフォンアプリ（理想像）。
  - 平文通信 → コミットメント方式（commit-reveal）による秘匿同時着手 → 盤面ハッシュ相互検証 → Ack による進行同期 → 中断救済。
  - これらの通信・暗号・救済はすべて、ルールエンジンの**外側**に交換可能な殻として積む。エンジンには一切手を触れない。

### 1.3 本段階で「実装する」もの

- ルールエンジン（合法手生成・同時着手の三段階解決・衝突解決・終了判定）。
- 棋譜と局面の直列化（第二段階のハッシュ化に備えた前方互換の形まで）。
- 検証用 CLI（平文・単一プロセス・一人が両者を操作）。
- テスト群（後述の具体例を網羅）。

### 1.4 本段階で「実装しない」もの（非ゴール、第11章で再掲）

通信・ソケット・コミットメント方式・Ack・ハッシュによる相互検証や中断救済・本人認証、CPU/AI 対戦、Web/Wasm、二プロセス間の秘匿。これらに踏み込まないこと。

## 2. 設計哲学（最優先で保つこと）

今日の設計セッションを貫く哲学があり、これはコードの構造にもそのまま適用される。**迷ったらこの哲学に従う。**

### 2.1 共通の核と交換可能な殻

中心に「現局面と両者の着手から次局面を返す純粋関数」としてのルールエンジンを置く。その周囲を、棋譜（局面再現・千日手判定・中断救済の共通の源泉）が取り巻く。さらに外側に、通信・暗号・救済を担うプロトコル層。最も外に、CLI・Web・AI といった交換可能な入出力の殻。**エンジンは I/O・非同期・乱数・暗号・ネットワークを一切知らない。** これらは全て殻の責務である。

### 2.2 三つの方針（設計判断の指針）

1. **損得勘定の維持** — 従来の将棋の駒の損得感覚を可能な限り保つ。
2. **先後の公平性** — 手番を廃した同時着手で構造的に担保（解決済み）。
3. **膠着打開への報酬** — 消極的な反復より積極策が報われる方向。

### 2.3 中核命題（衝突解決の設計目標）

- 衝突調停は**損得を変えず、結果が手に入る確実性だけを変える**装置である。
- 守るべき不変条件: **移動先に相手の駒が現に留まっていたなら、取りに来た駒の価値の高低によらず取得が成立する。**

### 2.4 時間の定義（実装の根幹）

移動は連続的な走行ではなく、**着手開始時点の盤面を基準に合法性を確定し、その後すべての駒が一斉に最終位置へ飛ぶ瞬間移動**である。干渉が生じるのは二つの瞬間のみ:

- **着手開始時点** — 各着手の合法性を判定する。
- **移動完了時点** — 最終マスでの相討ち・取得を判定する。

その間の連続的な時間は存在しない。走行表現（アニメーション等）は見かけにすぎず、ロジックは完全に離散的。この前提が、後述の解決アルゴリズムの正当性を支える。

## 3. 技術スタックと成果物の形

- **言語**: Rust（安定版、edition 2021 以降）。
- **構成**: Cargo ワークスペース。最低限、エンジンのライブラリクレートと CLI のバイナリクレートの二つに分割する。
- **エンジンクレートの依存方針**: 依存を極力持たない（dependency-light）。標準ライブラリ中心。乱数・ハッシュ・シリアライズ用の重いクレートをエンジン本体に入れない。直列化は手書きの正準形式で行う（後述）。`serde` 等を使う場合も、エンジンの中核ロジックがそれに縛られない形に保つ。
- **エンジンの禁止事項**: `std::io`、ファイル I/O、`async`、ネットワーク、乱数、時刻取得、暗号。これらはすべて CLI 側（殻）に置く。
- **前方互換**: エンジンの局面型は、第二段階で `wasm-bindgen` 経由でブラウザから呼べるよう、また正準直列化からハッシュを取れるよう設計する（実装は第一段階では直列化まで。ハッシュ計算と Wasm 化は第二段階）。

### 3.1 表記法の採用（重要）

着手と局面の表記は、既存標準である **USI（Universal Shogi Interface）**の記法を採用する。これは曖昧さがなく、パース可能で、将来既存の将棋ツールとの相互運用も見込める。

- **着手（移動）**: `7g7f`（7g から 7f へ）。成りは末尾に `+`、例 `2b3a+`。
- **着手（打ち）**: `P*5e`（持ち駒の歩を 5e に打つ）。駒文字は USI 準拠（後述）。
- **マス座標**: 筋は 1–9、段は a–i（USI 準拠。1 段目が a、9 段目が i）。内部表現は別途定めてよいが、I/O は USI 文字列で行う。

ただし不完全将棋は同時着手のため「手番」が無く、局面表現の SFEN における手番フィールドは意味を持たない。局面直列化はこの点を調整する（第7章）。

## 4. アーキテクチャとクレート構成

推奨するワークスペース構成（クレート名・モジュール名は提案であり、Rust の慣習に沿って調整してよい）:

```
fukanzen-shogi/                 # ワークスペースルート
├─ Cargo.toml                   # [workspace]
├─ engine/                      # 純粋なルールエンジン（ライブラリ）
│  ├─ Cargo.toml
│  └─ src/
│     ├─ lib.rs                 # 公開 API の集約
│     ├─ types.rs               # Side, PieceKind, Piece, Square, Action, ...
│     ├─ board.rs               # Board, Hand, Position（ハッシュ可能状態）
│     ├─ movegen.rs             # 合法手生成（= 伝統的将棋の合法手）
│     ├─ resolve.rs             # 同時着手の三段階解決 + 衝突解決
│     ├─ terminate.rs           # 終了判定（確定的詰み・玉の死・千日手検出）
│     ├─ kifu.rs                # 棋譜（初期局面 + 着手ペア列）
│     └─ serialize.rs           # USI/SFEN 入出力と正準直列化（ハッシュ前方互換）
└─ cli/                         # 検証用 CLI（バイナリ）
   ├─ Cargo.toml                # depends on engine
   └─ src/
      └─ main.rs                # 一人が両者を操作する検証モード
```

**境界の原則**: `engine` は `cli` を知らない。`cli` は `engine` の公開 API のみを叩く。エンジンの公開 API は概ね次のような純粋関数群になる（シグネチャは Rust 慣習で調整）:

- `legal_actions(position, side) -> Vec<Action>` — ある側の合法着手の全列挙。
- `resolve(position, sente_action, gote_action) -> Resolution` — 両着手から次局面と結果を返す。
- `status(position) -> Status` — 着手選択前に、いずれかの側が着手不能か等を判定。
- 棋譜・直列化のための関数群。

## 5. データモデル（精密に）

以下は型の意味と責務の定義。Rust の具体的な表現（enum か struct か、`u8` 詰めか等）は実装者判断でよいが、**意味は厳守**すること。

### 5.1 Side（陣営）

二つの陣営を区別する列挙。手番の優劣は無いが、駒の所有と盤の向き（各陣営の駒は相手方向を向く・成れる段が逆）を区別するために必要。慣例に従い `Sente`（先手側＝下手前）/ `Gote`（後手側＝上手前）と呼んでよいが、**「先後」は単なる陣営ラベルであり手番順を意味しない**ことをコメントで明示する。

### 5.2 PieceKind（駒種）

基本 8 種と成駒 6 種。基本種 + 成りフラグで表すか、14 種を直接列挙するかは実装判断。ただし**パターンマッチの網羅性をコンパイラに検査させられる形**を強く推奨（場合分けの取りこぼし防止は本プロジェクトの肝）。

- 基本: 歩(P)、香(L)、桂(N)、銀(S)、金(G)、角(B)、飛(R)、玉(K)。
- 成駒: と金(+P)、成香(+L)、成桂(+N)、成銀(+S)、馬(+B)、龍(+R)。
- 金と玉は成れない。
- 駒文字は USI 準拠（先手は大文字、後手は小文字、成りは `+` 前置）。

### 5.3 Piece（駒）

盤上の一駒。駒種・成り状態・所有陣営を持つ。

### 5.4 Square（マス）

9×9 盤上の位置。内部表現（0–80 のインデックス等）は自由。I/O は USI 座標（筋 1–9 × 段 a–i）。

### 5.5 Board（盤面）

9×9 の各マスに `Option<Piece>`。

### 5.6 Hand（持ち駒）

各陣営の持ち駒。打てる駒種（歩・香・桂・銀・金・角・飛）ごとの枚数。玉は持ち駒にならない。**取った駒は基本種に戻る**（成駒を取ったら成りが解除されて持ち駒になる）。

### 5.7 Position（局面 — ハッシュ可能状態）

**第二段階の盤面ハッシュ相互検証と中断救済が依存する、最重要の型。** 局面を一意に再現するのに必要十分な状態を漏れなく含むこと。具体的には:

- 盤面（Board）
- 双方の持ち駒（Hand × 2）
- 手数（move number、第何ターンか）

> 持ち駒は必須。同じ駒配置でも持ち駒が違えば別局面。手数も含める（中断救済で局面を時間軸上の一点として一意に指すため、また千日手判定の補助のため）。**この三要素の組が、後にハッシュ対象＝中断救済のチェックポイント＝千日手判定の鍵となる。** 手番フィールドは持たない（同時着手のため存在しない）。

### 5.8 Action（着手 — 一方の一手）

一方のプレイヤーの一手。次のいずれか:

- **移動（Move）**: 移動元 `from`、移動先 `to`、成りフラグ `promote`。
- **打ち（Drop）**: 駒種 `kind`、着地先 `to`。

**第一段階では「パス（着手しない）」は存在しない。** 各ターン、各陣営は必ず一つの合法着手を行う。合法着手が無い状態は確定的詰み（第6.4節）。

### 5.9 Ply（同時着手 — 一ターン分）

一ターンを構成する両陣営の着手ペア: `{ sente: Action, gote: Action }`。これがゲーム進行の最小単位。

### 5.10 Kifu（棋譜）

`{ initial_position: Position, plies: Vec<Ply> }`。

> **各プレイヤーは対局開始からの全 Ply 履歴を保持する。** 棋譜さえあれば初期局面からエンジンを回して任意の局面を決定的に再現できる。これが第二段階の中断救済（ハッシュが指す局面の復元）と、千日手判定（局面の反復回数の計数）の共通基盤になる。第一段階では単一プロセスなので一つの棋譜を保持すればよいが、この「棋譜が局面再現の唯一の源泉」という構造は最初から守ること。

## 6. ルールエンジンの中核ロジック（本体）

ここが第一段階の心臓部。以下を**仕様書の規則に忠実に**実装する。

### 6.1 合法手生成 ＝ 伝統的将棋の合法手（重要な明確化）

**最重要の設計上の気づき:** 不完全将棋の「ある側の合法着手」は、**伝統的な将棋でその側が手番だと仮定したときの合法手と完全に一致する**。すなわち:

> 着手開始時点の盤面を固定し（相手の駒も自分の駒もその位置のまま）、その側の手番とみなして、伝統的将棋のルールで合法手を列挙する。

この単一の規則が、仕様書が個別に述べていた次のすべてを包含する:

- **玉の自殺手禁止** — 玉は相手の利きのあるマスへ移動できない。
- **王手放置の禁止** — 自玉が相手の利きに晒されたままになる手は非合法（着手開始時点の盤面で、自分の手だけを適用した結果、自玉が取られる位置にあれば非合法）。王手された状態では、玉を逃がす・王手駒を取る・合駒する、のいずれかしか合法手に現れない。
- **ピン（飛び駒に貫かれた駒の移動制限）**、**玉が動いて開く王手**、等の伝統的将棋の合法性のすべて。

加えて、打ちの制約も伝統的将棋どおり実装する:

- **二歩**（同じ筋に自分の歩が二枚になる打ちは禁止）。
- **行き所のない駒**（一段目への歩・香・桂打ち、二段目への桂打ち等、成らねば動けなくなる打ちは禁止）。
- **打ち歩詰め**（歩を打って相手を詰ます手は禁止。※「詰み」概念に依存する。第6.5節と未確定事項を参照しつつ、当面は伝統的定義で実装）。
- 成った状態では打てない（持ち駒は常に基本種）。

> **実装上の含意（重要）:** 合法手生成は、既知でよく定義された伝統的将棋の合法手生成アルゴリズムをそのまま実装すればよい。不完全将棋に固有の新規性は合法手生成には無い。新規性は次の「同時着手の解決」にある。この事実は実装リスクを大きく下げる。既存の将棋エンジンの合法手生成の定石（利き計算、ピンの検出、玉の移動先からの自玉攻撃判定、王手回避手の列挙、各種打ち禁則）を踏襲すること。

### 6.2 同時着手の三段階解決

一ターンは次の三段階で解決する（仕様書 第2章）:

1. **着手選択** — 両陣営が、直前の解決後の局面（相手の駒の現在位置）に基づき、各自の合法着手を一つ選ぶ。第一段階の CLI では一人が両方を順に入力する。
2. **同時移動** — 両着手を同時に適用し、すべての駒を移動先（打ちは着地先）へ移す。
3. **衝突解決** — 移動後の最終マスで衝突を解決する（第6.3節）。

`resolve(position, sente_action, gote_action)` は、**両着手が既に合法であることを前提**（CLI 側で選択時に検証済み）としてよいが、防御的に `debug_assert` で再検査することを推奨。

### 6.3 衝突解決規則（精密な列挙）

仕様書 第4章。判定は**最終マスでのみ**行い、移動経路の途中マスは一切干渉しない。

> **構造的前提（コードで `assert` してよい不変条件）:** 各プレイヤーは 1 ターンに 1 着手のみを行うため、**同一マスの当事者となる駒は常に最大 2 枚**である。3 枚以上が関与する衝突、および「留まり駒＋複数の移動駒」は構造的に発生しない。したがって場合分けは、二つの着手の移動先（と移動元）の関係だけを見ればよい。

二つの着手を `As`（先手）, `Ag`（後手）とし、各々の移動先を `Ts`, `Tg`、移動元を `Fs`, `Fg`（打ちは移動元なし）とする。次の順で判定する:

1. **同一マスの相討ち（4.3）**: `Ts == Tg` のとき。両方の移動駒が相討ち。両駒は盤上から除かれ、**それぞれ相手の持ち駒となる**（交換）。
   - このとき当該マスは着手開始時点で空である（留まり駒があり得ないことは前提から従う）。

2. **相互移動＝スワップの相討ち（4.4）**: 1 に該当せず、両者が移動（打ちでない）で、かつ `Ts == Fg` かつ `Tg == Fs` のとき。互いに相手の旧位置へ向かう正面衝突。両駒は相討ち（交換）。両移動元は空になる。
   - （純粋な移動先判定ではすり抜けてしまうが、仕様で明示的に相討ちと定める。）

3. **独立解決**: 1 にも 2 にも該当しないとき。各着手を着手開始時点の盤面に対して独立に解決する。各着手 `Ax`（移動先 `Tx`）について:
   - `Tx` が着手開始時点で空 → 当該駒が `Tx` を占める。
   - `Tx` に相手の駒があり、**相手の着手がその駒を動かして退かした**（相手の移動元 = `Tx`、すなわち「逃げた」）場合（4.2）→ 取得は発生せず、当該駒が空いた `Tx` を占める。
   - `Tx` に相手の駒があり、それが**留まっている**（相手の着手が `Tx` を退かしていない、または相手が打ち・無関係な移動）場合（4.1 **取得**）→ その相手駒を取って自分の持ち駒とし、当該駒が `Tx` を占める。**取りに行った駒の価値の高低は問わない（守るべき不変条件）。**
   - `Tx` に自分の駒がある場合は起こり得ない（合法手なら自分の駒のマスは移動先に選べない）。
   - 移動元 `Fx` は空になる（打ちは移動元なし）。成りフラグがあれば当該駒を成らせる。

4. **盤面の構築と結果判定**: 上記に基づき次局面を作る。取られた駒・相討ちの駒を適切な持ち駒に加え、生存駒を移動先へ（成りを反映）、移動元を空にする。手数を 1 増やす。

> **相討みと取得は常に「交換」または「取得」であって、盤外消滅は存在しない**（これは将棋の根幹）。打った駒が相討ちになった場合も、その駒は相手の持ち駒となる（盤を経由せず相手の駒台へ移る形）。

### 6.4 終了判定

仕様書 第5章。次を判定する。

**玉の死（5.2）** — 解決の結果、いずれかの玉が取られた（相手に取得された、または相討ちで除かれた）場合。

- 一方の玉のみ取られた → その陣営の負け。
- 両玉が同時に取られた（例えば両玉が同一マスへ移動して相討ち、という稀なケース）→ 引き分け。
- **実装注意**: 玉は持ち駒にならない。玉が取られる事象は「持ち駒に加える」のではなく、終了フラグとして扱う。

**確定的詰み＝着手不能（5.1）** — ある陣営が、着手開始時点において合法着手を一つも持たない場合。`legal_actions(position, side)` が空のとき。

- これは前述のとおり**伝統的将棋の詰み（および極めて稀な手詰まり）と一致する**。`status(position)` を着手選択の前に評価し、いずれかの側が着手不能ならその陣営の負け（両者同時に着手不能なら引き分け、ただし通常の進行ではほぼ生じない）。

**投了（5.3）** — CLI のコマンドで発生。投了した陣営の負け。両者同時投了は引き分け（第一段階の単一プロセスでは通常起きないが、API としては対称に扱う）。

**評価順序**: 一ターンの流れは「(a) 着手選択前に両陣営の着手可能性を確認（着手不能なら確定的詰み）→ (b) 両者が合法着手を選択 → (c) 解決 → (d) 玉の死の判定 → 続行なら次ターンへ」。玉の死が生じたターンは、それ以降の判定を行わず終局する。

> **仕様から導かれる、必ずテストすべき帰結（実装を誤りやすい箇所）:**
>
> 1. **玉の逃げはそのターン必ず安全**: 玉が合法な移動先（着手開始時点で相手の利きの無いマス）へ移動した場合、そのマスへは相手のどの駒も到達できない（移動先＝利きであり、利きが無いマスには相手は移動できない）。よって**移動先での相討ちによる玉の死は起こり得ない**。玉が取られるのは、玉が留まったまま（＝玉以外の駒を動かした、または合駒の賭けに出た）、相手の駒が玉のマスへ到達したときに限られる。これは必ず「着手開始時点で玉が王手されていた」ことを意味する。
> 2. **玉の死が起きるのは王手回避の賭けに失敗したとき**: 王手された状態で、玉を安全マスへ逃がせば必ず助かる。取る・合駒で応じるのは賭けであり、相手の実際の着手が玉を取れば玉の死となる。
> 3. **走り駒の王手に対する合駒はそのターンの玉を救わない**: 経路非干渉（4.6）ゆえ、走り駒が合駒を素通りして玉まで到達できる。したがって合駒は「着手開始時点の盤面では王手を解除する合法手」ではあるが、相手が走り駒を玉まで進めれば玉の死を防げない。**ゆえに、合駒できる局面は確定的詰みではない（着手不能でない）が、玉の死は依然起こり得る**。確定的詰みが伝統的な寄せより成立しにくい、という仕様書 第7章の予見はここに由来する。

### 6.5 千日手の検出（限定的に実装）

仕様書では千日手の成立時の扱い（指し直しか引き分けか）は**未確定**（第7章）。第一段階では:

- **検出は実装する**: 同一局面（第5.7節の Position の三要素＝盤面＋双方の持ち駒＋…ただし手数は除いて比較）が一定回数（伝統では 4 回）出現したことを、棋譜から再現される局面履歴で計数して検出する。
  - 注意: 千日手判定では「手数」は同一性の比較対象から除く（同一の盤面・持ち駒が異なる手数で反復することを数えるため）。第5.7節のハッシュ用 Position（手数を含む）とは比較キーが異なる点に注意。局面の「内容ハッシュ（盤面＋持ち駒）」と「位置ハッシュ（＋手数）」を区別できる設計にしておくとよい。
- **成立時の扱いは暫定的に引き分け**とし、「これは仕様書 第7章で未確定。方針3（膠着打開の奨励）との整合は今後再検討」という旨をコードコメントに明記する。**未確定の規則を勝手に確定させない。** あくまで暫定・要再検討の印を残す。
- 連続王手の千日手・打ち歩詰めの厳密な読み替えは未確定事項（第13章）。第一段階では打ち歩詰めは伝統的定義で実装し、連続王手の千日手は実装しない（または暫定無効とする）。

## 7. 棋譜と直列化（第二段階への前方互換）

### 7.1 棋譜の保持と再現

棋譜 = 初期局面 + Ply 列（第5.10節）。初期局面から Ply を順に `resolve` していくことで任意の局面を決定的に再現できること。この「棋譜→局面再現」の決定性は、第二段階の中断救済の前提なので、第一段階から関数として用意する（例: `replay(kifu, up_to_ply) -> Position`）。

### 7.2 着手の表記（USI）

第3.1節のとおり。`legal_actions` が返す Action を USI 文字列へ、USI 文字列を Action へ、相互変換する関数を用意する。CLI の入出力はこの文字列で行う。

### 7.3 Ply の表記

一ターン（両者の着手ペア）を一行で表す形式を定める。推奨:

```
<手数>: <先手着手USI> | <後手着手USI>
```

例: `1: 7g7f | 3c3d`、`12: P*5e | 8h2b+`。棋譜ファイルはこの行の列＋ヘッダ（初期局面）からなるテキストとする。

### 7.4 局面の直列化と正準形式（ハッシュ前方互換）

第二段階の盤面ハッシュ相互検証のため、**局面を一意なバイト列へ直列化する正準形式**を第一段階で定義・実装する（ハッシュ計算自体は第二段階だが、直列化の形を今固める）。

- 基本は **SFEN** を踏襲する（盤面フィールド＋持ち駒フィールド）。ただし SFEN の「手番」フィールドは不完全将棋に存在しないため、**省略するか固定のセンチネルを置く**こと（どちらにするか決めてコメントで明記）。
- ハッシュ用の正準直列化には、**盤面・双方の持ち駒・手数**を含める（第5.7節）。千日手検出用の内容直列化には手数を含めない（第6.5節）。両者を別関数として用意する。
- 直列化は決定的（同一局面 → 必ず同一バイト列）であること。順序・空白・桁揃えを厳密に定義し、実装間で一致するようにする（将来の多言語実装での差分テストと、第二段階の相互検証のため）。

> 第一段階ではハッシュ関数（SHA-256 等）の選定・実装は不要。正準直列化が決定的に得られる状態まで作ればよい。ハッシュは第二段階で、この直列化結果に対して被せる。

## 8. CLI（第一段階の検証モード）

### 8.1 目的と性格

第一段階の CLI は対戦アプリではなく、**ルールエンジンの正しさを人手で確かめる検証ツール**である。秘匿性は無く、一人が両陣営の着手を順に入力する。「自分一人で両手を指して、ルールが仕様どおり動くか確かめる」モード。

### 8.2 表示

毎ターン、次を表示する:

- 盤面（ASCII の将棋盤。筋・段の座標付き。先手/後手の駒を大文字小文字や向きで区別）。
- 双方の持ち駒。
- 手数。

### 8.3 一ターンの流れ

1. 着手選択前に両陣営の着手可能性を確認。いずれかが着手不能なら確定的詰みを宣言して終局。
2. 先手の着手を入力（USI）。合法手集合に照合し、非合法なら**理由を添えて**差し戻し再入力（例: 「その手は自玉が取られる位置に残る（王手放置）ため指せません」「二歩です」「行き所のない駒です」）。
3. 後手の着手を同様に入力・検証。
4. 両着手を `resolve` で解決し、**解決結果を明示**（何がどのマスへ動いたか、取得が発生したか、相討ちが発生したか、逃げた駒があったか、玉が取られたか）。
5. 新しい盤面・持ち駒・手数を表示。
6. 終了判定（玉の死／確定的詰み／千日手）。終局ならその旨を宣言。

### 8.4 コマンド

着手入力に加え、検証を助ける補助コマンドを用意する（プレフィクスは任意、例 `:`）:

- `moves <side>` — 指定陣営の合法着手を一覧表示（検証の要。期待手が出るか、非合法手が除かれているかを目視確認できる）。
- `board` — 盤面を再表示。
- `undo` — 直前の Ply を取り消す（棋譜の末尾を削って再 replay）。
- `save <path>` / `load <path>` — 棋譜の保存・読み込み（第7.3節の形式）。
- `resign <side>` — 指定陣営の投了。
- `quit` — 終了。

### 8.5 完了局までの一貫動作

初期局面から、一人が両陣営の手を入力し続けて、**詰み（確定的詰み）・玉の死・投了のいずれか**で一局を完了できること。これが完了基準（第10章）の中核。

## 9. テスト群（回帰スイート — 具体的シナリオ）

今日の設計で確定した具体例を、そのまま回帰テストとして実装する。各テストは「初期配置（最小限の駒を置いた局面）＋両者の着手＋期待される解決結果」で記述する。**これらのテストは仕様そのものであり、第一段階の主役。** 可能な限りテストファーストで進めること。

### 9.1 衝突解決のテスト

1. **取得（価値非依存）**: 先手歩が、後手飛の留まるマスへ移動 → 歩が飛を取り、飛が先手の持ち駒へ。歩は飛のいたマスを占める。
2. **逃げた駒は取られない**: 先手飛が後手角のマス X へ移動しようとし、同時に後手角が X から別マス Y へ移動 → 取得は発生せず、先手飛は X を占め、後手角は Y にいる。
3. **空きマスの相討ち**: 先手・後手の駒が同一の空きマス Z へ移動 → 双方相討ち、各駒が相手の持ち駒へ。
4. **スワップの相討ち**: 先手駒 A が後手駒 B のマスへ、後手駒 B が A のマスへ同時移動 → 相討ち、交換。両移動元は空。
5. **経路すり抜け（Y の銀）**: 先手飛が横に走って遠方の空きマス X へ、同時に後手銀が飛の経路途中のマス Y へ移動 → 干渉せず、飛は X に到達、銀は Y に留まる。
6. **打ちの相討ち**: 先手が持ち駒の歩をマス Z へ打ち、同時に後手の駒が Z へ移動 → 相討ち。打った歩は後手の駒を取って先手持ち駒へ戻る扱い…ではなく、**打った歩は後手の持ち駒へ、後手の駒は先手の持ち駒へ**（双方向の交換）。正しい交換方向をテストで固定する。
7. **取得時の成り解除**: 成駒（例: 龍）を取った場合、持ち駒には基本種（飛）として加わる。

### 9.2 合法手生成のテスト

8. **玉は利きへ侵入できない**: 玉が相手の利きのあるマスへ移動する手が、合法手集合に**現れない**。
9. **王手放置は合法手に現れない**: 自玉が王手されている局面で、王手を解除しない手（無関係な駒の移動等）が合法手集合に現れない。王手を解除する手（玉の退避・王手駒の取得・合駒）のみが現れる。
10. **二歩・行き所のない駒・成駒打ち禁止**: これらの打ちが合法手集合から除かれる。
11. **打ち歩詰めの禁止**: 歩を打って相手を詰ます手が除かれる（伝統的定義で）。
12. **ピン・玉が動いて開く王手**: 飛び駒に貫かれた駒が自玉を晒す移動を行えない、玉が移動して背後の利きに入る手が除かれる、等。既知の将棋局面で検証。

### 9.3 終了判定のテスト

13. **確定的詰み（着手不能）**: 伝統的な詰み局面で `legal_actions` が空になり、その陣営の負けが宣言される。
14. **玉の逃げは安全（重要帰結）**: 王手された局面で玉が合法な退避マスへ移動 → そのターン玉は取られない（相手がどの着手をしても玉の死が起きないことを、相手の全合法着手について確認）。
15. **玉の死（王手回避の賭けの失敗）**: 後手飛に王手された先手が、玉を退避させず別の手（飛を取りに行く等）で応じ、同時に後手飛が先手玉のマスへ移動 → 先手玉が取られ、先手の負け。
16. **合駒は走り駒の王手からそのターンの玉を救わない（重要帰結）**: 走り駒の王手に対し合駒する手は合法（確定的詰みではない）が、相手が走り駒を玉まで進めると玉の死が生じることを確認。
17. **両玉同時の相討ち → 引き分け**: 両玉が同一マスへ移動して相討ちになる稀なケースで引き分けが宣言される。
18. **投了**: `resign` で当該陣営の負け。

### 9.4 棋譜・直列化のテスト

19. **往復変換**: Action ⇄ USI 文字列、Ply ⇄ 行表記の往復で不変。
20. **再現の決定性**: 棋譜を初期局面から replay した結果が、逐次解決した結果と一致。
21. **正準直列化の決定性**: 同一局面が常に同一バイト列に直列化される。異なる局面は異なる（最低限、テストケースの範囲で）。千日手用の内容直列化（手数を除く）とハッシュ用直列化（手数を含む）が、意図どおり区別される。

### 9.5 将来への布石（実装は任意）

- **差分テストの口**: 後の多言語実装と結果を照合できるよう、「局面＋ Ply 列を入力すると、各ターンの解決後局面の正準直列化を出力する」ような決定的な検証用エントリ（CLI サブコマンドや関数）があると、将来の C/Lisp/Perl 実装との差分テストに使える。第一段階では必須ではないが、用意できるなら歓迎。

## 12. 進め方の指針

Claude Code への作業順序の推奨:

1. **型の定義**（types/board）。Side・PieceKind・Piece・Square・Board・Hand・Position・Action・Ply・Kifu。網羅性をコンパイラに検査させられる形を優先。
2. **直列化の骨格**（serialize）。USI 着手の往復、SFEN ベースの局面入出力、正準直列化。早めに用意すると、以降のテストで局面を簡潔に記述できる。
3. **合法手生成**（movegen）。伝統的将棋の合法手として実装。既知局面に対する単体テストで、利き・ピン・王手回避・各種打ち禁則を固める。**ここを堅くするのが最優先**（不完全将棋の新規性はここには無いが、ここが正しくないと全体が崩れる）。
4. **同時着手の解決と衝突**（resolve）。第6.3節のアルゴリズムを実装し、第9.1節の具体例テストで一つずつ固める。テストファースト推奨（具体例が仕様そのもの）。
5. **終了判定**（terminate）。玉の死・確定的詰み・千日手検出。第9.3節の帰結テスト（玉の逃げの安全性、合駒の弱さ等）を必ず入れる。
6. **CLI**（cli/main）。検証モードを実装し、一局を最後まで通せることを確認。
7. **通し確認**。初期局面から実際に何局か指してみて、仕様の穴やバランスの違和感（特に千日手の起こりやすさ、確定的詰みの成立しにくさ）を観察し、所見を記録する（修正はせず、所見として残す。ルール変更は人間の判断事項）。

横断的な注意:

- **仕様からの逸脱が必要に見えたら、実装で勝手に解決せず、曖昧点として報告する。** 複数の解釈があり得る箇所を見つけたら、それは仕様 v0.1 の穴を炙り出したことを意味し得る（将来の多言語差分テストと同じ価値）。記録して確認を求める。
- コミットは論理単位で小さく刻む。各中核モジュールが対応テストとともに入る形が望ましい。
- パフォーマンス最適化は第一段階の主目的ではない。正しさと明瞭さを優先する（速度が要るのは将来の CPU 対戦・探索の段階）。

## 13. 既知の未確定事項（将来へ引き継ぐ印）

仕様書 第7章および本指示書で浮かんだ未確定事項。第一段階では確定させず、印として引き継ぐ:

- **千日手の成立時の扱い** — 同一局面（盤面＋双方の持ち駒）が 4 回で成立とするのが自然だが、成立時を指し直しとするか引き分けとするかは、先後が無いため未決定。方針3（膠着打開の奨励）との整合も含め再検討。第一段階は検出＋暫定引き分け＋印。
- **王手概念の再形式化** — 同時着手では「王手」が確率的になる。連続王手の千日手、打ち歩詰めの厳密な読み替えはこれに依存し未確定。第一段階は打ち歩詰めを伝統的定義で実装、連続王手の千日手は未実装。
- **確定的詰みの実現可能性** — 玉が着手開始時点で逃げ場を持ちやすく、かつ合駒が走り駒の王手を当ターンには防げないため、確定的詰みは伝統的な寄せより成立しにくい可能性。実戦観察の結果次第で、確率的詰みの導入や玉の制約の調整に立ち戻る余地（ルール変更は人間の判断）。
- **入玉宣言法の手番依存箇所** — 「宣言側の手番で」等、手番を前提とする手続きの読み替えが必要（第二段階以降）。
- **第二段階の本人認証の強度** — 再接続時の本人性を、素朴な共有秘密にするかハッシュコミット形式にするか（commit-reveal と同型の漏洩耐性版）。第一段階の範囲外だが、エンジン・棋譜の設計がこれを妨げないことだけ担保する。
