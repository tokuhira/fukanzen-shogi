# 不完全将棋 実装指示書 — 終局判定の改訂（ルール v0.6：引き分けの確定と最長手数）

> 対象実行者: Claude Code（Sonnet）
> 前提: 配布 v0.8.2 リリース済み。関連する現物:
> - `engine::terminate`: `GameStatus { Ongoing, SenteLoses, GoteLoses, Draw }`、`check_status(pos) -> GameStatus`（着手開始時点の着手不能＝確定的詰み・両者不能の引き分け）、`check_king_death(event: &ResolutionEvent) -> Option<GameEnd>`、`GameEnd { SenteLoses, GoteLoses, Draw }`、`check_sennichite(kifu: &Kifu) -> bool`（同一局面4回。「暫定引き分け」TODO 付き）。
> - `engine::kifu`: `Kifu { initial_position, plies: Vec<Ply> }`、`replay(up_to)`、`current()`。`plies.len()` がそのまま**組手数**（`move_number = 1 + 組手数`）。
> - `engine::archive`: `ResultKind { Mate, KingDeath, SwapDraw, Sennichite, Resign, Unfinished, Other }`、`Outcome { SenteWins, GoteWins, Draw, None }`（各 `to_str`/`from_str`）。`ARCHIVE_FORMAT_VERSION = 1`。
> - `engine::RULE_VERSION = (0, 5)`。`protocol::MY_VERSION` が rule＋protocol を束ね、`protocol-wasm::version_tuple()` が web へ渡す。
> - `engine-wasm`: `resolve_ply`・`game_status`（`check_status` 相当）・`legal_actions`・`build_archive`・`parse_archive`。v0.8.2 の安全網 `MAX_ARCHIVE_PLIES = 500`（Ply＝組手）を `parse_archive` に持ち、web 側にも同値 `MAX_ARCHIVE_PLIES = 500`＋`MAX_ARCHIVE_BYTES = 512KB` を「一致させること」コメント付きで二重に持つ。
> - `web/board.js`: `currentResult()` が終局結果を判定し archive に刻む。現状 mate/draw/king_death/resign を検出、千日手・両玉スワップは `other` に倒している。
> 関連文書: `不完全将棋_ルール仕様_v0.6.md`（本実装が体現するルール。特に 5.0・5.6・5.7・5.8・§6）、`不完全将棋_バージョン互換性管理_方針.md`、`不完全将棋_実装指示書_互換性確認プロトコルv0.6.0.md`。
> 性格: ルール v0.6 の終局系を engine（核）に実装する。核心は **(1) 最長手数＝500組手で引き分け（5.7）、(2) 千日手＝引き分けの確定（5.6、暫定の解消）、(3) 引き分けを含む終局を一元評価する権威関数（5.8）**。engine を終局判定の**単一の権威**とし、殻（web）はそれを呼ぶだけにする。**加算的**（既存の `check_status`/`check_king_death`/`check_sennichite` は壊さない）。**RULE_VERSION を 0.5 → 0.6**。これに伴い v0.8.2 の重複した安全網 500 も engine の単一値へ寄せて解消する。

- **位置づけ**: 第三歩（ライブ観戦・サーバ）の前に、ルールそのものを整える改訂。終局判定を engine に集約し、殻はそれに従う。

## 1. engine::terminate（核・単一の権威）

### 1.1 最長手数（5.7）

```rust
/// 最長手数の上限（組手）。500組手を終えて決着なければ引き分け（ルール v0.6 §5.7）。
/// 手番が無いため単位は個別着手ではなく組手。基底500手ルールの読み替え。
pub const MAX_TURNS: usize = 500;

/// 500組手に達しているか（＝これ以上は最長手数の引き分け）。
pub fn check_max_turns(kifu: &Kifu) -> bool {
    kifu.plies.len() >= MAX_TURNS
}
```

### 1.2 千日手の確定（5.6）

`check_sennichite` の**ロジックは変えない**（同一局面4回）。doc コメントから「暫定」「TODO: 指し直しか引き分けか要再検討」を外し、**引き分けで確定**（ルール v0.6 §5.6）と明記する。連続王手の千日手＝反則負けは v0.6 で廃止のため、王手の連続性は一切見ない（現状のまま）。

### 1.3 一元評価の権威関数（5.8）

盤上で導ける終局を、ルール §5 の順序で一度に判定する。**投了（5.3・同時投了 5.4）は盤面から導けないため本関数の対象外**——クライアントが別途注入する。

```rust
/// 盤上で導ける終局結果（投了を除く）。archive の (ResultKind, Outcome) へ対応づく。
#[derive(Debug, Clone, PartialEq, Eq)]
pub enum Terminal {
    Ongoing,
    Loss { loser: Side, kind: LossKind },   // 5.1 確定的詰み / 5.2 玉の死
    Draw { kind: DrawKind },                 // 5.4 両者成立 / 5.6 千日手 / 5.7 最長手数
}
#[derive(Debug, Clone, PartialEq, Eq)]
pub enum LossKind { Mate, KingDeath }
#[derive(Debug, Clone, PartialEq, Eq)]
pub enum DrawKind { MutualMate, BothKingsDied, Sennichite, MaxTurns }

/// 棋譜（履歴）から盤上の終局を評価する。
/// 直前の組手の解決イベントは kifu を再生して内部で取得する（呼び出し側で渡さなくてよい）。
/// 評価順序（ルール §5・§6.4）: 玉の死 → 確定的詰み/両者不能 → 千日手 → 最長手数 → 続行。
pub fn evaluate(kifu: &Kifu) -> Terminal {
    // 1. 直前の組手による玉の死（5.2）。plies が空なら初期局面なのでスキップ。
    if let Some(last) = kifu.plies.last() {
        let pre = kifu.replay(kifu.plies.len() - 1);
        let res = crate::resolve::resolve(&pre, last.sente, last.gote);
        match check_king_death(&res.event) {
            Some(GameEnd::SenteLoses) => return Terminal::Loss { loser: Side::Sente, kind: LossKind::KingDeath },
            Some(GameEnd::GoteLoses)  => return Terminal::Loss { loser: Side::Gote,  kind: LossKind::KingDeath },
            Some(GameEnd::Draw)       => return Terminal::Draw { kind: DrawKind::BothKingsDied },
            None => {}
        }
    }
    // 2. 現局面での着手不能（5.1 / 両者不能 5.4）。
    match check_status(&kifu.current()) {
        GameStatus::SenteLoses => return Terminal::Loss { loser: Side::Sente, kind: LossKind::Mate },
        GameStatus::GoteLoses  => return Terminal::Loss { loser: Side::Gote,  kind: LossKind::Mate },
        GameStatus::Draw       => return Terminal::Draw { kind: DrawKind::MutualMate },
        GameStatus::Ongoing    => {}
    }
    // 3. 千日手（5.6）。
    if check_sennichite(kifu) { return Terminal::Draw { kind: DrawKind::Sennichite }; }
    // 4. 最長手数（5.7）。
    if check_max_turns(kifu) { return Terminal::Draw { kind: DrawKind::MaxTurns }; }
    Terminal::Ongoing
}
```

- 決定的な結果（玉の死・確定的詰み）が引き分け（千日手・最長手数）に優先する。500組手目が同時に詰み/玉の死なら、決着が勝つ。
- 千日手と最長手数が同一組手で両成立する場合は、より具体的な**千日手**を先に返す（上の順序どおり）。
- `evaluate` が終局判定の**単一の源泉**。各クライアントが順序をばらばらに実装しない。

## 2. engine::archive：ResultKind に MaxTurns を追加

- `ResultKind` に `MaxTurns` を追加。`to_str` → `"max_turns"`、`from_str("max_turns")` → `Some(MaxTurns)`。往復テストの kind 網羅リストに加える。
- `Terminal` → `(ResultKind, Outcome)` の対応（クライアント／`evaluate_terminal` が使う）:
  - `Loss{Sente, Mate}` → `(Mate, GoteWins)` ／ `Loss{Gote, Mate}` → `(Mate, SenteWins)`
  - `Loss{Sente, KingDeath}` → `(KingDeath, GoteWins)` ／ `Loss{Gote, KingDeath}` → `(KingDeath, SenteWins)`
  - `Draw{MutualMate}` → `(Mate, Draw)`
  - `Draw{BothKingsDied}` → `(SwapDraw, Draw)`（両玉同時取りの主経路。既存 kind を流用）
  - `Draw{Sennichite}` → `(Sennichite, Draw)`
  - `Draw{MaxTurns}` → `(MaxTurns, Draw)`
  - `Ongoing` → 未終局なら `(Unfinished, None)`
  - 投了（クライアント注入）→ `(Resign, 勝者)`
- **書式版は 1 のまま**。`MaxTurns` は enum 変種の追加であり、書式レイアウトは不変。旧版エンジンが `max_turns` を含む棋譜を読むと `from_str` が `None` を返し parse 失敗しうるが、これはルール版 0.6 の棋譜を 0.5 エンジンが読む版スキューであり、第二歩の**版不一致注意**が働く正常系（RULE_VERSION が 0.6 に上がるため）。

## 3. RULE_VERSION と版交渉

- `engine::RULE_VERSION` を **`(0, 6)`** に。これで `protocol::MY_VERSION`・`version_tuple()`・archive のヘッダ・接続時の版交渉に自動伝播する。
- **対戦互換の境界**: ルール 0.6 は 0.5 と終局規則が異なる（0.6 は500組手・千日手で引き分け、0.5 は終局しない）。オンライン対局で 0.6 と 0.5 が混ざると終局判定が食い違う。したがって**版交渉はルール版不一致を対戦非互換として弾く**必要がある。既存 `negotiate` がそうなっているか確認し、なっていなければ本改訂で担保する（プロトコル版 2 は不変＝ワイヤは変えないが、ルール版一致を対戦の前提とする）。
- **プロトコル版・アーカイブ書式版は据え置き**（それぞれ 2・1）。変わったのはルールの意味のみ。

## 4. engine-wasm：評価の露出と、安全網の一本化

### 4.1 `evaluate_terminal` の露出

```rust
/// 棋譜（初期局面＋着手列）から盤上の終局を評価して JSON で返す。
/// request 例（build_archive のリクエストから meta を除いた形）:
///   {"initial_sfen":"...", "plies":[{"s":"7g7f","g":"3c3d"}, ...]}
/// 返値例:
///   {"status":"ongoing"}
///   {"status":"loss","loser":"sente","kind":"mate"}
///   {"status":"draw","kind":"max_turns"}
/// これに対応する (ResultKind, Outcome) は §2 の表のとおり（web が archive 刻印に使う）。
#[wasm_bindgen]
pub fn evaluate_terminal(request_json: &str) -> String;
```

- 実装は `build_archive` と同じ流儀で initial_sfen＋plies から `Kifu` を構成し、`engine::terminate::evaluate(&kifu)` を呼んで JSON 化。

### 4.2 安全網を単一値へ寄せる（v0.8.2 の重複解消）

- `engine-wasm` の `MAX_ARCHIVE_PLIES = 500` を、**`engine::terminate::MAX_TURNS` を参照**するよう変更（`use engine::terminate::MAX_TURNS;`）。ルールが500組手で引き分けと定めた以上、正当な棋譜は500組手を超えず、安全網はルールの真の境界と一致する。ハードコードの重複を消す。
- web 側が 500 を必要とする箇所（バイト上限計算・エラーメッセージ）は、engine-wasm から値を読む getter（例 `pub fn max_turns() -> usize`）を足して JS がそれを使う。JS 内のハードコード `MAX_ARCHIVE_PLIES = 500` と「一致させること」コメントを撤去する（核が自己申告する値、ハードコードしない＝第一歩からの原則）。

## 5. web：権威関数への載せ替えと引き分け終局

### 5.1 結果刻印の正確化（`currentResult` の載せ替え）

- 終局結果の判定を、ad hoc なロジックから **`evaluate_terminal`（§4.1）＋ §2 の対応表**に載せ替える。これで千日手・両玉同時取り・最長手数が `other` 倒しにならず、正しい kind で archive に刻まれる。
- 投了だけは盤面から導けないので、従来どおりクライアントが `(Resign, 勝者)` を注入する（online の投了フローはそのまま）。

### 5.2 引き分け終局の実施

- 各組手の解決後（ホットシート再生・オンライン対局の双方）、`evaluate_terminal` が `draw`（`max_turns`／`sennichite`／`mutual_*`）または `loss` を返したら**対局を終局させる**。従来 mate/king_death/resign で終局していた導線に、**引き分け（千日手・最長手数）を加える**。
- オンライン: 両者は commit-reveal で同一の `plies` を共有するため、各自が `evaluate_terminal` で**独立に同じ引き分けを検出**して一致終局できる。既存の `endOnlineGame` に引き分け理由（`max_turns`／`sennichite`）を受けられるよう最小拡張する。新しいプロトコルメッセージは原則不要（決定的に一致するため）。
- 500組手に達した対局は 5.7 により引き分け終局し、その棋譜は500組手ちょうどで閉じる（§4.2 の安全網と整合）。

### 5.3 バージョン表記

- `index.html`/`board.js`/`style.css` の `?v=` を新版へ。読み込んだ棋譜の版タプルに rule `0.6` が出ること（第二歩の鑑賞表示）。

## 6. テストと受け入れ基準

### 6.1 engine（必須）

- `check_max_turns`: 499組手の kifu → `false`、500組手 → `true`。
- `evaluate` の順序: (a) 最後の組手が玉の死 → `Loss{..,KingDeath}`（後続の千日手/最長手数より優先）、(b) 現局面が着手不能 → `Loss{..,Mate}`、両者不能 → `Draw{MutualMate}`、(c) 同一局面4回 → `Draw{Sennichite}`、(d) 500組手で無決着 → `Draw{MaxTurns}`、(e) それ以外 → `Ongoing`。
- `check_sennichite` は従来テストが不変で通る（4回検出）。
- 500組手ちょうどで無決着の合成 kifu → `Draw{MaxTurns}`。501組手は evaluate 以前に安全網（§4.2）が拒否する経路も確認（web/parse 経由）。

### 6.2 engine-wasm（必須）

- `evaluate_terminal`: 代表 kifu で `{"status":"draw","kind":"max_turns"}` 等が返る。`build_archive`→`evaluate_terminal` の整合。
- 安全網が `engine::terminate::MAX_TURNS` を参照している（コードレビュー）。`max_turns()` getter が 500 を返す。

### 6.3 web（受け入れ・手動）

- 500組手に到達する対局（合成・またはデモ）→ **引き分けで終局**し、archive に `result max_turns draw` が刻まれる。
- 同一局面4回の対局 → **千日手引き分けで終局**し、archive に `result sennichite draw`（従来の `other` ではない）。
- 版タプルに rule `0.6` が出る。ルール 0.5 の旧棋譜を読むと第二歩の版不一致注意が出る。
- JS 内に重複した 500 のハードコードが残っていない。
- デモ・オンライン・新局など既存機能が回帰しない。

## 7. 版の刻み

- **配布版 v0.8.2 → v0.9.0**（マイナー）。理由: これはルールセットの改訂であり、合法な対局の定義そのもの（何をもって終局・引き分けとするか）を変え、対戦互換の境界（0.6 と 0.5 は混じれない）を動かす。地続きの前進（パッチ）を超える「出来事」——ゲームそのものの変更——ゆえマイナーが似合う。RULE_VERSION 自身のマイナー刻み（0.5→0.6）とも平仄が合う。
- **RULE_VERSION 0.5 → 0.6**。**プロトコル版は 2 のまま**（ワイヤ不変）、**アーカイブ書式版は 1 のまま**（`MaxTurns` は enum 追加でレイアウト不変）。

## 8. 非ゴールと次段への申し送り

- **入玉・持将棋・入玉宣言法**（ルール v0.6 5.9）: 点数法と宣言手続き（手番前提）の読み替え。千日手・最長手数の backstop が立っているため別層で後日。整備時も無勝負・指し直しは引き分けへ写す（5.0）。
- **引き分けの非対称化**（方針3）: 反復・最長手数到達を能動的に罰する仕組み。先後が無く「責」を定義しにくい。プレイテストで引き分けへの逃げ込みが問題化したら検討。
- **オンライン終局の演出・第三歩**: 引き分け終局の豊かな UX、ライブ観戦での終局提示。ここで秘匿境界（淀川 §4）も実物で決める。
- **CLI/TUI の載せ替え**: engine の終局判定関数は使えるようになる。UI 追随は Web 盤の概念が固まるまで凍結（作り手の判断）。
- **玉の死の残存経路・両玉隣接・大駒の強さ・確定的詰みの実現性**: いずれもプレイテスト観察対象（ルール v0.6 §7）。データが出たら次版で手当て。

## 9. 不変の原則（本実装が守るもの）

1. **終局判定は engine が単一の権威**: 盤上で導ける終局は `evaluate` 一つに集約し、殻はそれを呼ぶ。順序をクライアントごとに再実装しない。
2. **引き分けは指し直しの像**: 手番消滅により基底の指し直しは引き分けへ写る（ルール v0.6 5.0）。実装もそれを一級の結果として扱う。
3. **加算的**: 既存の `check_status`/`check_king_death`/`check_sennichite` は壊さず、`check_max_turns`・`evaluate`・`ResultKind::MaxTurns` を足す。
4. **核が値を自己申告する**: `MAX_TURNS` は engine の単一値。安全網も web もそれを参照し、ハードコードの重複を残さない（v0.8.2 の緩みをここで解消）。
5. **ワイヤと書式は変えない**: 変わるのはルールの意味のみ。プロトコル版 2・書式版 1 は据え置き、ルール版のみ 0.6 へ。
6. **設計は全体、実装は最小から**: 入玉・非対称・オンライン演出は据え置き、終局系の核だけを確定する。

*終局判定の改訂（ルール v0.6）。engine::terminate に最長手数（500組手で引き分け）を新設し、千日手を引き分けとして確定、盤上の終局を順序どおり一元評価する `evaluate` を単一の権威として据える。archive に `MaxTurns` を追加、RULE_VERSION を 0.6 へ、v0.8.2 の重複した安全網 500 を engine の単一値へ寄せて解消。web は権威関数に載せ替え、最長手数・千日手で引き分け終局し結果を正しく刻む。ワイヤ・書式は不変、配布版を v0.9.0 へ。*
