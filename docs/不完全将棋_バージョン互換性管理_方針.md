# 不完全将棋 バージョン・互換性管理 方針

> 位置づけ: 配布（早期評価者の別 PC への展開）の開始に伴い必要になった、バージョンと互換性の管理方針。実装の前に全体像を定める背骨であり、三段階の導入計画の拠り所とする。
> 対応リポジトリ現状: ワークスペース members = `engine`・`cli`・`tui`・`protocol`（TCP の殻は一旦 `tui` 内モジュール）。ルール仕様は v0.5 まで。

---

## 0. なぜ今これが必要か

一人開発の間は、すべての部品の整合は作者一人の頭の中で同期されていた。配布が始まり、**別々の PC で、別の人間が、独立にビルドして動かす**ようになると、「二つの実行体のバージョンが食い違う」という、一人開発には存在しなかった問題が現実になる。配布の開始と同時に、明示的なバージョン管理が要る。

ただし、必要が生じる時点に合わせて段階的に導入する（先回りしすぎず、後手にも回らない）。本方針は三段階の配分を定める。

---

## 1. 管理対象とするバージョン（区別）

このプロジェクトには性質の異なる「バージョン」が複数あり、混同しないよう区別する。

- **ルール仕様版** — ゲームのルールそのものの版（現 v0.5）。`docs/` のルール仕様文書と、`engine` の実装が対応する。**変わると、同じ局面・同じ着手から異なる次局面が生じうる**＝対戦の根幹に関わる。
- **プロトコル版** — 通信の約束事（commit-reveal の形式、メッセージ構造、ハッシュ照合、Ack、本人認証など）の版。`protocol` クレートが対応する。**変わると、二つの実行体が通信で噛み合わない**。
- **配布バイナリ版** — 配布される実行体（当面は TUI）の版。利用者が「自分は何版を持っているか」を識別する番号。
- **指示書版・文書版** — ルール仕様文書・各実装指示書の版（管理の参考。実行時の互換性には直接関わらない）。

---

## 2. 保証すべき互換性の三層

配布が始まると、二つの独立した実行体が次の三層で噛み合っている必要がある。

- **対戦の互換性** — あなたの実行体と相手の実行体が、通信して正しく対戦できるか。**最も切実**。プロトコル版とルール版の双方が噛み合う必要がある。
- **ルールの互換性** — 双方のエンジンが同じルール版を実装しているか。食い違えば、同じ着手から異なる次局面が生まれる。Phase3 の**盤面ハッシュ相互検証**が、この食い違いを事後検出する装置として既に存在する（バグ・改竄の検出と同じ仕組みが、ルール版不一致の検出も兼ねる）。
- **データの互換性** — save/load する棋譜や、中断救済で交換される棋譜・ハッシュが、版をまたいで同じ意味を持つか。当面は優先度低だが、版を重ねると効いてくる。

---

## 3. 対戦互換性の核：ルール版とプロトコル版のタプル

**対戦が成立するかは、単一の版番号では決まらず、「ルール版 × プロトコル版」のタプルで決まる。** この二つは独立に進化しうるが、対戦の成立にはペアで噛み合う必要がある（例: ルール v0.5・プロトコル v1）。ルールだけ上がってプロトコルが据え置きのことも、その逆もある。

これは設計哲学「核と交換可能な殻」がバージョニングに現れたものである。

- **変えると対戦が壊れる「核」** = ルール（`engine`）とプロトコル（`protocol`）。これらの版が対戦互換性を決める。
- **変えても対戦は壊れない「殻」** = TUI の見た目・操作性。これらの改善は、ルール版・プロトコル版を変えない限り、対戦互換性に影響しない。

### 3.1 互換性判定の規約：完全一致（当面の仮定）

互換性判定は、当面**タプルの完全一致**とする。すなわち、両者の (ルール版, プロトコル版) が完全に一致するときのみ対戦可能とする。

- 利点: 最も単純。判定が曖昧さなく決まる。版が浅く、利用者が閉じた少人数（あなたと息子さん）の現段階に適する。
- 代償: 些細なルール変更でも、全員が同時に更新しないと対戦できない。
- 将来: 運用して窮屈さを感じたら、**範囲許容**（例: プロトコル版が同じなら、後方互換なルール差分は対戦可能とする）へ緩める余地を残す。「何を後方互換とみなすか」という重い判断を抱えるのは、必要が生じてからでよい。**完全一致はあくまで仮定であり、可変である。**

---

## 4. 三段階の導入計画

必要が生じる時点に、仕組みの重みを段階的に配分する。

### 第一段（今）— 配布バイナリ版の表示とリリースタグ連動

配布が始まる今、最小限の背骨を入れる。

- **バイナリへの版埋め込みと表示**: `Cargo.toml` の version をビルド時にバイナリへ取り込み（Rust では `CARGO_PKG_VERSION` 環境変数がコンパイル時に利用可能）、起動時のヘッダ・`--version` 引数・TUI 内のいずれかで表示する。あなたと相手が「お互い何版か」を目視確認できるようにする。
- **タグとバイナリ版の一致**: Git のリリースタグ（例 `v0.5.0`）と、バイナリが名乗る版を**一致させる**。手で二箇所を直すと必ずいつか忘れるため、片方を正本にして導く仕組みにする（タグから version を導く、または version を正本にタグを打つ運用を決める）。詳細は第一段の実装指示で詰める。
- **リリースタグ連動**: 既存の GitHub Actions ワークフロー（push トリガーで Windows バイナリを Artifacts に上げる）を拡張し、**タグ push 時に GitHub Releases を作成しバイナリを添付する**。日常の push は従来どおり Artifacts で確認、節目にタグを打って確定版をリリース、という二系統とする（リリースは「人に渡す確定版」という特別な意味を持たせる）。
- この段階では、ルール版・プロトコル版を個別に分離管理する必要はまだ無い（TUI のみで、あなたと息子さんが口頭で版を確認できる）。配布バイナリ版という単一の番号で足りる。

> なぜ今か: 配布が始まり、二人が別々にビルド/取得した実行体を持つため、「どの版か」を識別する最小手段が要る。口頭確認が効く規模なので、自動交渉まではまだ不要。

### 第二段（Web に進む前）— プロトコルへのバージョン交渉の組み込み

Web で不特定の相手と繋がるようになる前に、プロトコル自身に互換性確認手段を組み込む。

- **接続時のバージョン交渉**: 接続確立時、両者がまず自分の (ルール版, プロトコル版) を名乗り合い、§3.1 の完全一致規約で互換性を確認してから対戦を始める。食い違えば**対戦開始前に弾く**。
- **事前検出と事後検出の重層**: バージョン交渉は「対戦開始前に弾く」事前検出。既存の盤面ハッシュ相互検証は「対戦開始後に局面のずれで気づく」事後検出。両者は重層的に効く（事前に版で弾き、すり抜けた齟齬は事後にハッシュで捕まえる）。
- ここで初めて、配布バイナリ版とは別に、**ルール版・プロトコル版を実行体が自己申告できる形**で保持する必要が出てくる（交渉の材料として）。

> なぜ Web 手前か: TUI で閉じた二人の間は口頭で版を確認できるが、Web で不特定の相手と繋がると口頭確認が効かず、プロトコルに組み込んだ自動交渉が必須になる。Web に入る前に仕込む。

### 第三段（Web 初版リリース）— ルールとプロトコルのタプルの体系的管理

配布が広がり版が重なる Web 初版で、タプルを体系的に管理開始する。

- **タプルの正本管理**: (ルール版, プロトコル版) を、コード・文書・リリースを貫く正本として一元管理する。どのリリースがどのタプルを実装するかを明示する（リリースノート／対応表）。
- **配布バイナリ版とタプルの対応づけ**: 配布バイナリ版（利用者向けの単一番号、おそらくセマンティックバージョニング的）と、その実体である (ルール版, プロトコル版) の対応を記録する。利用者は単一版で識別し、互換性判定はタプルで行う。
- **セマンティックバージョニングとの接続**: 「対戦互換性を壊す変更（ルールまたはプロトコルの版が上がる）」をメジャー級、「壊さない改善（TUI の見た目等）」をマイナー/パッチ級として扱う、という方針を確立する（核と殻の区別がそのまま版の刻みに対応する）。
- **完全一致からの見直し**: この段階で、§3.1 の完全一致規約を運用実績に照らして再評価し、範囲許容へ緩めるかを判断する。

> なぜ Web 初版か: 配布が不特定多数に広がり、異なる版の実行体が同時に世に存在するようになると、単一番号では表現しきれず、タプルの体系的管理が必要になる。

---

## 5. 当面の宿題（各段で詰める論点、今は置く）

- 第一段: タグと version の一致を「どちらを正本に、どう導くか」。ワークスペースの各クレート version を一元管理するか（`workspace.package` での集約など）。
- 第二段: 交渉メッセージの形式。版を名乗るタイミング（本人認証・ハッシュ交換との順序）。不一致時の利用者への通知の仕方。
- 第三段: 完全一致を緩めるか。緩めるなら「後方互換なルール差分」の定義。データ（棋譜・セーブ）の版間互換。

---

## 6. 不変の原則（全段共通）

- **対戦互換性を決めるのは核（ルール・プロトコル）の版であり、殻（TUI の見た目）の改善は対戦互換性に影響しない。** この区別を、版の刻み方の基準とする。
- **エンジンとプロトコルは純粋な核として保たれ、バージョン情報の表示・交渉といった I/O は殻が担う。** 版交渉のメッセージ送受はソケットの殻、版の値そのものは核が自己申告する、という分離を保つ。
- **完全一致は仮定であり可変。** 単純な規約から始め、必要が生じたら緩める。

---

*バージョン・互換性管理 方針 v1.0 — 配布開始に伴う背骨。三段階（今＝バイナリ版表示とリリースタグ連動、Web 手前＝プロトコルのバージョン交渉、Web 初版＝ルール×プロトコルのタプル体系管理）。対戦互換性は (ルール版, プロトコル版) のタプルで決まり、判定は当面「完全一致」と仮定する。核と殻の区別が、版の刻み方の基準となる。*
