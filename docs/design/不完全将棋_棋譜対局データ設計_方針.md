# 不完全将棋 棋譜・対局データ設計 方針 — 進行イベントストリーム（ライブ／アーカイブ／機械学習）

> 位置づけ: 対局の進行データを、ライブ観戦・アーカイブ・機械学習という複数の用途へ整合的に供給するための全体設計（背骨）。**設計は全体、実装は最小から**の二段構え。観戦のライブ性とチート排除（秘匿境界）のバランスは**未決の論点として先送り**し、本文書では旗を立てるに留める。
> 動機: 息子との実戦（TUI・家庭内 LAN）で棋譜を保存し損ねたことが直接の発端。棋譜保存を、TUI のローカル save/load ではなく、Web 側で観戦フレームワークと整合する形（ライブとアーカイブの両面）に設計したい。将来は機械学習にも活用したい。
> 前提: ルールは v0.5。棋譜は「初期局面（SFEN）＋着手列（各ターンの両 USI）」を最小の真実とし、各局面はエンジンが再計算する（解決後局面を保存しない）方針が確立済み。バージョン互換性は (ルール版, プロトコル版) のタプルで管理（完全一致を仮定）。

---

## 0. 中心的な洞察：観戦・ライブ・アーカイブ・学習は同じ一つの流れの変奏

観戦・ライブ配信・アーカイブ保存・機械学習ログは、別々の機能ではなく、**「対局の進行を、対局者以外の受け手へ取り出す」という同一の行為の変奏**である。

- **観戦** = 進行を第三者へ**リアルタイム**に流す。
- **アーカイブ** = 進行を未来へ（時間を超えて）流し、**蓄える**。
- **機械学習ログ** = 進行を**構造化して集める**。

したがって設計の核心は一つに収束する: **対局の進行を、構造化されたイベントの流れ（進行イベントストリーム）として、対局者の外へ取り出せる口を作る。** この一つの本流から、ライブ・アーカイブ・学習の三つの河口が派生する。三つに別々の仕組みを作らない（淀川: 三川が合流して一本の本流となり、下流に河口を開く）。

---

## 1. 進行イベントストリーム（本流）の定義

対局を、構造化されたイベントの列として表現する。これが全用途の共通の源。

### 1.1 対局開始イベント
- 初期局面（SFEN）。
- 先後（陣営）の割り当て。
- **対局者の識別（principal）**: 各対局者を `principal`（主体）として刻む。principal は対局の当事者なので、本流（中身）に入る。その構造・型（signed_in / anonymous /〔将来〕agent）・自己申告属性・同意や帰属の扱いの詳細は、別文書『プリンシパル設計』に従う。本書はそれを「principal が本流に刻まれる」とだけ受け、中身（誰が指したか）として扱う。（初稿の「必要なら、最小では匿名でよい」という軽い扱いから格上げ——principal は規約・同意・削除権・機械学習のデータ品質の根っこであり、版タプルと並ぶ刻印。）
- **バージョンタプル（重要・§3）**: (ルール版, プロトコル版)。この対局がどの版で指されたかを刻む。
- 対局の一意な識別子、開始時刻（任意）。

### 1.2 各ターンのイベント
同時着手の一ターンを、進行として表せる粒度で記録する。最小では:
- そのターンの **(先手 USI 着手, 後手 USI 着手)**。
- 解決結果の要約（取得・相討ち・スワップ・戦国無双・両玉スワップ引き分け・玉の死・成り等。エンジンが算出。表示と分析に有用）。
- （ライブ用にはさらに、commit 段階・reveal 段階という**進行中の状態遷移**もイベントになりうる。§4 参照。アーカイブには確定した両着手だけで足りる。）

### 1.3 終局イベント
- 終局種別（確定的詰み・玉の死・両玉スワップ引き分け・投了）。
- 勝敗（または引き分け）。

> 本流の最小の真実は「対局開始（初期局面＋版タプル＋先後＋principal）＋ 各ターンの両着手 ＋ 終局」。各局面はこの本流からエンジンが再計算できる（解決後局面を保存しない方針を維持）。

### 1.4 中身（本流）とエンベロープ（取り扱い層）の分離

対局そのものの記録（中身＝本流）と、その記録をどう取り扱ったかの情報（エンベロープ）を、別の層に分ける。封筒の比喩: 手紙（中身）と、それを包む封筒（取り扱いメタ）。**当事者が生成したものは中身、当事者でない者・取り扱いに関する情報はエンベロープ。**

- **中身（本流）** = 対局そのもの: 初期局面・着手列・版タプル・終局、そして **principal（誰が指したか＝対局の当事者）**。これが §2 のアーカイブ・ライブ・機械学習が共有する純粋な源。記録係の有無で中身の形は変わらない（記録係つき対局も野良対局も、中身は同一構造）。
- **エンベロープ（取り扱い層）** = 中身をどう取り扱ったかのメタ情報:
  - **記録係の両面署名（本人性・非改竄性）**。記録係は対局の当事者ではない（権威なき観察者）ので、中身に含めず、丸ごとエンベロープに置く。署名が担保するのは**棋譜の中身（対局そのもの）の完全性だけ**——帰属や consent といった他者の取り扱い情報には及ばない（記録係は自分が観察した事実についてのみ証言する。審判なし・各自が自分の側に責任、の思想に忠実）。
  - **帰属・consent receipt 参照・法的根拠**（『プリンシパル設計』の同意・帰属の記録）。これらは principal 自身や場が保証するもので、記録係署名の対象外。
- **対局者各々の署名**: 中身に入れるか（本人性を棋譜自体に持たせる）、エンベロープに置くかは、**議論の余地として保留**（本人性をどこまで棋譜本体に持たせたいかの設計判断。今は決めきらない）。

### 1.5 対局中のハッシュは揮発してよい

進行中の **盤面ハッシュ相互検証**（Phase3）は、対局を正しく進めるためのリアルタイムの整合性チェックであり、**使い捨ての足場**。対局後は**揮発してよい**（棋譜＝初期局面＋着手列が残れば、任意局面のハッシュはエンジンがいつでも再計算できる）。

- これは、エンベロープで記録係署名が中身の完全性を**記録時点で封じる**ためのハッシュ（封蝋）とは**別物**。前者は対局を進めるための揮発する検証、後者は記録を固定するための封蝋。目的も寿命も異なる。混同しない。

## 2. 三つの河口（本流からの派生）

同一の進行イベントストリームを、用途ごとに異なる形で受ける。

### 2.1 アーカイブ（蓄える）
- 本流を**固めて保存**する。最小構成は「対局開始（初期局面＋版タプル＋先後）＋着手列＋終局」。これだけで全局面を再現できるので軽い。
- 指し終わった対局は**全て開示済み**なので、**秘匿境界の問題が無い**（アーカイブを観るのは常に全知の状態）。
- 用途: 後から自分や他者が振り返る、水墨様式の局面図で鑑賞する（既存の再生・鑑賞デモの素材）、機械学習の素材になる。

### 2.2 ライブ観戦（リアルタイムに流す）
- 本流を、確定するそばから観戦者へ**ブロードキャスト**する。中継点（Durable Object）が、対局メッセージの中継に加えて観戦者への配信を兼ねる自然な位置にいる。
- **ライブだけが秘匿境界を持つ**（§4。未決の論点）。
- 用途: 対局を第三者がリアルタイムに観る。

### 2.3 機械学習ログ（構造化して集める）
- 本流を、学習に使える形で**集積**する。アーカイブが貯まれば、それ自体が学習データセットになる。
- 不完全将棋の AI は不完全情報ゲーム（相手の手を確率分布で読む、ポーカー AI に近い）であり、**人間が実際にどう裏をかいたか**のデータが貴重。実戦の棋譜がその宝の山。
- **版タプルが品質を守る**（§3）: 版が混ざった対局を一緒くたに学習しないよう、ルール世代で揃えて学習できる。ルール変更で人間の戦略がどう変わったかの分析もできる。
- 学習に効く追加文脈（各局面の合法手集合・思考時間・勝敗など）は、**必要が生じたら足す**（最小のアーカイブには含めない。学習を試みて初めて何が効くか分かることも多い）。

## 3. バージョンタプルを棋譜に刻む（不完全将棋に固有の必須要件）

将棋の棋譜は四百年を超えて通じる——ルールが安定しているため、棋譜は版を名乗らずとも（初期局面＋着手列だけで）いつでも再現できる。**不完全将棋はその安定をまだ持たない**（v0.5、これからも変わる。安定がいつ来るか未知）。

- **問題**: 解決後局面をエンジンが再計算する設計のため、**同じ着手列でも、再計算するルール版が違えば違う結末になりうる**（戦国無双は v0.3→v0.4→v0.5 で変化、両玉スワップは v0.5 で追加）。ゆえに棋譜は、着手列に加えて**どのルール版で指されたか**を記録しなければ、後から正しく再現できない。
- **解**: 棋譜（対局開始イベント）に **(ルール版, プロトコル版) のタプル**を刻む。再現に必須なのはルール版（解決を計算するのはルール）。対局の出自の完全な記録として、プロトコル版も併せてタプルで残す。
  - *追記（家元システム・2026-07）*: このルール版の身元は**家系図**（宗家＝札なしの canonical な一本・分家＝札つきの枝）として捉え直された。刻まれる素の `major.minor` は**宗家**を意味し、分家は札つきで降りる。既存の刻印は前方互換（札なし＝宗家）で移行不要、表現の拡幅は最初の分家発火時。→ `design/不完全将棋_ルール版の系譜と対局モード_方針` §1.0。
- **思想的一貫性**: 実行体が版を名乗り（`--version`）、通信が版を名乗り（接続時のバージョン交渉）、そして**記録（棋譜）が版を名乗る**。版という自己申告が、コードから通信を経てデータにまで及ぶ。
- **将来の果実**:
  - **再現性**: 未来（v1.0 等）から過去（v0.5）の棋譜を正しく再生できる。版を記録したルールで再計算すればよい。ルール進化の考古学（v0.5 で合法・v0.6 で非合法な手を含む棋譜、等）も可能。
  - **データ品質**: 機械学習で版世代を揃えられる。ルール変化に伴う戦略変化の分析もできる。

## 4. ライブの秘匿境界（未決の論点・先送り）

ライブ観戦のみが持つ、本質的なトレードオフ。**本文書では旗を立てるに留め、解は観戦実装時に委ねる。**

- **問い**: ライブ配信で、観戦者に何を、いつ見せるか。対局者は互いの手を開示の瞬間まで知らない。観戦者が、両者の commit 済み・未開示の両手を先に見られるなら、観戦者は**神の視点**（対局者より多くを知る）になる。見せないなら、観戦者も開示まで待ち、対局者と同じ不完全情報を共有する。
- **トレードオフ（ライブ性 vs チート排除）**:
  - 観戦を「開示の瞬間まで伏せる」＝対局者と同じ情報状態にすると、ゲームの秘匿哲学に忠実だが、配信遅延や、対局者と観戦者の同期（観戦経路から手が漏れて対局者へ伝わる経路＝チートの懸念）の設計が要る。
  - 観戦を「全知」にすると単純だが、観戦のスリル（読み合いを観る面白さ）が損なわれ、また観戦経由の情報漏洩（対局者へのチート援助）の温床になりうる。
- **アーカイブにはこの問題が無い**（指し終わりは全開示）。**ライブ固有**の論点。
- **方針**: 観戦・ライブ配信を実装する段階で、実物の制約（遅延・配信機構）を見ながら決める。本流（進行イベントストリーム）は、ライブ配信の有無・秘匿方式に依存しない形で先に定義しておく（本流は「確定した進行」を表し、それを観戦へどう絞って流すかは河口の設計）。

## 5. 既存設計との接続

- **最小の真実は着手列**: 既に「初期局面＋着手列、局面はエンジン再計算」という構造がある。本流の定義はこれを素直に拡張したもの（版タプルと終局・解決要約を明示化）。アーカイブはこの最小構造＋版タプルで足りる。
- **Durable Object が中継点**: ブラウザ秘匿対戦の中継（審判なしの不透明な郵便配達）が、観戦ブロードキャストとアーカイブ書き出しを兼ねうる自然な位置。ただし**審判なしの原則は不変**——Durable Object は進行を流す／蓄えるが、判定はしない。
- **棋譜構造の流用**: 操作可能化・中断救済で各クライアントが既にローカルに積んでいる棋譜（着手列）が、アーカイブの素材そのもの。
- **プリンシパル設計との接続**: 本流の中身に刻まれる principal（誰が指したか）の構造・型・同意・帰属・削除の扱いは、別文書『プリンシパル設計（対局主体スキーマ）』に従う。本書（データ設計）は対局データの世界観・全体設計を、プリンシパル設計はその中の principal を実装に落とす具体を担う（上位＝世界観／下位＝実装）。エンベロープに乗る帰属・consent receipt 参照・法的根拠も、プリンシパル設計が定義する。

---

## 6. 実装は最小から（段階）

設計は全体（§0〜§5）だが、実装は最小の河口から開け、下流を後付けする。

- **第一歩（最小・まず作る）**: **版タプル付きアーカイブ**。Web 盤の対局終了時（およびできれば対局中つねに）、本流の最小構成（初期局面＋(ルール版,プロトコル版)＋先後＋着手列＋終局）を、確実に保存／書き出しできるようにする。今回の「保存し忘れ」の痛みを直す。保存先は最小ではローカル（ダウンロード／ローカルストレージ／コピー可能テキスト）でよい。
- **第二歩**: アーカイブを**取り出して再生／鑑賞**（既存の水墨様式の再生・局面図デモへ流し込む。実戦が鑑賞素材になる）。
- **第三歩（後の殻）**: **ライブ観戦**。Durable Object のブロードキャスト。ここで §4 の秘匿境界を実物で決める。
- **第四歩（後の殻）**: **機械学習ログ**の集積と、学習に効く追加文脈の付与（必要が生じた分だけ）。

各河口は独立に後付けできる。本流（版タプル付き進行イベント）さえ定義しておけば、下流は無駄なく積める。

## 7. 当面の宿題（各段で詰める、今は置く）

- 第一歩: 版タプルの棋譜内表現（フォーマット）。保存の UX（自動／手動、保存先）。既存ローカル棋譜構造との整合。
- 第三歩: ライブの秘匿境界（§4）。観戦の同期とチート排除。
- 第四歩: 学習に効く文脈（合法手集合・思考時間・勝敗・対局者匿名 ID 等）の取捨。版世代でのデータ分割。

## 8. 不変の原則

- **一つの本流、複数の河口**: 観戦・アーカイブ・学習に別々の仕組みを作らず、一つの進行イベントストリームから派生させる。
- **版タプルを記録に刻む**: 不完全将棋が安定を得るまで（いつになるか未知）、棋譜は自分の出自（ルール版・プロトコル版）を名乗る。再現性とデータ品質の土台。
- **審判なしは不変**: 中継点が進行を流す／蓄えるが、判定はしない。
- **設計は全体、実装は最小から**: 全体の背骨を描いたうえで、最小の河口（版タプル付きアーカイブ）から開け、下流を後付けする。
