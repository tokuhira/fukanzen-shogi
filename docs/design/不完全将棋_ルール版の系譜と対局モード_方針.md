# 不完全将棋 ルール版の系譜と対局モード 方針 — レガシー対局と時間制の二軸

> 位置づけ: board.js の神ファイル解体（view 純粋化・I/O 分解）が尽き、地盤が整った先で見えてきた二つの現実的な必要——**過去のルール版でも対局できること**と、**時間制という対局モードのバリエーション**——に、実装の前に背骨を与える。淀川（棋譜対局データ設計）が「一つの本流・複数の河口」を、バージョン互換性管理が「(ルール版, プロトコル版) のタプル」を定めたのを受け、その二つが**変種と対局モード**として現れるときの設計方針を定める。
>
> 動機: ルールはこれからも変わる（v0.6 はまだ安定ではない）。だがルールが上がったとき、古い版を**葬りたくない**。「昔はこんなバランスの悪いルールもあったね」「懐かしい、ちょっと指してみようか」——過去のルール版を、記録として再生できるだけでなく、**選んで指し直せる**ようにする。版図の「完璧でない相棒と一緒に学ぶ」精神を、ルールの歴史そのものに及ぼす。あわせて、時間制という**ルールと直交する対局モード**が将来バリエーションを持つことを見越し、盤面・終局・本流に席を予約しておく。
>
> 前提: ルール v0.6・PROTOCOL 5。engine は単一の `RULE_VERSION = (0, 6)`（`engine/src/lib.rs`）で、`resolve(pos, sente, gote)`（`engine/src/resolve.rs`）は版を引数に取らず常に現行版で解決する。アーカイブには `rule: RULE_VERSION` が刻まれる（`engine/src/archive.rs`）。版交渉は完全一致（`protocol/src/negotiate.rs` の `negotiate_versions`）。終局判定は単一正本 `protocol::game_result` に合流済み（終局判定の単一正本化アーク）。**設計は全体、実装は最小から**の二段構えを踏襲する。

---

## 0. 中心的洞察：二つの**直交する**軸

この文書が扱うのは、独立に進化する二軸である。混同しないことが設計の要となる。

| | **軸A：ルール版の系譜** | **軸B：対局モード** |
|---|---|---|
| 具体像 | レガシールール対局のサポート | 時間制（複数フォーマット想定） |
| 変えるもの | **解決（resolution）** | **終局条件＋進行メタ**（解決は不変） |
| 担う機構 | 版タプルの `rule` 成分 | セッション設定（`protocol` 成分の上で交渉） |
| engine への影響 | `RULE_VERSION` 定数 → **版ディスパッチ** | **無し**（engine は時計を知らない） |
| 版図での座 | 記録の考古学・ルール進化の保存 | 「人と AI の速度差」への層（§E 時間制の種） |

軸Aは**同じ着手列が版で違う結末を生む**ことに向き合う（淀川 §3）。軸Bは**解決を一切変えず、対局の終わり方と進行の刻みだけを足す**。この直交が、両者を一枚で扱いつつ混ぜない根拠である。核と殻の区別（互換性方針 §6）が、そのまま二軸の性格を決めている——軸Aは核（ルール）の系譜、軸Bは核の外側の対局モード。

---

## 1. 軸A：ルール版の系譜——レガシー対局

### 1.1 何を約束するか

ルールが v0.6 から次版へ上がったとき、**旧版は engine から消えず、選んで指せる**。二つの河口を持つ：

- **再生**（read-only）: 旧版で綴じられたアーカイブを、その版の解決で正しく再現する。指し終わった対局は全開示なので**秘匿境界の問題は無い**（淀川 §2.1）。安い河口。
- **対局**（opt-in）: 現行クライアント同士が、合意のうえ**旧版のルールを選んで**新たに対局する。「懐かしいね、v0.5 で指してみよう」。

いずれも、engine が**版を指定して解決を計算できる**ことに帰着する。

### 1.2 鍵は engine の版ディスパッチ

現状 `resolve(pos, sente, gote)` は単一版を計算する。レガシー対応の要石は、**解決を版でディスパッチできる形**にすること——版を引数に通すか、版ごとの ruleset をゲーム開始時に一度選ぶ戦略オブジェクトとするかは、実装指示書で詰める（移植抽象化 §D が等価物を列挙したのと同じ流儀で、ここでは形を決めきらない）。

この要石は、前段の相談で二度突き当たった**同一の一粒**である。すなわち「旧棋譜を将来の engine で忠実再生する」（淀川 §3 の考古学）と「旧版で対局する」は、engine 版ディスパッチという**一つの解錠**を共有する。必要が生じたとき、一つの作業が二つの果実を実らせる。

### 1.3 完全一致から「版選択」へ（範囲許容ではなく）

現状の交渉は (ルール版, プロトコル版) の**完全一致**（`negotiate_versions`）。レガシー対局のために、これを**版選択**へ拡張する：

- クライアントは自分が**指せるルール版の集合**を名乗る（単一版でなく能力の集合）。
- 部屋は両者が共有する版から一つを選ぶ（既定は最新、懐古では明示的に旧版を選ぶ）。
- 選ばれた版は、これまで通りアーカイブに刻まれる（機構は既存）。

互換性方針 §3.1 が予約した「完全一致を緩める余地」を、ここでは**曖昧な後方互換判定（範囲許容）としてではなく、明示的な版選択として**使う。「何を後方互換とみなすか」という重い判断を抱えない——プレイヤーがどの版で指すかを選ぶだけ。「サイレント A/B をやるほどには進めない」という射程判断と整合する。

**プロトコル版は据え置き**：旧*ルール*を現行*プロトコル*の上で指す。古いプロトコルを甦らせる必要はない（現行クライアント同士が旧ルールセットを選ぶだけ）。ルール版とプロトコル版が独立という原則（互換性方針 §3）が、ここで効く。

### 1.4 引き受ける約束——engine は版を捨てない

これは意識的な設計上の負債である：**engine は過去の解決ロジックを、版で選べる形で保持し続ける**。戦国無双（v0.3→v0.4→v0.5 で変化）や両玉スワップ（v0.5 で追加）のような、同じ着手列を違う結末へ導く差分を、engine は忘れない。ルールの歴史が engine に堆積する。これがレガシー対局の代償であり、懐古の器を持つことの値段である。過ぎたるは及ばざる——だから**版を無限に保持し続けるとは約束しない**。どこまで遡れるようにするかは、必要と保守コストが釣り合う点で判断する（当面は「直前版」から始めれば足りる）。

### 1.5 発火の時——最初のルール変更が版ディスパッチを連れてくる

版が一つしかない今、ディスパッチ先は存在しない。**engine 版ディスパッチを今作るのは過ぎたる**。要石が打たれるべきは、v0.6→次版という**最初のルール変更が実際に企てられる、その時**である。レガシー対応は単独のインフラ工事ではなく、**最初のルール変更に付随する規律**として建てる——「次版へ上げるとき、v0.6 を置換するのでなく、版ディスパッチで併存させる」。この文書の今の仕事は、その約束を**先に言葉にしておく**ことにある。

## 2. 軸B：対局モード——時間制

### 2.1 ルールと直交する

時間制は**解決を一切変えない**。誰が相討ちに勝つか、玉がどう死ぬか、スワップがどう裁定されるか——盤面の理は時計と無関係。時計が足すのは**新しい終局条件（時間切れ負け）**と**進行のメタ（各手の消費時間）**だけ。ゆえに時間制は (ルール版) と直交し、版タプルの `rule` 成分を触らない。

### 2.2 時間切れは protocol 層の終局——投了と同じ座

「盤面設計への早めの配慮」の核心はここにある。**時間切れ負けは engine の終局ではなく、protocol の終局**とする。理由は既存の層構造から一意に落ちる：

- 盤面は終わっていない。**時計が終わった**。engine（盤の理）の外の出来事である。
- これは投了（`Action::Resign` が engine の `resolve` で `unreachable!` なのと同じ——盤の出来事でない表明）と**まったく同じ座**にある。
- 終局判定の単一正本化アークが作った単一窓口 `protocol::game_result` に、**第三の終局ソース（clock）**を足すだけで済む。「盤面終局＋投了」の二源が三源になる。その拡張点が今ちょうど単一で綺麗——このアークの果実がここで効く。

### 2.3 本流の予約席はすでに名付けられている

淀川 §1.2 は各ターンのイベントに「思考時間」を**必要が生じたら足す**メタとして明示済み。時間制のための席は**すでに本流に予約されている**。新設は要らない。終局イベント（淀川 §1.3）に「時間切れ」種別を足す口も、game_result の三源化と対応する。

### 2.4 「能力」は protocol 版、「各局の設定」はセッション交渉

時間制のバリエーション（ブリッツ／ラピッド／秒読み…）を、版を毎回刻まずに増やすための二段：

- **能力**：時計メッセージをワイヤに載せる＝ PROTOCOL 5→6（一度きり）。プロトコルが「時計を運べる」ようになる。
- **各局の設定**：この対局を*どの時間設定で指すか*は、部屋ごとの交渉（版でなくセッション設定）。ブリッツもラピッドも、同じプロトコル版の上の別設定。

チェスサーバの二段（プロトコルは時計を運べる／各卓が持ち時間を選ぶ）と同じ。これにより時間制の変種は protocol 版を汚さずに育つ。

### 2.5 今やる配慮＝焼き込まないこと

過ぎたるは及ばざる。今フィールドは足さない。**「終局は盤面＋投了の二源」という暗黙の前提を、拡張が高くつく場所に焼き込まない**ことだけが今の配慮であり、それは game_result 単一窓口がすでに満たしている。この文書の仕事は、clock を足す場所（game_result の三源目・淀川の思考時間席・終局種別の時間切れ）を**名指しして予約する**ことに尽きる。

## 3. DevOps：本番と実験の使い分け

二軸を**実 DO で検証する**段になると（このプロジェクトは実 DO＋本番 WS 検証が必須）、本番汚染が現実になる。現状 `server/wrangler.toml` は単一 Worker・単一の `ARCHIVES`/`SPECTATE_TOKENS` KV で、環境分けが無い（`wrangler dev` はローカル miniflare で隔離されるが、デプロイ先の staging が無い）。

- **本番/ステージング隔離**：`[env.staging]` を足し、専用 DO＋専用 KV を持たせ、web の dev デプロイ（pages.dev）をステージング Worker の URL へ向ける。レガシー対局・時間制を実 DO で検証する段になったら作る、標準的で小さな一手。
- **変種の名前空間**：多変種を実際にホストする段では、DO room キー・アーカイブ prefix を版／モードで分ける。ただし版タプル交渉が既に**変種同士を混ざらせない**（`negotiate_versions` が非共有版のペアリングを弾く）ので、名前空間は「共存」でなく「整理・列挙」のための後付けでよい。

いずれも先回りしない。必要が生じた段で重みを配分する（互換性方針の三段階配分と同じ流儀）。

## 4. 実装は最小から（段階）

設計は全体（§0〜§3）だが、実装は最小の河口から開ける。

1. **（発火待ち）engine 版ディスパッチ** — 最初のルール変更（v0.6→次版）に付随して建てる。まず旧版を捨てず版で選べる形へ。→ 軸A の要石、レガシー**再生**が最初の果実（安い河口）。
2. **レガシー対局の版選択交渉** — 完全一致を版選択へ拡張。engine ディスパッチの後。→ 軸A の**対局**河口。
3. **時間制の骨組み** — game_result に clock 終局ソース、本流に思考時間、PROTOCOL 5→6 の能力。→ 軸B。最小は単一の持ち時間から、フォーマット変種は後付け。
4. **ステージング隔離** — 1〜3 のいずれかを実 DO 検証する直前に。

各段は独立に後付けできる。今**着手すべきは何も無い**（変種が一つも無い＝過ぎたる）。この文書は背骨を描き、最初のルール変更・時間制の必要が来たときに無駄なく積める形にしておくためにある。

## 5. バックログとの接ぎ穂

- **§E 時間制の種**：本書 §2 がその背骨。「接続レベルの放棄のみ現状対応」から、game_result の clock 終局ソースへ育てる道を示した。
- **§E エコシステムの汎化／§D 移植の抽象化**：多変種ホスト（§3 変種名前空間）と版ディスパッチ（§1）は、これらの種に水をやる前段。「手番だけを引く」操作の他ゲーム応用も、版タプルを変種セレクタとする同じ機構の上に載る。
- **淀川 §3 の考古学**：軸A の engine 版ディスパッチが、淀川の約束した「未来から過去の棋譜を正しく再生」を実装で裏打ちする。記録（§A 記録係）とルール系譜（本書）が、版タプルという同じ刻印で結ばれる。

---

## 6. 不変の原則

- **二軸は直交する**：ルール版の系譜（解決を変える・核）と対局モード（解決を変えない・核の外）を、一枚で扱いつつ混ぜない。
- **ルールの過去を葬らない**：engine は版で解決を選べる形を保ち、懐古の対局を可能にする。ただし版を無限には抱えない（保守コストと必要の釣り合い）。
- **時間切れは盤の外の終局**：clock は engine でなく protocol の game_result に座る（投了と同じ）。盤面の理は時計を知らない。
- **能力はプロトコル版、設定はセッション交渉**：変種を版で毎回刻まずに育てる。
- **審判なしは不変**：どの版・どのモードでも、中継点は進行を流す／蓄えるが判定はしない。
- **設計は全体、実装は最小から／過ぎたるは及ばざる**：背骨を今描き、要石は最初のルール変更・時間制の必要に付随して建てる。今着手する実装は無い。

---

*ルール版の系譜と対局モード 方針 v1.0 — 二つの直交軸。軸A＝ルール版の系譜（レガシー対局）は engine の版ディスパッチを要石とし、完全一致交渉を「版選択」へ拡張して、旧版を葬らず選んで指せるようにする（再生→対局の二河口）。要石は最初のルール変更に付随して建てる。軸B＝対局モード（時間制）は解決と直交し、時間切れを protocol::game_result の第三の終局ソース（投了と同じ座）とし、本流の思考時間席（淀川 §1.2）と PROTOCOL 能力／セッション設定の二段で育てる。今の配慮は「二源終局を焼き込まない」ことに尽き、単一窓口 game_result が既に満たす。DevOps は必要が生じた段でステージング隔離・変種名前空間を配る。核と殻の区別が二軸の性格を決め、審判なしは不変。設計は全体、実装は最小から。*
