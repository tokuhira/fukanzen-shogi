> 対象実行者: Claude Code（Sonnet）
> 前提: 配布 v0.7.1 リリース済み。ワークスペース members = `engine`・`cli`・`tui`・`protocol`・`engine-wasm`・`protocol-wasm`・`notation`・`notation-wasm`。`version.workspace = true` で全クレートが配布版（0.7.1）を継承。棋譜の核 `engine::kifu::Kifu`（`initial_position + Vec<Ply>`、`replay(n)` で決定的に任意局面を再現）と、その素の直列化 `engine::serialize::{kifu_to_string, kifu_from_string}`（`sfen <初期局面>` ＋ `N: <先手USI> | <後手USI>` 行列）が実装済み。版タプルは `protocol::MY_VERSION = VersionTuple { rule: engine::RULE_VERSION /* (0,5) */, protocol: PROTOCOL_VERSION /* 2 */ }`。Web 盤は `engine-wasm`（`resolve_ply`・`game_status`・`legal_actions`）と `protocol-wasm`（`ProtocolSession`・版交渉）の Wasm を使い、`web/board.js` が `kifu.plies`（`{sUsi,gUsi,sText,gText}`）と `INITIAL_SFEN` を保持、終局フックは online で `endOnlineGame(msg)` 一点。
> 関連文書: `不完全将棋_棋譜対局データ設計_方針.md`（淀川。本実装はその §6 **第一歩**）、`不完全将棋_プリンシパル設計_対局主体スキーマ.md`（§5 継ぎ目予約）、`不完全将棋_版図_世界観と設計方針.md`（§3 棋譜への刻印・§7 実装は最小から）、`不完全将棋_実装指示書_互換性確認プロトコルv0.6.0.md`（版タプルの出どころ）、`不完全将棋_バージョン互換性管理_方針.md`。
> 性格: 棋譜に **版タプル（ルール版・プロトコル版）を刻んで、対局を確実に保存／書き出す**最小の河口。フォーマットは `engine` が定める**核**（中身とエンベロープの最小形）、保存 I/O は `web` が担う**殻**。今回の「保存し損ね」の痛みを直す。**principal の席は予約のみ**（中身＝識別・consent・帰属は別実装。本書では空欄で席だけ空ける）。**対戦互換性は変えない**（ルール版・プロトコル版・ワイヤプロトコルは不変。アーカイブは純粋に加算的）。

## 0. 目的と範囲

- **作るもの**:
  1. **アーカイブ書式 v1**（§1）— 既存の素の棋譜本体の上に、版タプルほかのヘッダを前置した、人が読めてコピペできるテキスト形式。
  2. `engine` にフォーマットの**正本**（書き出し・読み込み・往復テスト、旧書式の後方互換）（§3）。
  3. `protocol-wasm` に**版タプルの getter**、`engine-wasm` に**アーカイブ書き出し**を露出（§4）。
  4. `web` に、**終局時（および対局中つねに）の保存**（ダウンロード＋コピー可能テキスト）（§5）。
- **位置づけ**: 淀川 §0 の「一つの本流・複数の河口」のうち、**アーカイブ（蓄える）の河口を最初に開く**。本流の最小構成「初期局面＋(ルール版,プロトコル版)＋先後＋着手列＋終局」を、確実に固めて書き出せるようにする。指し終えた対局は全開示なので**秘匿境界の問題が無い**（淀川 §2.1）——だからこそ最初の河口に選ぶ。
- **作らないもの（非ゴール。§10 に申し送り）**:
  - アーカイブを**取り出して再生／鑑賞**する UI（淀川 §6 **第二歩**）。本書は書き出し（save）に徹する。読み込み関数は `engine` 側に往復テストの担保として実装するが、Web の再生導線は次段。
  - **サーバ側アーカイブ**（Durable Object への書き出し）・**ライブ観戦**（淀川 §6 第三歩。秘匿境界 §4 は未決）。
  - **principal の中身**（識別・認証・consent・帰属スタンプ）。版図／プリンシパルの第一歩＝別指示書。本書は `sente`/`gote` ヘッダの**席を予約**するのみ。
  - **記録係署名・エンベロープの封蝋**（中身の完全性の担保）。
  - **機械学習用の追加文脈**（合法手集合・思考時間など）。
  - **CLI/TUI の `:save` をアーカイブ書式へ移行**すること（§3 で engine の読み手は両書式対応にするので、CLI/TUI は再ビルドでアーカイブを**読める**。書きの移行は任意の follow-on）。
  - **app 版を再現に使う**こと（再現に必須なのはルール版のみ。app 版は出自の補助）。
  - **異なるルール版での再生**（複数 engine 版が要る。本書は「記録に版を刻む」までで、版違い再生は将来）。

## 1. アーカイブ書式 v1（§7 の宿題「版タプルの棋譜内表現」を確定する）

UTF-8 テキスト。**ヘッダ（エンベロープの最小形）** ＋ **本体（既存の素の棋譜）** の二部構成。例:

```
fukanzen-shogi-archive 1
rule 0.5
protocol 2
app 0.7.1
sente -
gote -
result mate gote_wins
sfen lnsgkgsnl/1r5b1/ppppppppp/9/9/9/PPPPPPPPP/1B5R1/LNSGKGSNL b - 1
1: 7g7f | 3c3d
2: 2g2f | 8c8d
```

### 1.1 構造規則

- **1行目（マジック＋書式版・必須）**: `fukanzen-shogi-archive <書式版>`。`<書式版>` は整数（現状 **1**）。この行の有無で「アーカイブか、旧来の素の棋譜か」を判別する。
- **ヘッダ行**: `sfen ` で始まる行に到達するまでが**ヘッダ**。各行は `<キー> <値…>`（空白区切り、先頭トークンがキー）。
  - `#` で始まる行と空行は**コメント**（無視）。
  - **未知のキーは無視**（前方互換。第二歩以降がキーを足しても旧読み手が壊れない）。
- **本体**: `sfen ` 行以降は、**既存 `kifu_to_string` の出力そのまま**（`sfen <初期局面>` ＋ `N: <先手USI> | <後手USI>` 行列）。**本体には一切手を入れない**（既存構造との整合の核）。

### 1.2 既知ヘッダキー

| キー | 例 | 必須 | 意味・出どころ |
|---|---|---|---|
| `fukanzen-shogi-archive` | `1` | ◎ | 書式版（このエンベロープ層自身の版）。 |
| `rule` | `0.5` | ◎ | ルール版。`engine::RULE_VERSION = (major, minor)` を `major.minor` で。**再現に必須**。 |
| `protocol` | `2` | ◎ | プロトコル版。`protocol::PROTOCOL_VERSION`。出自の完全記録（再現には使わない）。 |
| `app` | `0.7.1` | ○ | 配布版。`env!("CARGO_PKG_VERSION")`。出自の補助（任意）。 |
| `sente` | `-` | ◎ | 先手席の対局者識別（principal）。**今は予約**＝`-`（未設定）。先後の割り当てもこの2行が兼ねる。 |
| `gote` | `-` | ◎ | 後手席の対局者識別（principal）。**今は予約**＝`-`。 |
| `result` | `mate gote_wins` | ◎ | 終局（§7 の語彙）。未終局は `unfinished none`。 |

- **出力順は上表の固定順**（決定的・差分が読みやすい）。
- **再現に本当に必要なのは `rule` だけ**。残りは出自・取り扱い・表示のための情報。
- 思想的一貫性: 実行体が版を名乗り（`--version`）、通信が版を名乗り（版交渉）、そして**記録が版を名乗る**（淀川 §3／版図 §3）。版タプルの自己申告が、コードから通信を経てデータにまで及ぶ。

> **中身とエンベロープの最小形**（淀川 §1.4）: `sfen` 以降の本体が**中身**（対局そのもの）、ヘッダが**エンベロープ**（取り扱い層）の芽。今エンベロープに乗るのは版タプル・配布版・席の予約・終局のみ。記録係署名・consent 参照・法的根拠は**まだ乗せない**（席だけ空ける）。

## 2. 版の出どころ（核が自己申告する）

版は**核（engine・protocol）が自己申告する値**であり、表示・保存といった I/O は殻が担う（互換性確認プロトコル §1・§6 の分離原則）。Web に版を**ハードコードしない**——必ず核から取り出す。

- `rule`: `engine::RULE_VERSION`（`(0, 5)`）。
- `protocol`: `protocol::PROTOCOL_VERSION`（`2`）。`protocol::MY_VERSION` が両者を束ねている。
- `app`: `env!("CARGO_PKG_VERSION")`（ワークスペース継承で `0.7.1`）。

> `engine` は `protocol` に依存しない（純粋な核）。したがって `engine` のフォーマット関数はプロトコル版を**引数で受け取る**（§3）。両版を同時に知るのは両者へ依存する殻（`protocol-wasm` および web）。この向きを崩さない。

## 3. engine: フォーマットの正本（新規 `engine/src/archive.rs`）

エンベロープ層は素の SFEN/USI 直列化（`serialize.rs`）とは概念が異なる（中身 vs エンベロープ）。**新規モジュール `engine/src/archive.rs`** に置き、本体の組み立て・解釈は `serialize::{kifu_to_string, kifu_from_string}` を再利用する。`engine/src/lib.rs` に `pub mod archive;` を追加。

### 3.1 型

```rust
// engine/src/archive.rs

/// アーカイブのヘッダ（エンベロープの最小形）。
#[derive(Debug, Clone, PartialEq, Eq)]
pub struct ArchiveMeta {
    pub rule: (u32, u32),          // 既定 engine::RULE_VERSION
    pub protocol: u32,             // 呼び出し側（殻）が protocol::PROTOCOL_VERSION を渡す
    pub app: Option<String>,       // env!("CARGO_PKG_VERSION") 等。None なら app 行を省く
    pub sente: Option<String>,     // principal 予約。None → "-"
    pub gote: Option<String>,      // principal 予約。None → "-"
    pub result: ArchiveResult,
}

#[derive(Debug, Clone, PartialEq, Eq)]
pub struct ArchiveResult { pub kind: ResultKind, pub outcome: Outcome }

#[derive(Debug, Clone, PartialEq, Eq)]
pub enum ResultKind { Mate, KingDeath, SwapDraw, Sennichite, Resign, Unfinished, Other }

#[derive(Debug, Clone, PartialEq, Eq)]
pub enum Outcome { SenteWins, GoteWins, Draw, None }

pub const ARCHIVE_FORMAT_VERSION: u32 = 1;
```

### 3.2 関数

```rust
/// Kifu ＋ メタ → アーカイブ・テキスト（§1 の書式、固定順）。
pub fn kifu_to_archive(kifu: &Kifu, meta: &ArchiveMeta) -> String;

/// アーカイブ・テキスト → (Kifu, ArchiveMeta)。
/// ヘッダの無い旧来の素の棋譜（先頭が `sfen `）も受け入れ、
/// その場合 meta は既定（rule=不明扱い、result=Unfinished/None、席=None）で返す。
/// 解釈不能なら None。
pub fn archive_to_kifu(s: &str) -> Option<(Kifu, ArchiveMeta)>;
```

- `kifu_to_archive`: §1.2 の固定順でヘッダを出力し、続けて `serialize::kifu_to_string(kifu)` を連結。`app`/`sente`/`gote` の `None` の扱いは表のとおり（app は省略可、席は `-`）。
- `archive_to_kifu`: 先頭の非コメント行を見て分岐。
  - `fukanzen-shogi-archive <n>` で始まる → アーカイブ。ヘッダを `sfen ` 行まで読み（`#`・空行は飛ばす、未知キーは無視）、`sfen ` 行以降を `serialize::kifu_from_string` に渡して本体を得る。
  - `sfen ` で始まる → **旧来の素の棋譜**。`serialize::kifu_from_string` に丸投げし、`meta` は既定値。
- **`serialize::kifu_from_string` 自体は変更しない**（本体パーサは不変）。後方互換はアーカイブ層の分岐で吸収する。

### 3.3 値の表現

- `rule` (0,5) ⇄ 文字列 `"0.5"`。読み戻しは `major.minor` をパース。`"x.y"` 以外は None（堅牢に）。
- `result` 行: `<kind> <outcome>`。語彙は §7。`kind` が未知トークンなら `Other` に倒す（壊さない）。
- 決定性: 同じ `(kifu, meta)` から**バイト一致**の出力（往復テストで担保）。

## 4. Wasm: web への橋渡し

### 4.1 `protocol-wasm`: 版タプルの getter（一関数）

`protocol-wasm` は `MY_VERSION` を握り、`env!("CARGO_PKG_VERSION")` で配布版も得られる。版を JSON で返す getter を一つ足す:

```rust
#[wasm_bindgen]
pub fn version_tuple() -> String {
    // 例: {"rule":"0.5","protocol":2,"app":"0.7.1"}
    format!(
        r#"{{"rule":"{}.{}","protocol":{},"app":"{}"}}"#,
        MY_VERSION.rule.0, MY_VERSION.rule.1, MY_VERSION.protocol, env!("CARGO_PKG_VERSION")
    )
}
```

### 4.2 `engine-wasm`: アーカイブ書き出し（JSON リクエストを一つ受ける）

`engine-wasm` は `engine` のみに依存する（プロトコルに依存しない）。版値は**呼び出し側 JS が渡す**（4.1 の `version_tuple` で得たものを載せる）。

```rust
/// JSON リクエストからアーカイブ・テキストを組み立てる。
/// request 例:
/// {
///   "initial_sfen": "lnsgkgsnl/...",
///   "plies": [{"s":"7g7f","g":"3c3d"}, ...],   // USI のみ（表記は再計算可能なので持たない）
///   "rule": "0.5", "protocol": 2, "app": "0.7.1",
///   "sente": "-", "gote": "-",
///   "result": {"kind":"mate","outcome":"gote_wins"}
/// }
/// 返値: 成功 → アーカイブ・テキスト。失敗 → "ERROR: <理由>"（先頭が `fukanzen-shogi-archive` でないことで判別可）。
#[wasm_bindgen]
pub fn build_archive(request_json: &str) -> String;
```

- 実装: `initial_sfen` を `engine::serialize::sfen_to_position` で `Position` に、`plies` を順に積んで `engine::kifu::Kifu` を構成（`Action::from_usi` で各 USI をパース）。`request` の版・席・result から `ArchiveMeta` を作り、`engine::archive::kifu_to_archive` を呼ぶ。
- **本体は USI のみ**を持つ（日本語表記 `sText/gText` は notation で再計算できる冗長情報——中身に入れない）。
- JSON のパース: 既存 wasm が手書き JSON 文字列を組み立てている流儀に倣い、依存を増やさず最小実装でよい（`serde` 導入は任意。最小なら軽量パースで足りる）。

> **読み込み（parse）の Wasm 露出は本書では非ゴール**（第二歩で `parse_archive` を足す）。engine 側の `archive_to_kifu` は往復テストの担保として今実装するが、web の再生導線は次段。

## 5. web: 終局時と「対局中つねに」の保存（`web/board.js`）

保存先は最小では**ローカル**でよい（淀川 §6: ダウンロード／localStorage／コピー可能テキスト）。本書は **(a) ファイルのダウンロード** と **(b) コピー可能テキスト** を必須、localStorage は任意とする。

### 5.1 アーカイブ・テキストの生成

`board.js` に関数を一つ:

```js
function buildArchiveText() {
  const req = {
    initial_sfen: INITIAL_SFEN,
    plies: kifu.plies.map(p => ({ s: p.sUsi, g: p.gUsi })),
    ...versionTuple,                 // {rule, protocol, app}（§5.4 で起動時に取得・キャッシュ）
    sente: "-", gote: "-",           // principal 予約
    result: currentResult(),         // §7
  };
  return build_archive(JSON.stringify(req));   // engine-wasm
}
```

### 5.2 保存 UX

- **「棋譜を保存」** の操作を用意する。押下で:
  - **(a) ダウンロード**: `buildArchiveText()` を `Blob`（`text/plain`）にし、`<a download>` ＋ オブジェクト URL でファイル保存。拡張子は既存に揃え **`.kifu`**（engine の読み手が両書式対応になるので CLI/TUI でも開ける）。ファイル名は衝突しにくく（例 `fukanzen-YYYYMMDD-HHMMSS.kifu`）——既定固定名の上書き事故（保存し損ねの一因）を避ける。
  - **(b) コピー可能テキスト**: 同じテキストを `navigator.clipboard.writeText` でコピー、または読み取り専用 `<textarea>` に表示して手動コピーできるようにする。
- **終局時（自動的に前面化）**: online は `endOnlineGame(msg)`、ホットシートは `game_status(最終SFEN)` が終端を返した時点で、「棋譜を保存」を**目立つ形で提示**する（ブラウザはユーザ操作なしの自動 DL を弾くため、ワンクリックで保存できる導線を明示）。
- **対局中つねに**: 「棋譜を保存」を対局中も**常時押せる**ようにする。未終局でも書き出せる（`result = unfinished none`）。**これが「保存し損ね」への直接の治療**——途中でいつでも確実に手元に残せる。

### 5.3 終局結果（result）の判別

web は終局表示のために終端状態を既に算出している。それを §7 の語彙へ写す（`currentResult()`）:

- **online（投了）**: `handleTurnComplete` が `resign` を検出し勝敗メッセージを作っている。`endOnlineGame` を呼ぶ際、表示用 `msg` と並べて**構造化 result（kind/outcome）も保持**するよう小改修する（`onlineResult` 変数を足す）。投了 → `kind=resign`、勝者に応じて `outcome`。
- **詰み・引き分け**: 最終 SFEN に `game_status` → `sente_loses`/`gote_loses`/`draw` を `kind=mate`（着手不能＝確定的詰み）＋ `outcome`（`gote_wins`/`sente_wins`/`draw`）へ。
- **玉の死**: 最終手の `resolve_ply` の `event`（`sente_died`/`gote_died`/`both_died`）→ `kind=king_death` ＋ `outcome`。
- **未終局**: `kind=unfinished, outcome=none`。
- **判別しきれない終局種別**（千日手・両玉スワップ等を web がまだ区別表示していない場合）: `kind=other` に倒し、`outcome` だけ正しく入れる（壊さない）。`kind` の精緻化は follow-on。

### 5.4 版タプルの取得

起動時（`init`/`initProtocol` 完了後）に `version_tuple()`（§4.1）を一度呼び、パースして `versionTuple = {rule, protocol, app}` をキャッシュ。`board.js` から参照する（`online.js` 経由でも、`protocol-wasm` を直接 import してもよい）。

## 6. principal の席（継ぎ目だけ予約する）

プリンシパル設計 §5「継ぎ目を空け、作り込まない」に忠実に、**席だけ**用意する:

- 書式の `sente`/`gote` ヘッダ＝principal の席。今は常に `-`（未設定）。
- `ArchiveMeta.sente/gote: Option<String>`、`build_archive` の `sente`/`gote` 引数＝その口。
- **今やらないこと**: principal_id の発行、principal_type（signed_in/anonymous/agent）、subject、規約同意との結びつけ、帰属スタンプ、consent receipt 参照。これらは版図／プリンシパルの第一歩＝別指示書で、ここで作った席に後から嵌める（archive(席だけ予約) → principal キー → 中身に principal）。

## 7. 終局種別と勝敗の語彙

`result <kind> <outcome>`。

- `kind` ∈ `mate`（確定的詰み＝着手不能）, `king_death`（玉の死）, `swap_draw`（両玉スワップ引き分け）, `sennichite`（千日手）, `resign`（投了）, `unfinished`（未終局）, `other`（上記に当てはめきれない／判別未実装）。
- `outcome` ∈ `sente_wins`, `gote_wins`, `draw`, `none`（未終局）。
- engine 側の対応（参考。`terminate::GameStatus`/`GameEnd`・`resolve::ResolutionEvent` から）:
  - `GameStatus::SenteLoses` → `mate gote_wins` / `GoteLoses` → `mate sente_wins` / `Draw` → `mate draw`
  - `GameEnd::SenteLoses`（玉の死）→ `king_death gote_wins` / `GoteLoses` → `king_death sente_wins` / `Draw` → `king_death draw`
  - 投了 → `resign <勝者>` / 千日手 → `sennichite draw` / 両玉スワップ → `swap_draw draw`

## 8. テストと受け入れ基準

### 8.1 engine（`archive.rs`、必須）

- **往復**: 任意の `(Kifu, ArchiveMeta)` で `archive_to_kifu(kifu_to_archive(kifu, meta)) == (kifu, meta)`。
- **決定性**: 同じ入力でバイト一致の出力。
- **ヘッダ存在・固定順**: 出力1行目が `fukanzen-shogi-archive 1`、続いて `rule`・`protocol`・（`app`）・`sente`・`gote`・`result`・`sfen` の順。
- **後方互換**: 旧来の素の棋譜（先頭 `sfen `）が `archive_to_kifu` で読め、`Kifu` が一致する。`serialize::kifu_from_string` のテストは不変で通る。
- **前方互換**: ヘッダに未知キー行を差し込んでも `archive_to_kifu` が無視して正しく読む。
- **result 往復**: 全 `kind`×代表 `outcome` がパース往復する。未知 `kind` トークンが `Other` に倒れる。
- **rule パース**: `"0.5"` → `(0,5)`、不正文字列は弾く。

### 8.2 Wasm（必須）

- `protocol-wasm::version_tuple()` が `{"rule":"0.5","protocol":2,"app":"0.7.1"}` を返す（値は定数から）。
- `engine-wasm::build_archive(req)` の出力が `engine::archive::archive_to_kifu` で読め、初期局面・着手列・result が一致する（Rust 側のテストで JSON リクエスト→出力→parse を検証）。不正リクエストは `ERROR:` を返す。

### 8.3 web（受け入れ・手動）

- online を一局終局まで指す → 「棋譜を保存」→ `.kifu` がダウンロードされ、中身に版ヘッダ（`rule 0.5` 等）＋着手列＋正しい `result` が入っている。コピーも効く。
- **対局中**に「棋譜を保存」→ `result unfinished none` で書き出せる。
- 再現: 息子さんとの一局を想定し、**詰みまで指す前でも・指した後でも、棋譜を失わずに手元へ残せる**こと（保存し損ねの治療の確認）。
- 版は web にハードコードされておらず `version_tuple()` 由来であること（コードレビューで確認）。

## 9. 版の刻み

- **配布版 v0.7.1 → v0.8.0**（機能追加・加算的）。`workspace.package.version` を更新し、タグ連動（既存の集約方式）。
- **ルール版・プロトコル版は据え置き**（ルール 0.5 / プロトコル 2）。アーカイブは**対戦互換性に触れない**（ワイヤプロトコル不変）。互換性管理 方針に照らし、対戦非互換を生まない加算機能ゆえメジャー級の刻みは不要。
- **アーカイブ書式版 = 1**（`ARCHIVE_FORMAT_VERSION`）。これはエンベロープ層自身の版で、配布版・ルール版・プロトコル版とは独立に進む。

## 10. 非ゴールと次段への申し送り

- **第二歩（再生／鑑賞）**: `parse_archive` の Wasm 露出と、読み込んだアーカイブを既存の水墨様式の再生・局面図デモへ流し込む導線。本書の `archive_to_kifu` がその土台。
- **第三歩（ライブ観戦）**: Durable Object のブロードキャストとサーバ側アーカイブ書き出し。ここで淀川 §4 の**秘匿境界（ライブ性 vs チート排除）**を実物で決める（未決）。
- **principal の中身**: 版図／プリンシパル設計の第一歩（規約同意＋記名の最小キー）。本書で予約した `sente`/`gote` の席に嵌める。
- **エンベロープの拡張**: 記録係署名（中身の完全性の封蝋）、consent receipt 参照、法的根拠、帰属スタンプ。席（ヘッダのキー追加）は前方互換で足せる。
- **CLI/TUI の書き移行**: engine の読み手が両書式対応になるので CLI/TUI はアーカイブを**読める**。`:save` をアーカイブ書式で**書く**よう揃えるのは任意の follow-on（揃えれば TUI 保存も版を名乗る）。
- **app 版を再現に使う／版違い再生**: 再現に必要なのは rule のみ。複数ルール版での再生は複数 engine 版が要る将来の課題。

## 11. 不変の原則（本実装が守るもの）

1. **版タプルを記録に刻む**: 不完全将棋が安定を得るまで、棋譜は出自（ルール版・プロトコル版）を名乗る（淀川 §8）。
2. **核が形を定め、殻が I/O を担う**: 書式は `engine`、保存は `web`。版は核から自己申告、ハードコードしない。
3. **中身は変えず、エンベロープを足す**: 本体（`sfen`＋着手列）は不変。版・席・終局はヘッダ＝エンベロープに。
4. **継ぎ目を空け、作り込まない**: principal は席だけ予約。加算で足す。
5. **後方・前方互換**: 旧来の素の棋譜を読め、未知ヘッダキーを無視する。
6. **設計は全体、実装は最小から**: 河口はアーカイブ一つから。再生・ライブ・学習は独立に後付けする。
