# 不完全将棋 バックログ（伏線と未決）

三者の共通リスト——**Opus**（伏線を言語化し、設計を詰める）／**あなた**（方向を選ぶ）／**Sonnet**（実装する）。「必要が生じたら足す」で進めてきたので、いま足さないと決めた種を、ここに一箇所へ綴じておく。頭や散らばったチャットに置くと、棋譜と同じく保存し損ねるから。

**使い方**: これは**索引（dashboard）**。項目の詳細は、それが既に棲む文書（ルール仕様 §7・各実装指示書・版図・淀川・プリンシパル）を**指す**。真実は一箇所、ここは参照。どの文書にもまだ家の無い伏線だけ、本文をここに抱える。実感が「これだ」と言った種は、設計を詰めて実装指示書へ昇格させ、済へ移す。自由に追記・剪定してよい。

**状態の印**: 【種】まだ設計を詰めていない ／【待ち】前提が揃うのを待つ ／【進行中】実装指示書が動いている ／【済】着地。

**現在地**: 配布 v0.10.3。記録係一段目＝書庫（対局の永続と身元）が着地（`実装指示書_対局の永続と身元_記録係一段目`）。新KV `ARCHIVES` に確定局は内容ハッシュ、放棄断片は暫定IDで綴じ、`GET /archive/:id` で部屋を介さず取り出せる。「綴じてから拭く」不変条件を終局・再戦・全員離脱の三契機すべてで検証済み（同じ部屋で再戦しても前局は消えない）。v0.10.3 でこの実装自身へのセキュリティレビューを行い、①競合修正が生んでいた退行（`archived` フラグを KV 書き込み成功の前に立てていたため、書き込み失敗がサイレントな記録喪失になりうる作りだった）、②`spectate_turn`/`spectate_result.text` にサーバ側の上限が無かった点、の2種を締めた。

---

## A. 記録係への道（現在のアーク）

「観られる」まで来た観戦の口を、版図の描く**権威なき観察者**へ育てる二段構え。一段目＝存在と身元、その先＝見る目・封蝋・名乗り・時間。

- 【済】**書庫（対局の永続と身元）** — 対局を部屋（使い回す殻）から解き、内容ハッシュの身元で別ストア（KV `ARCHIVES`）へ綴じた（v0.10.2）。終局（内容ハッシュで確定綴じ）・再戦開始・全員離脱（いずれも断片として暫定ID綴じ）の三契機で「綴じてから拭く」を実装・検証済み。`GET /archive/:id` で部屋 DO を介さず取り出し可能。旧クライアント（`text` 非同梱）でも断片として救われる。
- 【待ち】**二証人の交差確認** — 両プレイヤーの正準本文の一致で改竄検知（審判なし）。種は蒔き済み（急所指示書 §5 で片側本文を content-address）。書庫が立ったので、もう一方の本文の hash 一致を要求するだけで育つ。版図「審判なし・侘びで競う」の芯。
- 【待ち】**記録係署名** — content-address した id に完全性の封蝋。内容ハッシュが土台。→ 中身/エンベロープ分離（淀川 §1.4）。
- 【待ち】**principal による帰属（席を埋める）** — 規約同意＋記名の最小キー。→ `プリンシパル設計`・版図 §7。第一歩の予約席 `sente`/`gote` に嵌める。Anonymous と Agent を受け入れる土壌として。
- 【待ち】**書庫の一覧・鑑賞・検索** — 過去局ブラウザ。KV の prefix 列挙で素直に。→ 急所指示書 §11。※`archived{id}` はサーバから配信済み・`online.js` の受信配線もあるが、`board.js` 側に `onArchived` を受ける先が無く、id を画面に出す・コピーする導線は現状ゼロ（v0.10.3 時点で確認）。この最小UIを単独で先に足す案も出たが、鑑賞機能が来るまで待つと判断（2026-07-04）。
- 【待ち】**保持ポリシー・匿名化・再識別性** — 削除＝Anonymous 化では再識別が残る（プリンシパル §3.3 が版図 §2.4 を修正）。→ 名乗りの層と同時。

## B. ルールのプレイテスト待ち

いずれも机上で動かすと過剰。実戦データ（人・AI 双方）が signal を出したら次版で手当て。**詳細と正本はルール仕様 v0.6 §7**。

- 【待ち】確定的詰みの実現可能性（実戦で成立しにくくないか）
- 【待ち】大駒の打ち込みの強さ（積極性への報酬 vs 過剰圧力）
- 【待ち】玉の死の残存経路（玉を動かさない賭けの裏目・合駒貫き、経路非干渉 4.6 への依存）
- 【待ち】両玉隣接と一騎打ち（豊かな終盤 vs 相討ち引き分け頻発）
- 【待ち】打ち歩詰めの、確率的王手下での意味づけ
- 【待ち】引き分けの非対称化（方針3、先送り。逃げ込み濫用が出たら）
- 【待ち】入玉・持将棋・入玉宣言法（ルール v0.6 5.9。点数法と宣言手続きの手番読み替え）

## C. プロダクト／UI（回収したい伏線）

- 【種】**チュートリアル** — 不完全将棋が人を驚かせる瞬間＝**同時着手の反直感**（両玉スワップの相討ち・戦国無双特則で裸の駒を斬る玉・合駒貫き・経路非干渉）を、口で説くより**その局面を再生して一手指させる**。原理的には「教材アーカイブ（版タプル付き）数枚＋読み込み導線」で組める。教材アーカイブは書庫（A）に置ける——伏線と本筋が同じ器を共有する。
- 【待ち】**一人指しの検証盤** — v0.9.2 でボタン廃止（commit 9488599）。機能は核に残る（任意アーカイブを読み込み、その局面から盤クリックで分岐再生＝第二歩 `loadPlies`）。チュートリアルの「教材局面プレイグラウンド」として自然に戻る。
- 【待ち】**デモ局面再生** — v0.9.2 でボタン廃止。棋譜再生で賄い済み。チュートリアル導線で回収候補。

## D. 基盤／インフラ

- 【待ち】**移植の抽象化** — DO を離れるときは、四つの汎用プリミティブ（部屋ごとの調整点・追記履歴・永続 KV・キー routing）を契約にインターフェース抽出。→ 第三歩 §9。二つ目の基盤が**現実の必要になったとき**初めて作る（過ぎたるは及ばざる）。等価物: Elixir/Phoenix、単一プロセス＋部屋 Map、Colyseus、Akka/Orleans。
- 【待ち】**CLI/TUI の載せ替え** — 凍結中。Web 盤の概念が実装レベルで固まるまで保留（作り手の判断）。engine の終局判定・アーカイブ書式は既に使える状態。

## E. 大きな未決（別の畑）

- 【種】**時間制** — 人と AI の速度差が牙を剥く。持ち時間なしを超える層。ライブでの停滞・速度差の本格対応。→ 原引き継ぎ・ルール v0.6 §7・第三歩 §8（接続レベルの放棄のみ現状対応）。
- 【種】**CPU 対戦** — 不完全情報 AI、「人間臭く」。作り手の動機の核（ドラえもんとのび太、完璧でない相棒）。
- 【種】**エコシステムの汎化** — チェス等への応用。「手番だけを引く」操作が他ゲームで何を生むか。
- 【種】**淀川 第四歩：機械学習ログ** — 蓄積の河口。→ 淀川 §6。合法手集合・思考時間など機械学習用の追加文脈。

## F. 済・解決済み（再燃防止の記録）

- 【済】データ喪失（棋譜が残らない）— 第一歩で保存、第二歩で再生。※サーバ層での再発は A の書庫で閉じる。
- 【済】版タプルの記録への刻印・版交渉・版不一致注意 — 第一歩〜、ルール 0.6 で初めて実地に機能（v0.9.0）。
- 【済】アーカイブ読込の DoS（組手数・制御文字）— v0.8.2。安全網の重複 500 は v0.9.0 で `engine::terminate::MAX_TURNS` へ一本化。
- 【済】**ライブ観戦の秘匿境界** — commit-reveal が構造的に溶かす（隠すのは状態でなく「次の一手」）。観戦は読み取り専用・公開組手のみ・back-channel なしで by construction 安全、**遅延不要**。→ 第三歩 §1。観戦者の request_reset 経路も塞いだ（v0.10.1）。
- 【済】**観戦の口を開けたことで再発したデータ喪失** — 部屋の記録は再戦（`spectate_meta`）で拭かれる作りだったため、観戦を実装した第三歩で棋譜喪失が一段上で再発していた。記録係一段目（書庫・内容ハッシュ）で閉じた（v0.10.2）。
- 【済】**書庫の競合修正が生んだ退行、および上限欠如** — v0.10.2 の「二重書き込みを避ける」修正が、`archived` フラグを KV 書き込み成功より前に立てる作りになっており、書き込み失敗時にサイレントな記録喪失を招きうる状態だった（無害な競合を、この機能が防ぐはずだった致命的な喪失と交換してしまっていた）。あわせて `spectate_turn`（着手数）・`spectate_result.text`（本文サイズ）にサーバ側の上限が無かった点も発見。実装直後のセキュリティ自己レビューで発見・修正（v0.10.3）。
