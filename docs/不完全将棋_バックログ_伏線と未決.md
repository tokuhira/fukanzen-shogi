## A. 記録係への道（現在のアーク）

「観られる」まで来た観戦の口を、版図の描く**権威なき観察者**へ育てる二段構え。一段目＝存在と身元、その先＝見る目・封蝋・名乗り・時間。

- 【済】**書庫（対局の永続と身元）** — 対局を部屋（使い回す殻）から解き、内容ハッシュの身元で別ストア（KV `ARCHIVES`）へ綴じた（v0.10.2）。終局・再戦開始・全員離脱の三契機で「綴じてから拭く」を実装・検証済み。`GET /archive/:id` で取り出し可能。→ `archive/implementation/記録係一段目`。
- 【済】**記録係の招待と二証人の交差確認** — 相互同意のオプトイン（`record_invite`/`accept`/`decline`）で招かれた対局のみ綴じる。両プレイヤーの証言（`record_testimony`）の内容ハッシュ一致で `witnesses:2` 確定綴じ、不一致は裁定せず `record_disagreement` で両証言を保存・surface（v0.11.0）。未招待は揮発（意図的）。※既知の制約（意図的に許容, 2026-07-05）: 招待の提案・承諾は `window.confirm()` に依り、主要ブラウザは非アクティブタブの `confirm()` を即座に自動キャンセルする。別端末・別人の対戦では問題にならないが、同一端末 2 タブ対局には影響（コードの不具合でなく `confirm()` の仕様上の限界）。→ `archive/implementation/記録係二段目`。
- 【待ち】**記録係署名** — content-address した id・二証人一致に、記録係の署名（完全性の封蝋）。二証人が第一の防波堤、署名はその上。→ 中身/エンベロープ分離（淀川 §1.4）。
- 【待ち】**principal による帰属（席を埋める）** — 規約同意＋記名の最小キー。招待/承諾が principal と consent 参照を運ぶ受け皿になる。→ `design/プリンシパル設計`・版図 §7。第一歩の予約席 `sente`/`gote` に嵌める。Anonymous と Agent を受け入れる土壌として。
- 【種】**記録係の中立性・複数記録係** — 公式記録係（DO）が嘘を綴じないと誰が保証するか。二証人が第一の防波堤だが、究極的には複数記録係の突き合わせ・記録係署名・検証可能な記録が要る。版図「正体を問わない世界」の遠い地平（記録係二段目 §13）。
- 【待ち】**書庫の一覧・鑑賞・検索** — 過去局ブラウザ。KV の prefix 列挙で素直に。`archived{id}` の最小 surface（リンク表示・コピー）は記録係二段目で着地済み、完全な導線（一覧・鑑賞）はここで待つ。
- 【待ち】**保持ポリシー・匿名化・再識別性** — 削除＝Anonymous 化では再識別が残る（プリンシパル §3.3 が版図 §2.4 を修正）。→ 名乗りの層と同時。
- 【済】**TUI クラウド対局の生観戦（延長4b改訂）** — 通信核の一本化 第四段（v0.12.0）で TUI が DO の部屋へ入れるようになった後も、`spectate_meta`/`spectate_turn`/`spectate_result` の送出は先手の web クライアントだけが担う実装のままだった。TUI が先手でクラウド対局するとき同じ3種を送出するようにし、`/watch` の観戦者が生で見られるようになった（`spectate_result` は単一正本 `protocol::game_result` を直接呼ぶ）。実 DO で meta/turn/result の生配信を確認済み（v0.12.3）。→ `archive/implementation/不完全将棋_実装指示書_延長4b改訂_TUI先手の観戦配信_game_result直呼び.md`。
- 【待ち】**TUI クラウド対局の記録係対応** — 記録係の招待/受諾フロー（`record_invite`/`record_accept`/`record_testimony`）は web クライアントだけが実装しており、TUI は DO からの `record_*` メッセージを未知として無視する。web 側の相手が招待を送っても TUI は応答できず、招待は成立しない——**TUI が関与するクラウド対局は記録係アーカイブの対象にならない**（対局・生観戦は成立する。延長4bで解消済み）。意図的に見送った（記録係アークと結合すると重くなるため・クラウド主導で Web/TUI 双方を見据える別アークで拾う）。→ `archive/protocol-unification_総括_第一段から第四段` §3。

- 【済】**board.js I/O 分解** — `handleTurnComplete`・`enterWatchMode`・`endOnlineGame` に埋め込まれていた純粋な判断ロジック（投了判定・meta マッピング・archivedLink 導出・終局 patch）を `web/reducers.js` の純粋 reduce（`turnCompleteDecision`・`metaToLoadedMeta`・`archivedLinkFor`・`endGameReduce`）へ抜いた（IO-1〜IO-3、web `?v=` 0.12.5→0.12.8、配布据え置き）。本質的 I/O（wasm 呼び出し・WS 送信・disconnect・記録係待ちの setTimeout）は無変更で殻に残した。web テスト 118→131 件。**board.js の残り本丸（view 純粋化＋I/O 分解）がこれで尽きた**。→ `archive/board.js_IO分解_総括_IO1からIO3`。
- 【済】**board.js view 純粋化** — `render()` の「state → 表示値」の導出（ラベル・ボタン・overlay・cursor）を `web/view-model.js` の純粋関数群（`labelView`・`buttonView`・`overlay`・`cursorInteractive`・合成 `viewModel`）へ抜き出した（View-1〜View-3、web `?v=` 0.12.3→0.12.5、配布据え置き）。`render()` は「wasm 依存の pos・gameOver を作る→ `viewModel` を一度呼ぶ→ DOM に流す」薄い殻になった。board.js 1263→1130 行、web テスト 57→118 件。ルール変更で終局種別や phaseText が増えても golden snapshot で守れる地盤ができた。→ `archive/board.js_view純粋化_総括_View1からView3`。
- 【済】**終局判定の単一正本化** — TUI と web が別々に持っていた終局判定ロジック（盤面終局のマッピング・投了の勝敗判定）を `engine::terminate::terminal_to_result` と `protocol::game_result` へ一本化した（Step A〜D、配布 v0.12.1→v0.12.2）。TUI の `resolve_turn` が手組み判定でなく `game_result` を呼ぶようになったことで、**最長手数500組手で終局しないバグ**（クラウドで web と desync する実害あり）を構造的に塞いだ。投了の勝敗判定（TUI online・TUI local・web の三箇所で乖離していた）も単一正本へ寄せた。既知の限界は §A（TUI クラウド対局の生観戦・記録係対応＝別畝4b、このアークでは未着手のまま）。→ `archive/終局判定の単一正本化_総括_StepAからD`。
- 【済】**通信核の一本化** — TUI（LAN・TCP）と web（DO・WS）が別々に持っていたワイヤ語彙とセッション進行を `protocol` 核（`WireMessage`・`ClientSession`）へ一本化し、TUI がクラウドの部屋へ入って web と同じ盤へ座れるようになった（第一段〜第四段、配布 v0.11.2→v0.12.0、PROTOCOL 4→5）。LAN は無変更で併存。既知の限界（TUI 先手のクラウド対局が観戦されない）は延長4b（v0.12.3）で解消、記録係対応は §A に残る。→ `archive/protocol-unification_総括_第一段から第四段`。
- 【待ち】**移植の抽象化** — DO を離れるときは、四つの汎用プリミティブ（部屋ごとの調整点・追記履歴・永続 KV・キー routing）を契約にインターフェース抽出。→ 第三歩 §9。二つ目の基盤が**現実の必要になったとき**初めて作る（過ぎたるは及ばざる）。等価物: Elixir/Phoenix、単一プロセス＋部屋 Map、Colyseus、Akka/Orleans。通信核の一本化（本節上）はこの抽象化の前段であり、DO 固有の四プリミティブそのものにはまだ触れていない。

## E. 大きな未決（別の畑）

*ルール版の系譜（レガシー対局）と対局モード（時間制）は、方針文書 `design/不完全将棋_ルール版の系譜と対局モード_方針` に背骨がある。以下はその接ぎ穂と、まだ設計を詰めていない種。*

- 【待ち】**ルール版の系譜＝レガシールール対局** — ルールが次版へ上がったとき旧版を葬らず、再生だけでなく**選んで指せる**ようにする（「懐かしいね、v0.5 で指してみよう」）。要石は engine の版ディスパッチ（`RULE_VERSION` 定数→版で解決を選ぶ形）、完全一致交渉を**版選択**へ拡張（範囲許容ではない）。発火は**最初のルール変更に付随**（今は版が一つで過ぎたる）。淀川 §3 の考古学を実装で裏打ちする。→ 方針文書 §1。
- 【種】**時間制** — 人と AI の速度差が牙を剥く。持ち時間なしを超える層。**対局モードとしての背骨は方針文書 §2**（時間切れ＝`protocol::game_result` の第三の終局ソース＝投了と同じ座・本流の思考時間席〔淀川 §1.2〕・PROTOCOL 能力／セッション設定の二段）。→ ルール v0.6 §7・第三歩 §8（接続レベルの放棄のみ現状対応）。
- 【種】**CPU 対戦** — 不完全情報 AI、「人間臭く」。作り手の動機の核（ドラえもんとのび太、完璧でない相棒）。
- 【種】**エコシステムの汎化** — チェス等への応用。「手番だけを引く」操作が他ゲームで何を生むか。**多変種ホスト・版ディスパッチは方針文書 §1・§3 が前段**（版タプルを変種セレクタとする）。移植の抽象化（§D）と地続き。
- 【種】**淀川 第四歩：機械学習ログ** — 蓄積の河口。→ 淀川 §6。合法手集合・思考時間など機械学習用の追加文脈。

## 使い方（末尾・参照用）

三者の共通リスト——**Opus**（伏線を言語化し設計を詰める）／**あなた**（方向を選ぶ）／**Sonnet・Haiku**（実装する）。「必要が生じたら足す」で進めるので、いま足さないと決めた種をここへ綴じる。頭や散らばったチャットに置くと、棋譜と同じく保存し損ねるから。

- **これは索引**。項目の詳細は、それが棲む文書（ルール仕様 §7・各実装指示書・版図・淀川・プリンシパル・総括）を**指す**。真実は一箇所、ここは参照。どの文書にもまだ家の無い伏線だけ、本文をここに抱える。
- **状態の印**: 【種】まだ設計を詰めていない ／【待ち】前提が揃うのを待つ ／【進行中】実装指示書が動いている ／【済】着地（済んだものは総括・指示書へ逃がし、ここには残さないか一行で参照）。
- 実感が「これだ」と言った種は、設計を詰めて実装指示書へ昇格させる。自由に追記・剪定してよい。
- **完了記録の逃がし先**: アークが区切れたら、完了の詳細は総括文書（`archive/`）へ移し、バックログには「開いている次の畝」だけを残す。解決済みの再燃防止記録も総括の付録（解決済み台帳）へ。
