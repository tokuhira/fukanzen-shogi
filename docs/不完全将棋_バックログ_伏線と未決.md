# 不完全将棋 バックログ（伏線と未決）

三者の共通リスト——**Opus**（伏線を言語化し、設計を詰める）／**あなた**（方向を選ぶ）／**Sonnet**（実装する）。「必要が生じたら足す」で進めてきたので、いま足さないと決めた種を、ここに一箇所へ綴じておく。頭や散らばったチャットに置くと、棋譜と同じく保存し損ねるから。

**使い方**: これは**索引（dashboard）**。項目の詳細は、それが既に棲む文書（ルール仕様 §7・各実装指示書・版図・淀川・プリンシパル）を**指す**。真実は一箇所、ここは参照。どの文書にもまだ家の無い伏線だけ、本文をここに抱える。実感が「これだ」と言った種は、設計を詰めて実装指示書へ昇格させ、済へ移す。自由に追記・剪定してよい。

**状態の印**: 【種】まだ設計を詰めていない ／【待ち】前提が揃うのを待つ ／【進行中】実装指示書が動いている ／【済】着地。

**現在地**: 配布 v0.11.2。記録係二段目＝招待と二証人（`実装指示書_記録係の招待と二証人_記録係二段目`）が着地（v0.11.0）。対局は相互同意のオプトイン（`record_invite`/`accept`/`decline`）で招かれた場合のみ書庫へ綴じられ、終局時に両プレイヤーが証言（`record_testimony`）として正準本文を送り、内容ハッシュが一致すれば `witnesses:2` で確定綴じ、不一致なら裁定せず `record_disagreement` として両証言を保存・surface する。未招待の対局は終局しても揮発する（意図的な選択）。`archived{id}` の最小 surface（リンク表示・コピー）も本段で初めて画面に出た。PROTOCOL_VERSION 3→4。ローカル検証（生WSスクリプト＋Playwright実ブラウザ）で2件の実装バグを発見・修正済み——① 二証人の突き合わせが `getWebSockets("player")` の現在接続と照合していたため、証言送信直後に自分から切断する実際のクライアント挙動では二人目の証言が届く頃に一人目が既に居なくなっており一致判定が起きなかった（Map の size で判定するよう修正）、② 綴じの結果（archived/record_disagreement）は非同期のためクライアントが証言送信後すぐ切断すると通知を受け取れず終わっていた（結果を受け取るか保険のタイムアウトまで切断を待つよう修正）。v0.11.1 で、実装直後の振り返りにより見つかった冗長な状態を2件削除: `Testimony.kind`/`.outcome`（正準本文に既に埋め込まれ、一致判定・綴じのどちらでも参照しない書き込み専用フィールド）、`online.js` の `_recordInvitePending`/`isRecordInvitePending`（呼び出し元が存在しない死んだ export）。いずれも将来必要になれば作り直す前提で削除（挙動は無変更）。続けて**足場の整備**（`実装指示書_足場の整備_CI_serverテスト_docs索引`）に着手: CI門番（`cargo fmt`/`clippy -D warnings`/`cargo test` と server の `tsc`/`vitest` を push/PR で自動実行）・server の単体テスト20本（`server/src/logic.ts` へ純粋ロジックを抽出——秘匿境界のrouting・二証人の評決・content-address・envelope構築・「綴じてから拭く」ゲート）・`docs/README.md` 索引（生きた設計／歴史／完了記録の仕分け）・ルール確定後の取り残しコメント一掃（CLI の千日手コメント、README の「未確定事項」リスト・stale なテスト数）が対象。製品挙動は不変のため配布版は据え置き（指示書 §6 の既定どおり）。続けて**盤面解釈を核へ返す（board.js 分割 第〇段）**（`実装指示書_盤面解釈を核へ返す_board分割第零段`）に着手・着地: web/board.js の自前 `parseSfen`（SFEN 解釈の JS 側二重定義）を削除し、engine-wasm に新設した `position_view` getter（`engine::serialize::sfen_to_position` を再利用。抽出した `piece_kind_char`/`piece_view_kind` は `position_to_sfen` とも共有）へ置換。消費者の `{board:Map, handS, handG}` 形・`renderSvg`/クリック処理は無変更。JSON view を消費者形へ組み替える純粋アダプタ `positionViewToState` は `web/position-view.js` へ分離し、DOM にもWasmにも触れないため、初の web テスト（vitest）をここに立てた。CI にも `web` ジョブを追加（`server` ジョブと並行）。protocol-wasm は `CARGO_PKG_VERSION` を埋め込むため合わせて再ビルド。v0.11.2（Cloudflare Pages へ本番デプロイ済み）。バックログ D の board.js 分割の項へ状態を刻んだ——第〇段 済、第一段（純粋 JS 切り出し＋Wasm-in-node テスト足場）・第二段（状態を持つ分割）は引き続き待ち。

## A. 記録係への道（現在のアーク）

「観られる」まで来た観戦の口を、版図の描く**権威なき観察者**へ育てる二段構え。一段目＝存在と身元、その先＝見る目・封蝋・名乗り・時間。

- 【済】**書庫（対局の永続と身元）** — 対局を部屋（使い回す殻）から解き、内容ハッシュの身元で別ストア（KV `ARCHIVES`）へ綴じた（v0.10.2）。終局（内容ハッシュで確定綴じ）・再戦開始・全員離脱（いずれも断片として暫定ID綴じ）の三契機で「綴じてから拭く」を実装・検証済み。`GET /archive/:id` で部屋 DO を介さず取り出し可能。旧クライアント（`text` 非同梱）でも断片として救われる。
- 【済】**記録係の招待と二証人の交差確認** — 相互同意のオプトイン（`record_invite`/`accept`/`decline`）で招かれた対局のみ綴じる。終局時に両プレイヤーが証言（`record_testimony`）を送り、内容ハッシュ一致で `witnesses:2` の確定綴じ、不一致は裁定せず `record_disagreement` として両証言を保存・surface（v0.11.0）。未招待は終局しても揮発（意図的な選択）。`archived{id}` の最小 surface もここで着地。※既知の制約（意図的に許容、2026-07-05）: 招待の提案・承諾は `window.confirm()` に依っており、主要ブラウザは非アクティブ（バックグラウンド）タブから呼ばれた `confirm()` を画面表示せず即座に自動キャンセル（＝辞退）する。実際に別々の端末・別々の人が対戦する場合は各自の画面が常にアクティブなので問題にならないが、同一端末で2タブ・2ウィンドウを使う対局には現状の設計のまま影響する（先手タブが非アクティブだと後手の招待提案が本人の目に触れず自動辞退される）。コード上の不具合ではなく `confirm()` の仕様上の限界。
- 【待ち】**記録係署名** — content-address した id・二証人一致に、記録係の署名（完全性の封蝋）。二証人が第一の防波堤、署名はその上。→ 中身/エンベロープ分離（淀川 §1.4）。
- 【待ち】**principal による帰属（席を埋める）** — 規約同意＋記名の最小キー。招待/承諾が principal と consent 参照を運ぶ受け皿になる。→ `プリンシパル設計`・版図 §7。第一歩の予約席 `sente`/`gote` に嵌める。Anonymous と Agent を受け入れる土壌として。
- 【種】**記録係の中立性・複数記録係** — 公式記録係（DO）が嘘を綴じないと誰が保証するか。二証人が第一の防波堤だが、究極的には複数記録係の突き合わせ・記録係署名・検証可能な記録が要る。版図「正体を問わない世界」の遠い地平（記録係二段目 §13 より）。
- 【待ち】**書庫の一覧・鑑賞・検索** — 過去局ブラウザ。KV の prefix 列挙で素直に。→ 急所指示書 §11。`archived{id}` の最小 surface（リンク表示・コピー）は記録係二段目で着地済み。完全な導線（過去局一覧・鑑賞）はここで引き続き待つ。
- 【待ち】**保持ポリシー・匿名化・再識別性** — 削除＝Anonymous 化では再識別が残る（プリンシパル §3.3 が版図 §2.4 を修正）。→ 名乗りの層と同時。

## B. ルールのプレイテスト待ち

いずれも机上で動かすと過剰。実戦データ（人・AI 双方）が signal を出したら次版で手当て。**詳細と正本はルール仕様 v0.6 §7**。

- 【待ち】確定的詰みの実現可能性（実戦で成立しにくくないか）
- 【待ち】大駒の打ち込みの強さ（積極性への報酬 vs 過剰圧力）
- 【待ち】玉の死の残存経路（玉を動かさない賭けの裏目・合駒貫き、経路非干渉 4.6 への依存）
- 【待ち】両玉隣接と一騎打ち（豊かな終盤 vs 相討ち引き分け頻発）
- 【待ち】打ち歩詰めの、確率的王手下での意味づけ
- 【待ち】引き分けの非対称化（方針3、先送り。逃げ込み濫用が出たら）
- 【待ち】入玉・持将棋・入玉宣言法（ルール v0.6 5.9。点数法と宣言手続きの手番読み替え）

## C. プロダクト／UI（回収したい伏線）

- 【種】**チュートリアル** — 不完全将棋が人を驚かせる瞬間＝**同時着手の反直感**（両玉スワップの相討ち・戦国無双特則で裸の駒を斬る玉・合駒貫き・経路非干渉）を、口で説くより**その局面を再生して一手指させる**。原理的には「教材アーカイブ（版タプル付き）数枚＋読み込み導線」で組める。教材アーカイブは書庫（A）に置ける——伏線と本筋が同じ器を共有する。
- 【待ち】**一人指しの検証盤** — v0.9.2 でボタン廃止（commit 9488599）。機能は核に残る（任意アーカイブを読み込み、その局面から盤クリックで分岐再生＝第二歩 `loadPlies`）。チュートリアルの「教材局面プレイグラウンド」として自然に戻る。
- 【待ち】**デモ局面再生** — v0.9.2 でボタン廃止。棋譜再生で賄い済み。チュートリアル導線で回収候補。

## D. 基盤／インフラ

- 【待ち】**移植の抽象化** — DO を離れるときは、四つの汎用プリミティブ（部屋ごとの調整点・追記履歴・永続 KV・キー routing）を契約にインターフェース抽出。→ 第三歩 §9。二つ目の基盤が**現実の必要になったとき**初めて作る（過ぎたるは及ばざる）。等価物: Elixir/Phoenix、単一プロセス＋部屋 Map、Colyseus、Akka/Orleans。
- 【待ち】**CLI/TUI の載せ替え** — 凍結中。Web 盤の概念が実装レベルで固まるまで保留（作り手の判断）。engine の終局判定・アーカイブ書式は既に使える状態。
- 【待ち】**board.js の分割（方針相談待ち）** — 三歩＋記録＋招待で肥大した最大のファイル（盤描画・再生・ホットシート・オンライン・観戦・書庫・招待が同居。1500行超）。どう分けるか（責務境界・モジュール構成・型の導入可否）は方針から相談したい。→ `足場の整備` §7。CI門番・server テストが立ったので、分割で何かを壊しても即座に分かる状態にはなった——整備が分割の安全網。**第〇段（盤面解釈を核へ返す）済**——自前 `parseSfen` を engine-wasm の `position_view` getter へ置換、純粋アダプタ `positionViewToState` を `position-view.js` に分離し初の web テスト（vitest）を据えた。v0.11.2。第一段（純粋 JS の切り出し＋Wasm-in-node のテスト足場）・第二段（状態を持つ分割）は引き続き待ち。

## E. 大きな未決（別の畑）

- 【種】**時間制** — 人と AI の速度差が牙を剥く。持ち時間なしを超える層。ライブでの停滞・速度差の本格対応。→ 原引き継ぎ・ルール v0.6 §7・第三歩 §8（接続レベルの放棄のみ現状対応）。
- 【種】**CPU 対戦** — 不完全情報 AI、「人間臭く」。作り手の動機の核（ドラえもんとのび太、完璧でない相棒）。
- 【種】**エコシステムの汎化** — チェス等への応用。「手番だけを引く」操作が他ゲームで何を生むか。
- 【種】**淀川 第四歩：機械学習ログ** — 蓄積の河口。→ 淀川 §6。合法手集合・思考時間など機械学習用の追加文脈。

## F. 済・解決済み（再燃防止の記録）

- 【済】データ喪失（棋譜が残らない）— 第一歩で保存、第二歩で再生。※サーバ層での再発は A の書庫で閉じる。
- 【済】版タプルの記録への刻印・版交渉・版不一致注意 — 第一歩〜、ルール 0.6 で初めて実地に機能（v0.9.0）。
- 【済】アーカイブ読込の DoS（組手数・制御文字）— v0.8.2。安全網の重複 500 は v0.9.0 で `engine::terminate::MAX_TURNS` へ一本化。
- 【済】**ライブ観戦の秘匿境界** — commit-reveal が構造的に溶かす（隠すのは状態でなく「次の一手」）。観戦は読み取り専用・公開組手のみ・back-channel なしで by construction 安全、**遅延不要**。→ 第三歩 §1。観戦者の request_reset 経路も塞いだ（v0.10.1）。
- 【済】**観戦の口を開けたことで再発したデータ喪失** — 部屋の記録は再戦（`spectate_meta`）で拭かれる作りだったため、観戦を実装した第三歩で棋譜喪失が一段上で再発していた。記録係一段目（書庫・内容ハッシュ）で閉じた（v0.10.2）。
- 【済】**書庫の競合修正が生んだ退行、および上限欠如** — v0.10.2 の「二重書き込みを避ける」修正が、`archived` フラグを KV 書き込み成功より前に立てる作りになっており、書き込み失敗時にサイレントな記録喪失を招きうる状態だった（無害な競合を、この機能が防ぐはずだった致命的な喪失と交換してしまっていた）。あわせて `spectate_turn`（着手数）・`spectate_result.text`（本文サイズ）にサーバ側の上限が無かった点も発見。実装直後のセキュリティ自己レビューで発見・修正（v0.10.3）。
- 【済】**二証人の突き合わせが接続の現存を前提にしていた** — 記録係二段目の実装直後、Playwright での実ブラウザ検証で発覚。証言の収集を `getWebSockets("player")` との突き合わせで判定していたが、実際のクライアントは証言送信直後に自分から切断するため、二人目の証言が届く頃には一人目の ws が既に「player」から外れており、二人分揃っても一致判定に進めなかった（生 WS のスクリプトでは見えず、実ブラウザの資源で初めて再現した）。`Map` の `size` で判定するよう修正。あわせて、綴じの結果（`archived`/`record_disagreement`）は非同期（ハッシュ計算・KV 書き込み）のため、証言送信後すぐに切断する実装のままでは通知がほぼ常に届かない構造的な穴も発見——結果を受け取るか保険のタイムアウトまで切断を待つようクライアントを修正した（v0.11.0）。
