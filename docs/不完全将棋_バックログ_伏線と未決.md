## A. 記録係への道（現在のアーク）

「観られる」まで来た観戦の口を、版図の描く**権威なき観察者**へ育てる二段構え。一段目＝存在と身元、その先＝見る目・封蝋・名乗り・時間。

- 【済】**書庫（対局の永続と身元）** — 対局を部屋（使い回す殻）から解き、内容ハッシュの身元で別ストア（KV `ARCHIVES`）へ綴じた（v0.10.2）。終局・再戦開始・全員離脱の三契機で「綴じてから拭く」を実装・検証済み。`GET /archive/:id` で取り出し可能。→ `archive/implementation/記録係一段目`。
- 【済】**記録係の招待と二証人の交差確認** — 相互同意のオプトイン（`record_invite`/`accept`/`decline`）で招かれた対局のみ綴じる。両プレイヤーの証言（`record_testimony`）の内容ハッシュ一致で `witnesses:2` 確定綴じ、不一致は裁定せず `record_disagreement` で両証言を保存・surface（v0.11.0）。未招待は揮発（意図的）。※既知の制約（意図的に許容, 2026-07-05）: 招待の提案・承諾は `window.confirm()` に依り、主要ブラウザは非アクティブタブの `confirm()` を即座に自動キャンセルする。別端末・別人の対戦では問題にならないが、同一端末 2 タブ対局には影響（コードの不具合でなく `confirm()` の仕様上の限界）。→ `archive/implementation/記録係二段目`。
- 【待ち】**記録係署名** — content-address した id・二証人一致に、記録係の署名（完全性の封蝋）。二証人が第一の防波堤、署名はその上。→ 中身/エンベロープ分離（淀川 §1.4）。
- 【待ち】**principal による帰属（席を埋める）** — 規約同意＋記名の最小キー。招待/承諾が principal と consent 参照を運ぶ受け皿になる。→ `design/プリンシパル設計`・版図 §7。第一歩の予約席 `sente`/`gote` に嵌める。Anonymous と Agent を受け入れる土壌として。
- 【種】**記録係の中立性・複数記録係** — 公式記録係（DO）が嘘を綴じないと誰が保証するか。二証人が第一の防波堤だが、究極的には複数記録係の突き合わせ・記録係署名・検証可能な記録が要る。版図「正体を問わない世界」の遠い地平（記録係二段目 §13）。
- 【待ち】**書庫の一覧・鑑賞・検索** — 過去局ブラウザ。KV の prefix 列挙で素直に。`archived{id}` の最小 surface（リンク表示・コピー）は記録係二段目で着地済み、完全な導線（一覧・鑑賞）はここで待つ。
- 【待ち】**保持ポリシー・匿名化・再識別性** — 削除＝Anonymous 化では再識別が残る（プリンシパル §3.3 が版図 §2.4 を修正）。→ 名乗りの層と同時。
- 【待ち】**TUI クラウド対局の生観戦・記録係対応（別畝 4b）** — 通信核の一本化 第四段（v0.12.0）で TUI が DO の部屋へ入れるようになったが、`spectate_meta`/`spectate_turn`/`spectate_result` の送出と記録係の招待/受諾フローは先手の web クライアントだけが担う実装のまま——**TUI が先手のクラウド対局は、まだ生観戦・アーカイブされない**（対局自体は成立する。game channel は DO が relay するので影響はない）。意図的に見送った（記録係アークと結合すると重くなるため）。→ `archive/protocol-unification_総括_第一段から第四段` §3。

- 【待ち】**board.js 分割の次アーク** — アーク第〇〜三段b-3 は完了（総括 `archive/board-split_総括_第零段から第三段b-3`）。残る本丸は二つ、いずれも設計相談から・地面を測り直してから: **種類2（I/O 絡みの分解）** ＝ `handleTurnComplete`・`enterWatchMode`・`endOnlineGame` を tui の `online.rs` に倣い純粋遷移と I/O に割る（非同期コールバックの状態更新をどう純粋遷移に括り出すかが核）。**view 純粋化** ＝ render() 本体（DOM 書き込み 20 箇所超）の phaseText/ボタン分岐を「`state` スナップショット→表示値」の純粋関数へ（`navView()` が芽）。golden snapshot への局面追加もここで。
- 【待ち】**移植の抽象化** — DO を離れるときは、四つの汎用プリミティブ（部屋ごとの調整点・追記履歴・永続 KV・キー routing）を契約にインターフェース抽出。→ 第三歩 §9。二つ目の基盤が**現実の必要になったとき**初めて作る（過ぎたるは及ばざる）。等価物: Elixir/Phoenix、単一プロセス＋部屋 Map、Colyseus、Akka/Orleans。通信核の一本化（本節上）はこの抽象化の前段であり、DO 固有の四プリミティブそのものにはまだ触れていない。

## E. 大きな未決（別の畑）

- 【種】**時間制** — 人と AI の速度差が牙を剥く。持ち時間なしを超える層。→ ルール v0.6 §7・第三歩 §8（接続レベルの放棄のみ現状対応）。
- 【種】**CPU 対戦** — 不完全情報 AI、「人間臭く」。作り手の動機の核（ドラえもんとのび太、完璧でない相棒）。
- 【種】**エコシステムの汎化** — チェス等への応用。「手番だけを引く」操作が他ゲームで何を生むか。
- 【種】**淀川 第四歩：機械学習ログ** — 蓄積の河口。→ 淀川 §6。合法手集合・思考時間など機械学習用の追加文脈。

## 使い方（末尾・参照用）

三者の共通リスト——**Opus**（伏線を言語化し設計を詰める）／**あなた**（方向を選ぶ）／**Sonnet・Haiku**（実装する）。「必要が生じたら足す」で進めるので、いま足さないと決めた種をここへ綴じる。頭や散らばったチャットに置くと、棋譜と同じく保存し損ねるから。

- **これは索引**。項目の詳細は、それが棲む文書（ルール仕様 §7・各実装指示書・版図・淀川・プリンシパル・総括）を**指す**。真実は一箇所、ここは参照。どの文書にもまだ家の無い伏線だけ、本文をここに抱える。
- **状態の印**: 【種】まだ設計を詰めていない ／【待ち】前提が揃うのを待つ ／【進行中】実装指示書が動いている ／【済】着地（済んだものは総括・指示書へ逃がし、ここには残さないか一行で参照）。
- 実感が「これだ」と言った種は、設計を詰めて実装指示書へ昇格させる。自由に追記・剪定してよい。
- **完了記録の逃がし先**: アークが区切れたら、完了の詳細は総括文書（`archive/`）へ移し、バックログには「開いている次の畝」だけを残す。解決済みの再燃防止記録も総括の付録（解決済み台帳）へ。
