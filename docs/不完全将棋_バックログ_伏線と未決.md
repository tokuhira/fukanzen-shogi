## A. 記録係への道（現在のアーク）

「観られる」まで来た観戦の口を、版図の描く**権威なき観察者**へ育てる二段構え。一段目＝存在と身元、その先＝見る目・封蝋・名乗り・時間。

- 【済】**書庫（対局の永続と身元）** — 対局を部屋（使い回す殻）から解き、内容ハッシュの身元で別ストア（KV `ARCHIVES`）へ綴じた（v0.10.2）。終局（内容ハッシュで確定綴じ）・再戦開始・全員離脱（いずれも断片として暫定ID綴じ）の三契機で「綴じてから拭く」を実装・検証済み。`GET /archive/:id` で部屋 DO を介さず取り出し可能。旧クライアント（`text` 非同梱）でも断片として救われる。
- 【済】**記録係の招待と二証人の交差確認** — 相互同意のオプトイン（`record_invite`/`accept`/`decline`）で招かれた対局のみ綴じる。終局時に両プレイヤーが証言（`record_testimony`）を送り、内容ハッシュ一致で `witnesses:2` の確定綴じ、不一致は裁定せず `record_disagreement` として両証言を保存・surface（v0.11.0）。未招待は終局しても揮発（意図的な選択）。`archived{id}` の最小 surface もここで着地。※既知の制約（意図的に許容、2026-07-05）: 招待の提案・承諾は `window.confirm()` に依っており、主要ブラウザは非アクティブ（バックグラウンド）タブから呼ばれた `confirm()` を画面表示せず即座に自動キャンセル（＝辞退）する。実際に別々の端末・別々の人が対戦する場合は各自の画面が常にアクティブなので問題にならないが、同一端末で2タブ・2ウィンドウを使う対局には現状の設計のまま影響する（先手タブが非アクティブだと後手の招待提案が本人の目に触れず自動辞退される）。コード上の不具合ではなく `confirm()` の仕様上の限界。
- 【待ち】**記録係署名** — content-address した id・二証人一致に、記録係の署名（完全性の封蝋）。二証人が第一の防波堤、署名はその上。→ 中身/エンベロープ分離（淀川 §1.4）。
- 【待ち】**principal による帰属（席を埋める）** — 規約同意＋記名の最小キー。招待/承諾が principal と consent 参照を運ぶ受け皿になる。→ `プリンシパル設計`・版図 §7。第一歩の予約席 `sente`/`gote` に嵌める。Anonymous と Agent を受け入れる土壌として。
- 【種】**記録係の中立性・複数記録係** — 公式記録係（DO）が嘘を綴じないと誰が保証するか。二証人が第一の防波堤だが、究極的には複数記録係の突き合わせ・記録係署名・検証可能な記録が要る。版図「正体を問わない世界」の遠い地平（記録係二段目 §13 より）。
- 【待ち】**書庫の一覧・鑑賞・検索** — 過去局ブラウザ。KV の prefix 列挙で素直に。→ 急所指示書 §11。`archived{id}` の最小 surface（リンク表示・コピー）は記録係二段目で着地済み。完全な導線（過去局一覧・鑑賞）はここで引き続き待つ。
- 【待ち】**保持ポリシー・匿名化・再識別性** — 削除＝Anonymous 化では再識別が残る（プリンシパル §3.3 が版図 §2.4 を修正）。→ 名乗りの層と同時。

- 【済】アーカイブ読込の DoS（組手数・制御文字）— v0.8.2。安全網の重複 500 は v0.9.0 で `engine::terminate::MAX_TURNS` へ一本化。
- 【済】**ライブ観戦の秘匿境界** — commit-reveal が構造的に溶かす（隠すのは状態でなく「次の一手」）。観戦は読み取り専用・公開組手のみ・back-channel なしで by construction 安全、**遅延不要**。→ 第三歩 §1。観戦者の request_reset 経路も塞いだ（v0.10.1）。
- 【済】**観戦の口を開けたことで再発したデータ喪失** — 部屋の記録は再戦（`spectate_meta`）で拭かれる作りだったため、観戦を実装した第三歩で棋譜喪失が一段上で再発していた。記録係一段目（書庫・内容ハッシュ）で閉じた（v0.10.2）。
- 【済】**書庫の競合修正が生んだ退行、および上限欠如** — v0.10.2 の「二重書き込みを避ける」修正が、`archived` フラグを KV 書き込み成功より前に立てる作りになっており、書き込み失敗時にサイレントな記録喪失を招きうる状態だった（無害な競合を、この機能が防ぐはずだった致命的な喪失と交換してしまっていた）。あわせて `spectate_turn`（着手数）・`spectate_result.text`（本文サイズ）にサーバ側の上限が無かった点も発見。実装直後のセキュリティ自己レビューで発見・修正（v0.10.3）。
- 【済】**二証人の突き合わせが接続の現存を前提にしていた** — 記録係二段目の実装直後、Playwright での実ブラウザ検証で発覚。証言の収集を `getWebSockets("player")` との突き合わせで判定していたが、実際のクライアントは証言送信直後に自分から切断するため、二人目の証言が届く頃には一人目の ws が既に「player」から外れており、二人分揃っても一致判定に進めなかった（生 WS のスクリプトでは見えず、実ブラウザの資源で初めて再現した）。`Map` の `size` で判定するよう修正。あわせて、綴じの結果（`archived`/`record_disagreement`）は非同期（ハッシュ計算・KV 書き込み）のため、証言送信後すぐに切断する実装のままでは通知がほぼ常に届かない構造的な穴も発見——結果を受け取るか保険のタイムアウトまで切断を待つようクライアントを修正した（v0.11.0）。
