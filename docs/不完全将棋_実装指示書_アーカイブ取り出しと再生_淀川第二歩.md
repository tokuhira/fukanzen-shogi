# 不完全将棋 実装指示書 — アーカイブの取り出しと再生（淀川 第二歩）

> 対象実行者: Claude Code（Sonnet）
> 前提: 配布 v0.8.0 リリース済み（淀川 第一歩＝版タプル付きアーカイブ保存）。既に実装済みの現物:
> - `engine::archive`: `ArchiveMeta { rule:(u32,u32), protocol:u32, app:Option<String>, sente:Option<String>, gote:Option<String>, result:ArchiveResult }`、`ArchiveResult { kind:ResultKind, outcome:Outcome }`、`ResultKind`/`Outcome`（各 `to_str`/`from_str`）、`ARCHIVE_FORMAT_VERSION = 1`。**`archive_to_kifu(&str) -> Option<(Kifu, ArchiveMeta)>` は実装済み**（第一歩で往復テストの担保として。旧 `sfen` 始まり棋譜の後方互換・未知ヘッダ無視の前方互換も内包）。
> - `engine-wasm`: `resolve_ply`・`game_status`・`legal_actions`・`build_archive`（書き）。**読み（parse）はまだ Wasm に露出していない**。
> - `protocol-wasm`: `version_tuple()` → `{"rule":"0.5","protocol":2,"app":"0.8.0"}`。
> - `web/board.js`: 保存導線（`btn-save`→`saveKifu`→`buildArchiveText`/`currentResult`/`archiveFilename`）、再生機構（`loadPlies(plies)`・`sfens`・`cursor`・棋譜ナビ・水墨盤・墨点・`usiToText(usi, sfen, side)` で日本語表記を再計算）。起動時に `versionTuple = JSON.parse(wasmVersionTuple())` をキャッシュ。初期局面は定数 `INITIAL_SFEN`。
> 関連文書: `不完全将棋_棋譜対局データ設計_方針.md`（淀川。本実装はその §6 **第二歩**）、`不完全将棋_実装指示書_版タプル付きアーカイブ_淀川第一歩.md`（第一歩。書式・型・§10 の申し送り）、`不完全将棋_版図_世界観と設計方針.md`（§3 棋譜への刻印）。
> 性格: 保存したアーカイブを**取り出して再生・鑑賞**する、読みの河口。実戦が鑑賞素材になる（淀川 §2.1）。engine の読みは既にあるので、第二歩は **(1) それを Wasm へ露出し、(2) web に読込導線を付け、(3) 既存の再生機構へ流し込み、(4) 刻まれた版タプルと結果を鑑賞として見せ、版が食い違えば正直に注意する**。**対戦互換性・アーカイブ書式は変えない**（書式版は 1 のまま。読み手を足すだけ）。指し終えた対局は全開示なので秘匿境界の問題は無い（淀川 §2.1）。

## 0. 目的と範囲

- **作るもの**:
  1. `engine-wasm` に **`parse_archive(text) -> JSON`**（`build_archive` の対）（§1）。
  2. `web` に **読込導線**（ファイル選択＋貼り付け。保存の「ダウンロード＋コピー」に対称）（§2）。

  4. 刻まれた **版タプル・結果の鑑賞表示** と、**版不一致の注意**（§4）。
- **位置づけ**: 淀川 §6 第二歩。第一歩で開けたアーカイブ（蓄える）の河口から、水を汲み出して再生する。engine 側 `archive_to_kifu` が土台、本書はその上の露出と web 導線。

  - **サーバ側アーカイブの取り出し**（Durable Object から一覧・取得）・**ライブ観戦**（淀川 §6 第三歩。秘匿境界 §4 未決）。
  - **異なるルール版での忠実な再生**（当時のルールで再計算するには複数 engine 版が要る。本書は現行エンジンで再生し、**版が違えば注意を出す**まで. §4）。
  - **principal の表示・検証**（席は `-` 予約のまま。中身は版図／プリンシパルの第一歩＝別実装）。
  - **記録係署名の検証**（エンベロープの封蝋。まだ無い）。
  - **アーカイブ一覧・ライブラリ管理 UI**（複数棋譜の保存・整理）。最小は「一つ読んで再生」。
  - **編集専用モード**の新設（読み込んだ局面から盤クリックで分岐再生できる既存挙動はそのまま継承してよい。§3 の注記）。

## 1. engine-wasm: `parse_archive` の露出（`build_archive` の対）

`engine::archive::archive_to_kifu` を呼び、web が扱いやすい JSON へ整形する。`build_archive` と同じ流儀（手書き or `serde_json`。既に `serde_json` を依存に持つのでそれでよい）。

```rust
/// アーカイブ書式 v1（または旧 sfen 始まり）のテキストを解釈して対局データを返す。
///
/// 成功:
/// {"ok":true,
///  "initial_sfen":"lnsgkgsnl/...",
///  "plies":[{"s":"7g7f","g":"3c3d"}, ...],
///  "meta":{"rule":"0.5","protocol":2,"app":"0.8.0",
///          "sente":null,"gote":null,
///          "result":{"kind":"mate","outcome":"gote_wins"}}}
/// 失敗: {"ok":false,"error":"<理由>"}
#[wasm_bindgen]
pub fn parse_archive(text: &str) -> String;
```

- 実装: `engine::archive::archive_to_kifu(text)` を呼ぶ。`None` なら `{"ok":false,"error":"parse_failed"}`。
- `initial_sfen`: `engine::serialize::position_to_sfen(&kifu.initial_position)`。
- `plies`: 各 `Ply` を `{"s": ply.sente.to_usi(), "g": ply.gote.to_usi()}`。**USI のみ**（日本語表記は中身に持たない＝web が再計算。淀川 §1.4）。
- `meta`: `ArchiveMeta` を写す。`rule` は `"maj.min"` 文字列、`protocol` は数値、`app`/`sente`/`gote` は `Option`（`None` → JSON `null`）、`result` は `{kind: kind.to_str(), outcome: outcome.to_str()}`。
- JSON 文字列生成時のエスケープは既存 `escape_json` に倣う（SFEN・USI に特殊文字は基本出ないが、安全に）。

> **engine 側の変更は不要**（`archive_to_kifu` は既にある）。第二歩の Rust 仕事はこの薄い露出一つ。

## 2. web: 読込導線（保存に対称）

保存が「ダウンロード＋クリップボードへコピー」なら、読込は「ファイル選択＋貼り付け」。両方を用意する。

- **UI**: `index.html` の `#kifu-controls` に **「棋譜を読込」ボタン（`btn-load`）** を追加。押下で:
  - **(a) ファイル選択**: 隠し `<input type="file" accept=".kifu,.txt,text/plain">` を開き、`FileReader` でテキストとして読む。
  - **(b) 貼り付け**: テキストエリアへアーカイブ本文を貼って読み込めるようにする（保存時のコピー可能テキストと対。既存の `#online-modal` に倣った軽量モーダル、または `#kifu-controls` 直下の折りたたみ領域でよい）。
- どちらの経路でも、得たテキストを §3 の `loadArchive(text)` に渡す。
- バージョン表記: `index.html`/`board.js` の `?v=` と `style.css?v=` を新版（§7）へ更新。

## 3. web: 再生機構への流し込み（`board.js`）

### 3.1 パース

```js
function parseArchiveText(text) {
  const r = JSON.parse(wasmParseArchive(text));   // engine-wasm
  if (!r.ok) return null;
  return r;   // { initial_sfen, plies:[{s,g}], meta }
}
```

`board.js` 冒頭の import に `parse_archive as wasmParseArchive` を追加。

### 3.2 既存 `loadPlies` の一般化（二点だけ）

現行 `loadPlies(plies)` は (i) 初期局面を定数 `INITIAL_SFEN` に固定、(ii) 各 ply に日本語表記 `sText/gText` が載っている前提（`DEMO_PLIES` はそう）。アーカイブは (i) 自身の初期局面を持ち、(ii) USI のみ。二点だけ一般化する:

```js
function loadPlies(plies, initialSfen = INITIAL_SFEN) {
  kifu.plies = [];
  sfens  = [initialSfen];
  events = [];
  for (const ply of plies) {
    const preSfen = sfens.at(-1);
    const r = JSON.parse(resolve_ply(preSfen, ply.sUsi, ply.gUsi));
    if (!r.ok) throw new Error(r.error);
    // 日本語表記が無ければ再計算（中身は USI、表記は導出）
    const sText = ply.sText ?? usiToText(ply.sUsi, preSfen, 'sente');
    const gText = ply.gText ?? usiToText(ply.gUsi, preSfen, 'gote');
    kifu.plies.push({ sUsi: ply.sUsi, gUsi: ply.gUsi, sText, gText });
    sfens.push(r.sfen);
    events.push(r.event);
  }
  cursor = 0; phase = 'position';
  resetInput();
  gameOverCache  = { cursor: -1, msg: null };
  resultOverride = null;
}
```

- `DEMO_PLIES`（`sText/gText` 有り）も従来どおり動く（`??` で既存表記を尊重）。
- 初期局面はアーカイブ由来を使う（**アーカイブが本流の真実**。現状の対局は標準初期局面のみだが、席を空けるのと同じ精神で、定数固定にしない）。

### 3.3 `loadArchive`

```js
function loadArchive(text) {
  const parsed = parseArchiveText(text);
  if (!parsed) { alert('棋譜を読み込めませんでした'); return; }

  // ローカル鑑賞として読む（オンライン状態は畳む）
  if (onlineGameOver || onlineMode) _resetOnlineState();
  onlineMode = false;

  const plies = parsed.plies.map(p => ({ sUsi: p.s, gUsi: p.g }));
  try {
    loadPlies(plies, parsed.initial_sfen);
  } catch (e) {
    alert('棋譜の再生に失敗しました: ' + e.message);
    return;
  }

  loadedMeta = parsed.meta;   // §4 の鑑賞表示・版不一致判定に使う
  render();
}
```

- 読み込んだ対局は **ローカル鑑賞**（`onlineMode=false`）。既存の棋譜ナビ（前／次、水墨盤、墨点、日本語表記）でそのまま鑑賞できる。
- **分岐再生**: 既存機構では、再生中に盤をクリックすると `branchAndAppend` で分岐して指し継げる。この挙動は継承してよい（実戦から変化手順を並べる鑑賞の楽しみになる）。分岐後に保存すれば、その分岐手順の新しいアーカイブが得られる（`buildArchiveText` が現 `kifu.plies` を書くので自然に正しい）。

## 4. 版タプル・結果の鑑賞表示と、版不一致の注意

刻まれた版と結果を見せることが、鑑賞の芯であり、版図 §3「記録が版を名乗る」を目に見える形にする。

- **鑑賞表示**: 読み込んだ `loadedMeta` から、`ルール <rule> / プロトコル <protocol>`（`app` があれば併記）と、`result`（§7 の語彙を日本語へ：例 `mate`+`gote_wins` → 「後手の勝ち（詰み）」）を、小さな情報行に出す（`#event-label` 付近か、専用要素を一つ追加）。principal 席は `-` なので今は出さない（または「記名なし」）。
- **版不一致の注意（第二歩の要）**: `loadedMeta.rule` と、現行エンジンの `versionTuple.rule` を比較する。
  - **一致** → そのまま再生（忠実）。
  - **不一致** → 再生は現行エンジンで走るため、当時と結末が変わりうる（淀川 §3）。**はっきり注意を出す**（例:「この棋譜は ルール {当時} で指されました。現在の再生エンジンは ルール {現行} です。再生結果が当時と異なる可能性があります。」）。再生自体は妨げない（鑑賞は許す）。
  - これは第一歩で「版違い再生は将来」と送った宿題の、誠実な最小の始末——**黙って誤再生しない**。当時のルールでの再計算（複数 engine 版の同梱）は将来（§8）。
- **旧 bare 棋譜の扱い**: `archive_to_kifu` は旧書式に対し `rule = 現行 RULE_VERSION`・`protocol = 0` を返す（第一歩実装）。すなわち版を持たない旧棋譜は「現行版」として読まれ、不一致注意は出ない（best effort。旧棋譜の真の版は不明なので、注意のしようがない）。必要なら「版情報なし（旧書式）」の一言を添えてもよいが、必須ではない。

## 5. 後方・前方互換（読み手として）

- **旧 bare 棋譜**（`sfen` 始まり）: `archive_to_kifu` が受理済み。`parse_archive` 経由でそのまま読めること。
- **未知ヘッダキー**: `archive_to_kifu` が無視（前方互換）。将来 principal やエンベロープのキーが増えた棋譜も、第二歩の読み手が壊れず読めること。
- **不正入力**: 壊れたテキスト・空文字・非アーカイブは `{"ok":false}` を返し、web は穏当に「読み込めませんでした」で止める（クラッシュしない）。

## 6. テストと受け入れ基準

### 6.1 engine-wasm（必須）

- `build_archive`（書き）→ その出力を `parse_archive`（読み）→ `initial_sfen`・`plies`・`meta`（rule/protocol/app/result）が**往復一致**する（Rust テストで検証）。
- 旧 bare 棋譜文字列を `parse_archive` → `ok:true`、plies が読め、`meta.result.kind == "unfinished"`。
- 壊れた入力 → `{"ok":false,...}`。

### 6.2 web（受け入れ・手動）

- 第一歩で保存した `.kifu` を **ファイル選択**で読み込み → 盤に初手から並び、棋譜ナビ・日本語表記・水墨・墨点で鑑賞できる。
- **貼り付け**でも同じテキストが読める（保存時のコピーと往復）。
- **版一致**時: 注意は出ず、そのまま鑑賞できる。
- **版不一致**の擬似確認: アーカイブの `rule` 行を手で別値（例 `rule 0.4`）に書き換えて読み込むと、**版不一致の注意が出る**（再生は妨げられない）。
- 刻まれた **版タプルと結果が鑑賞表示**に出る。
- 壊れたテキストを読ませても穏当に失敗し、クラッシュしない。
- `DEMO_PLIES`・オンライン対局・新局など既存機能が回帰しない（`loadPlies` 一般化の影響確認）。

## 7. 版の刻み

- **配布版 v0.8.0 → v0.8.1**（パッチ）。破壊的変更は無く、第一歩が開けた地平（棋譜が残せる）の、地続きのじわっとした延長＝取り出して眺められるようにする一歩。第一歩を minor にしたのは「棋譜が残せるようになった」という出来事の重みへの敬意（記念）であり、SemVer 的な非互換の有無ではない。その目盛りで測ると第二歩はパッチが似合う（マイナーは出来事に、パッチは地続きの前進に）。`workspace.package.version` 更新・タグ連動、`index.html`/`board.js`/`style.css` の `?v=` 更新。
- **ルール版・プロトコル版・アーカイブ書式版は据え置き**（ルール 0.5 / プロトコル 2 / 書式 1）。第二歩は**読み手を足すだけ**で、書式もワイヤも変えない。対戦互換は不変。

## 8. 非ゴールと次段への申し送り

- **当時のルールでの忠実な再生**: 版不一致時に、注意で済ませず**当時のルールで再計算**するには、複数のルール版エンジンを同梱・切替する仕組みが要る（淀川 §3 の「ルール進化の考古学」）。将来の層。
- **第三歩（ライブ観戦）**: Durable Object のブロードキャストと、サーバ側アーカイブの一覧・取得。ここで淀川 §4 の**秘匿境界**（ライブ性 vs チート排除）を実物で決める（未決）。
- **アーカイブ・ライブラリ**: 複数棋譜のローカル保存（localStorage 等）・一覧・整理 UI。最小の「一つ読んで再生」から必要が生じたら足す。
- **principal・エンベロープの表示／検証**: 席が埋まり（版図／プリンシパル 第一歩）、記録係署名が乗ったら、鑑賞表示に帰属・本人性・非改竄の確認を足す。
- **機械学習ログ**: 蓄積の第四歩。

## 9. 不変の原則（本実装が守るもの）

1. **一つの本流、複数の河口**: 保存（第一歩）と対の読み（第二歩）を、同じアーカイブ書式・同じ engine の正本から派生させる。別の仕組みを作らない。
2. **中身は USI、表記は導出**: 読み込みでも日本語表記は再計算する（`usiToText`）。中身に冗長を持ち込まない（淀川 §1.4）。
3. **アーカイブが本流の真実**: 初期局面はアーカイブ由来を使う（定数に固定しない）。
4. **記録が版を名乗るのを、目に見せる**: 刻まれた版タプルを鑑賞表示に出し、版が食い違えば正直に注意する（版図 §3・淀川 §3）。黙って誤再生しない。
5. **書式もワイヤも変えない**: 第二歩は読み手を足すだけ。対戦互換・書式版は不変。
6. **設計は全体、実装は最小から**: 河口は「一つ読んで再生」から。ライブ・ライブラリ・当時版再生は独立に後付けする。
