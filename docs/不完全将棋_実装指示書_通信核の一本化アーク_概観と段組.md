## 0. なぜこのアークか

TUI は純粋な P2P（一方が `listen`＝先手、一方が `connect`＝後手、二人が直接 commit-reveal を交わす。待ち受けた側がそのままサーバ）。web はスター型で、両者が Cloudflare Durable Object（DO）へ WebSocket で繋ぎ、DO が対局チャネルを**中身を解さず相手へ素通し**しつつ、その上に部屋・観戦・書庫・記録係の層を重ねている。

この二つを跨いで測ると、**核は既に共有されているのに殻が二重管理で乖離している**ことが分かった。

- **共有済み**: commit-reveal の純粋状態機械 `protocol::session::TurnSession`（I/O もワイヤ形も持たない referee-less な核）。web は protocol-wasm 経由、TUI はネイティブに、同じ `TurnSession` を使っている。
- **二重管理で乖離**: その上の二層——(B) ワイヤ語彙（JSON の形）と (C) セッション進行の orchestration（handshake 追跡・peer_commit の先着バッファ・ターン駆動）——が、TUI 側（`tui/src/net.rs` の `NetMessage` 手書き＋`tui/src/online.rs` の手続き的 inline 駆動）と web 側（`protocol-wasm/src/lib.rs` の `ProtocolSession`＝JSON 文字列手組み）に別々に実装されている。

web が「サーバを考えながら積んでスパゲッティ化し、一部整理して停止」した地点。TUI は「その前の純粋さと簡素さ」を保っている。サーバ側の基本機能が固まった今、**核へ寄せて最適なクライアント設計に進む好機**。一本化すれば web もこの核を後から使え、web の整理の続きにも接がる。

## 1. 四つの決定（作り手が選んだ）

1. **LAN モードは残す。** `ClientSession` を topology 非依存に作れば、LAN は「もう一枚の殻（TCP）」として安く保てる。TUI の純粋・オフラインという出自を守る。
2. **層 C まで一本化する。** ワイヤ serde 型（層 B）だけでなく、セッション orchestration（層 C）も `protocol` 核へ抜く。protocol-wasm と TUI が同じ `ClientSession` を wrap する。orchestration の二重管理まで断つ。
3. **再接続を再定義する。** 生 `secret` を送る TUI 方式を捨て、web の `auth_hash` 方式（生 secret を晒さない）を canonical にする。本人照合は「初回 hello で記録した peer_auth_hash との一致」で行う。
4. **PROTOCOL を改版する（4→5）。** 一本化した時点で旧 TUI（LAN）ビルドとは非互換。版交渉が綺麗に弾く。web と TUI を同時に 5 へ動かす。

## 2. 目標の四層構造

核へ寄せるのは下二層。上一層は既に核、最上一層だけが platform 別に残る。

```
┌─────────────────────────────────────────────────────────────┐
│ 層D・トランスポート殻（platform 別・唯一の分岐点）              │
│   TUI-LAN: TCP（4byte 長さプレフィックス, tui/src/net.rs）      │
│   TUI-cloud: WebSocket + TLS（新設 · 第四段）                  │
│   web: JS の WebSocket（web/online.js, 既存）                  │
│   ＋ DO のシステム/部屋メッセージ（room_ready 等）はここで処理    │
├─────────────────────────────────────────────────────────────┤
│ 層C・セッション orchestration（protocol へ抜く · ClientSession）│
│   handshake 追跡 · peer_commit 先着バッファ · ターン駆動        │
│   feed(WireMessage) → SessionEvent / *_msg() → WireMessage    │
├─────────────────────────────────────────────────────────────┤
│ 層B・ワイヤ語彙（protocol へ抜く · WireMessage · serde 正本）    │
├─────────────────────────────────────────────────────────────┤
│ 層A・純粋状態機械（既に protocol · TurnSession · 無変更）        │
└─────────────────────────────────────────────────────────────┘
```

`ClientSession` は「side と secret を渡されて hello を出し、届いた `WireMessage` を feed して次に送る `WireMessage` を得る」だけで、**topology を知らない**。誰が side を決めたか（LAN のポータル選択か、DO の room_ready か）は殻が吸収する。だから同じ核が LAN でもクラウドでも動く——「核と交換可能な殻」の実装的な結実。

**重要な境界**: `WireMessage` は**対局チャネル（DO の `routeDecision` で言う "other_player_only"）の語彙**——hello / commit / reveal / ack / reconnect / reconnect_ack / abort。DO のシステム・部屋メッセージ（peer_joined / room_ready / room_full / peer_disconnected / peer_reconnected / you_reconnected）は**含めない**。それらは層 D（WS 殻）の関心事で、`ClientSession` へは feed しない。この境界は DO の routeDecision と正確に一致する。

## 3. canonical ワイヤ語彙（PROTOCOL 5・web の形を正本に採る）

web の語彙のほうが進化している（hello 集約・生 secret を晒さない再接続）ので、それを canonical にする。serde タグ付き enum（`#[serde(tag = "type", rename_all = "snake_case")]`）で `protocol::wire::WireMessage` として一箇所に。

| type | フィールド | 由来 |
|---|---|---|
| `hello` | `rule_major:u32, rule_minor:u32, proto_ver:u32, auth_hash:String(hex), side:String("sente"\|"gote")` | web `hello_msg`。版＋認証＋陣営を一通に集約 |
| `commit` | `commitment:String(hex64)` | 両者一致 |
| `reveal` | `action:String(usi), nonce:String(hex64), board_hash:String(hex64)` | web の欄名 `action`（TUI の `action_usi` は捨てる） |
| `ack` | （なし） | 両者一致 |
| `reconnect` | `auth_hash:String(hex), board_hash:String(hex)` | web `reconnect_msg`。生 secret を晒さない |
| `reconnect_ack` | `board_hash:String(hex)` | 再開点の確認応答。現在は online.js が JS 側で組む——核へ移す |
| `abort` | `reason:String` | 両者一致 |

**捨てる TUI 独自語彙**: `version_hello`（別メッセージの版交渉）と `game_start(side, secret_hash)` は、`hello` に畳まれて消える。TUI の `NetMessage`（net.rs）はこの `WireMessage` に置き換わる。

## 4. 段組（四段・依存 1→(2・3 並行可)→4）

各段は独立に完了・検証できる小ささに割る。下流の指示書は上流が着地して**現物を測り直してから**書く（想像で書かない）。

- **第一段 — `protocol` 核: `WireMessage`＋`ClientSession`＋PROTOCOL 5**
  純粋 Rust。`protocol` クレートに層 B・C を新設。`cargo test` で完結（transport 非依存・wasm 非依存・TUI 非依存）。**要石**。他の三段はこの API を参照する。指示書: `不完全将棋_実装指示書_通信核の一本化_第一段_protocol核へWireMessageとClientSession`。
- **第二段 — protocol-wasm を薄いラッパへ**
  `ProtocolSession` を `protocol::ClientSession` の薄い wasm_bindgen ラッパへ痩せさせる。**online.js から見た契約（`feed`/`hello_msg`/`commit_move`/`reveal_msg`/`ack_msg`/`reconnect_msg`/`peer_auth_hash` の String 入出力とイベント形）と、DO を流れるワイヤ・バイト列を保存**。web の挙動は不変。web が緑のまま（web テスト＋実 DO への WS 接続）なら、核が実地で正しい証明になる。wasm 再ビルド要（この箱では走らないので Claude Code 側）。
- **第三段 — TUI をネイティブ `ClientSession` へ（LAN 殻＝TCP は維持）**
  net.rs の `NetMessage` を `WireMessage` に、online.rs の inline orchestration を `ClientSession` 駆動に置換。LAN（TCP）殻はそのまま。**再接続を新 auth_hash 方式へ再定義**（生 secret を送らない）。検証: 二つの TUI を localhost で相互対戦（LAN 自己対戦）。LAN のみで cloud topology から隔離。
- **第四段 — TUI に WS 殻を足す＝クラウド参加**
  同じ `ClientSession` 契約（send `WireMessage` / events）を満たす WebSocket 殻を新設（TLS・upgrade・WS framing）。DO のシステム/部屋メッセージを層 D で処理（room_ready/peer_joined→side 決定、room_full、peer_disconnected、peer_reconnected、you_reconnected）。ポータルに「クラウド対戦（部屋キー）」を追加。**side は listen/connect の選択でなく DO が告げる**。再接続は DO の you_reconnected/peer_reconnected の枠組みに乗る。検証: 実 DO の部屋に TUI が入り、ブラウザと対局。**アークの結実**。

## 5. 版の物語

- **PROTOCOL 4→5**。`protocol::negotiate::PROTOCOL_VERSION` は単一の正本。第一段でここを 5 へ上げると、web（protocol-wasm 経由）も TUI（ネイティブ）も、再ビルド時に自動で 5 を名乗る。
- **版跨ぎの安全**: 旧 TUI（`version_hello` 別メッセージ・PROTOCOL 4）と新実装（`hello` 集約・5）は、ワイヤ形も版番号も食い違うので版交渉が確実に弾く。web は配信で自動更新されるため、実害は「対局中の版跨ぎ」だけで、それは既存の negotiation が守る。同一クライアント族の内部（旧 web 同士・旧 TUI 同士）は各自の実装が内部整合しているので、第一段で版だけ 5 に上がっても壊れない（アーク途中の単段ビルドは release しない）。
- **配布版**: 第一〜三段は核・web・LAN の挙動保存／内部再構成の色が濃い。**利用者に見える新能力（TUI のクラウド参加）は第四段**で立つので、配布版のマイナー bump は第四段に合わせる（出来事＝マイナー）。それ以前の段は web `?v=` 前進や PROTOCOL 定数変更にとどめる判断が自然（各段指示書で確定させる）。

## 6. この箱の制約（実装者への申し送り）

Opus のこの環境では **cargo / wasm-pack が走らない**（rustup 配布元が不許可）。よって各段の Rust ビルド・`cargo test`・wasm 再ビルド・実 WS 検証は **Claude Code（Sonnet 5）側**で行う。Opus は現物のシグネチャ・JSON バイト列に接地して指示書を書き、リリース後にリポジトリを取り直して現物レビューする。**指示書はワイヤのバイト列（フィールド名・型・serde タグ）を特に厳密に記す**——compile で拾えない差異は指示書で潰す。

## 不変の原則（このアークで守る）

- **核と交換可能な殻**: 語彙と orchestration は核（`protocol`）へ、トランスポートは殻へ。
- **純粋の芯・注入で受ける**: 乱数（Nonce）は呼び出し側が生成して渡す（`TurnSession` の流儀を `ClientSession` も継ぐ＝決定的テスト可能）。盤面ハッシュ・USI 解釈も殻／呼び出し側で行い、核は typed な値で受ける。
- **審判なし**: DO は commit-reveal を裁定せず素通しする。各クライアントが相手の reveal を自分で検証する（`TurnSession` の検証は不変）。中継は referee-less 設計に透明。
- **挙動保存**: 第二段は web の、第三段は LAN の挙動を保存する（失敗時の握り潰し／伝播、送信順、先着バッファ）。差異は指示書と受け入れ条件の中心に据える。
- **細かく刻む・現物接地**: 各段を独立に検証できる小ささに。下流は上流着地後に現物を測り直してから書く。
- **過ぎたるは及ばざる**: 統一ディスパッチャや汎用抽象は必要が呼んでから。DO を離れる移植抽象は第四段でも作らない（バックログ §D のまま待つ）。

---

*核は既にあった。乖離していたのは殻の二枚だけ。それを核へ寄せれば、LAN とクラウドが同じ心臓で鼓動し、TUI のプレイヤーとブラウザのプレイヤーが同じ部屋に座れる。急ぐものは何もない。第一段の要石から、地面に接地して始める。*
