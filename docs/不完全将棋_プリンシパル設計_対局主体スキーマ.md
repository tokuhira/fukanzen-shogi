# 不完全将棋 プリンシパル設計 — 対局主体（principal）中心の最小キーと三層分離

> 位置づけ: 版図（世界観・設計方針）の下に置く、**実装のための設計図**。版図が「なぜ・どういう世界か」を語るのに対し、本書は「どう作るか」のスキーマを定める。棋譜の根っこ＝「誰が指したか（principal）」の設計。
> 階層の割り切り: **世界観（版図）は固く、法規は揺れ動く**。AI の進歩で法規が動く時代、法規対応は実務寄りの殻として吸収し、版図という核は無傷に保つ。本書は、固い世界観を principal 中心のスキーマに落としつつ、揺れる法規を「継ぎ目を空ける／中核を契約根拠に逃がす」実務で吸収する。
> 前提: 別途の法務・技術サーベイ（APPI/GDPR/EULA、Cloudflare Web Bot Auth 等）の結論を反映。ただし本書は実装スキーマに徹し、法解釈の詳細には踏み込まない（揺れ動く前提で柔軟に）。

---

## 0. 設計の背骨：principal 中心 ＋ 3層の分離

すべてのデータと同意を **principal（主体）** に紐づける。版図の四象限（Named/Anonymous × Human/AI）を、principal という一つの抽象に収める。

### 0.1 principal の構造
- `principal_id`: 主体の一意識別子。
- `principal_type`: `signed_in` / `anonymous` /〔将来〕`agent`。（版図の Named＝signed_in、Anonymous＝anonymous、Agent＝将来の agent に対応。）
- `subject`: その主体の背後にいる自然人。エージェントなら、背後の人や開発者（責任主体）。

> **最初から principal_type と subject を入れておく**ことが肝。将来エージェントを足すのは「型を1つ足す＋委任の辺を1本引く」だけで済み、作り直しにならない（版図の「Agent を受け入れる土壌として最初から作る」の実装的裏付け）。

### 0.2 混同しがちな3つを別レイヤーに分離

実務で一緒くたにされやすい3つを、別レイヤーとして扱う。**この分離が設計の要。**

1. **識別/認証（誰か）** — サインイン or 匿名 or〔将来〕エージェントのトークン。
2. **処理の法的根拠（なぜ扱ってよいか）＝プライバシーの背骨** — 後述 §1。
3. **規約受諾（EULA、データ帰属を含む）** — クリックラップ、版付き、principal 紐づけ。後述 §2。

> **最大の罠**: EULA の「同意（受諾）」と、プライバシーの「consent（処理根拠としての同意）」の混同。これを分けることが、版図の「同意疲れ・取り直し地獄を抑える」を法的に正しく実現する鍵。

---

## 1. 処理の法的根拠：中核は「契約」、consent は最小限に

棋譜・対戦記録のような**ゲーム中核データ**の扱いを、何を根拠に正当化するか。

- **中核データ＝契約の履行を根拠にする**（GDPR 6条1項b に相当する考え方）。「対局するというサービスを提供する契約の一部」として、棋譜の保存・処理を正当化する。**「あなたが同意したから」ではなく「対局を提供する契約だから」**。
  - 効果: 対局のたびに「棋譜を保存してよいか」と consent を取る必要が消える。版図の「記録は契約に含まれ、気楽に指せる」が実現する。
- **consent（処理根拠としての同意）は、法が明示的に求める所だけに限定する**: marketing、任意分析、外部共有（例: 機械学習への転用が中核を超える場合）、第三者提供・外国移転・要配慮データなど。
- **APPI 観点**: 利用目的の特定・通知公表が基本。本人同意が要るのは第三者提供・外国移転・要配慮あたり。中核処理は目的明示＋契約で回し、consent はピンポイントに。
- **なぜ consent を中核から外すか（二重の利益）**:
  1. 同意疲れ・取り直し地獄を防ぐ（版図の思想）。
  2. **未成年が有効な consent を出せない場面**でも、契約根拠なら（親権者関与の下で）サービスを提供できる。「全部 consent で通す」設計は将来詰む。
  3. 法改正耐性: 同意の置きどころは今後も動く前提。中核を契約に逃がしておけば、consent 規制の変動の影響を受けにくい。

---

## 2. 規約受諾（EULA）：版付きクリックラップ、principal 紐づけ

- EULA（データ帰属を含む）の受諾を、**クリックラップ**（明示的な受諾操作）で取る。
- **版付き**: terms_version を持ち、どの版に同意したかを記録。
- **principal 紐づけ**: 受諾は principal に結びつく。人間も将来のエージェントも、同じ仕組みで受諾する（版図の「同じ同意儀式の下に人間と AI が並ぶ」）。
- これは §1 の consent とは**別物**。EULA 受諾は「契約の成立」、consent は「特定処理の法的根拠」。

## 3. 匿名アカウントの扱い（設計の山場）

版図は「Anonymous はカオス、特定不能」と描いたが、**法的には「匿名」でも再識別できれば仮名＝個人情報側**（GDPR で in scope）。ここを実務で正しく扱う。

### 3.1 匿名でも要る3点
デバイスやセッションで再識別できる以上、匿名 principal にも:
1. **プライバシー通知**（何を、なぜ、どう扱うか）。
2. **法的根拠**（中核は契約、§1）。
3. **データ主体の権利（開示・削除など）への導線**。

### 3.2 リカバリキー（匿名のまま権利行使を可能にする）
匿名ユーザーは手がかりがないと本人特定できず、削除請求等に応じられない＝コンプラの穴になる。対策:

- **アカウント発行時に一度だけ、リカバリキーを本人に見せる**。保存側は**不可逆ハッシュ**で持つ（本体は保持しない）。
- 本人がそのキーを提示することで、匿名のまま権利（開示・削除）を行使できる導線を、**最初から**作る。
- **版図との一貫性**: これは Phase3 の本人認証（共有秘密のハッシュ交換、commit-reveal 同型）と**同じ構造**。運営は鍵のハッシュだけを持ち、本人が本体を提示して照合する。「秘密を見せずに本人性を証明する」思想が、対局の本人認証と、匿名アカウントの権利行使に、二度現れる。

### 3.3 版図の「削除＝Anonymous 化」への修正（重要）
版図 §2.4 は「削除を Named→Anonymous 化で果たし、棋譜は本流に残す」と描いたが、**これは半分しか正しくない**:

- **真の匿名化（不可逆・再識別不可能）だけが GDPR の対象外**。ゲーム記録に紐づき、指し手の癖・相手・時刻等から再識別できる限り、記名を剥がしても**まだ仮名**であり、削除を完全には満たさない可能性がある。
- ゆえに「Anonymous 化」は、**それだけでは削除の代替として不十分**。記名を剥がすこと（個人との紐づけを切る）と、対局記録を真に再識別不可能にすることは、別の作業。
- **実装方針**: 削除請求には、(a) principal と記録の紐づけを切る（記名剥がし）＋(b) 残す対局記録から再識別性をどこまで落とせるかを、データ最小化として設計する。完全な匿名化が難しい部分は、削除（その記録自体を落とす）も選択肢に含める。版図の「対局は公共に残る」理想と、GDPR の削除義務のバランスは、ケースに応じて実務で調整する（揺れ動く法規の側で吸収）。

## 4. 同意・帰属の記録（consent receipt 台帳）

- **同意台帳は追記専用（append-only）**。各レコード（consent receipt）が持つ: `principal_id` / `terms_version` / 規約ハッシュ / `legal_basis` / `timestamp` / `locale`。
- **人間も将来のエージェントも同じレコード形**。版図の「人間と AI が同じ同意儀式」が、共通の receipt 形として実現。
- **対戦記録（棋譜）へのスタンプ**: `owner principal` / `terms_version` / 利用・帰属フィールドを刻む。淀川の版タプル（ルール版・プロトコル版）に加え、principal と帰属がスタンプされる。
- **2者以上のデータの切り分け**: 棋譜は対局者2者（以上）の共同データ。「**各自は自分の側についてのみ同意し、結合レコード（対局全体）はプラットフォームが管理**」と規約で切り分ける。版図の「審判なし・両者がそれぞれ記録を持つ」と整合（各自は自分の手に責任、対局全体は場が管理）。
- **中身とエンベロープの線引き（データ設計と整合）**: 棋譜の**中身（本流＝対局そのもの）には principal が入る**（誰が指したか＝対局の当事者）。一方、**帰属スタンプ・consent receipt 参照・法的根拠は、中身ではなくエンベロープ（取り扱い層）に乗る**（『棋譜・対局データ設計 方針』§1.4）。すなわち「誰が指したか（principal）」は中身、「その principal がどう同意し、この棋譜がどう帰属し扱われるか」はエンベロープ。記録係の署名もエンベロープ側で、担保するのは中身の完全性のみ（帰属・consent には及ばない）。この分離により、principal の同意は principal 自身が、帰属は場が、中身の非改竄は記録係が——各々が自分の領分だけを保証する（審判なし・各自が自分の側に責任）。

## 5. エージェントの「口」を予約する（作り込まない）

将来 agent を足すときの追加を、**継ぎ目だけ空けて予約**する。今は作り込まない。

予約する継ぎ目（コストは安い）:
- `principal_type` に `agent` を足せる構造。
- **委任テーブル**（agent → 背後の人/開発者）。エージェントの権利・義務を責任主体に帰属させる辺。
- セッション/トークンに `scope`（例: read / write match data）。
- **EULA の「エージェント附則」スロット**（他者の記録での学習制限・帰属義務・レート制限など——**中身は今書かず、スロットだけ空ける**）。

> 将来 (agent) を足す実装は、実質「principal 型1つ＋委任の辺＋scope＋附則スロット」。規約を版・ハッシュ・取得可能な形に保っておけば、あとは機械可読化と Cloudflare Web Bot Auth（エッジ）＋エージェントトークンを足すだけで、地続きに繋がる。版図の「記名 AI＝クリックラップ＋Web Bot Auth」は、この継ぎ目の上に後から載る。

## 6. 年齢ゲート（版図に欠けていた軸）

法務サーベイで浮上した、版図に無かった必須の軸。

- **年齢ゲートを今のうちに方針として決めておく**（後が楽）。GDPR は16歳未満は親権者同意が原則（加盟国により13〜16歳）。APPI 改正でも16歳未満の保護強化が方向。
- 未成年は有効な consent を出せない場面がある → **中核を契約根拠に置く設計（§1）が、ここでも効く**（consent に依存しないので、未成年でも親権者関与の下でサービス提供できる）。
- 息子との対局という出発点を思えば、未成年が指すことは現実。年齢の扱いを、最小でも方針として持っておく。

## 7. いま「やる / まだやらない」

法務サーベイの切り分けを、実装の優先度として採用する。

### やる（最小キーに含める）
- **principal 中心のスキーマ**（principal_id / principal_type / subject）。
- **3層分離**（識別・法的根拠・規約受諾）。
- **中核データは契約根拠**、consent は法が求める所だけ。
- **匿名のリカバリキー**（不可逆ハッシュ）＋ 開示/削除の導線。
- **版付き EULA とプライバシー通知**。
- **consent receipt 台帳**（追記専用）と、棋譜への**帰属スタンプ**。
- **年齢ゲート方針**。
- **エージェントの継ぎ目予約**（型・委任テーブル・scope・附則スロットの“空き”だけ。中身は書かない）。

### まだやらない
- Web Bot Auth、エージェントトークンの本実装。
- KYA / mTLS / DID / mandate 等の高度な委任・認証機構。
- 機械可読規約の本実装。
- エージェント附則の中身（学習制限・帰属義務・レート制限の具体）。

> **継ぎ目さえ空けておけば、全部あとから加算で足せる。** これが版図の「実装は最小から、Agent を受け入れる土壌として」の、具体的な実装計画。

## 9. 不変の原則（実装層）

1. **principal を中心に据える**: すべてのデータ・同意を principal に紐づける。型を足すだけでエージェントが載る。
2. **3層を混同しない**: 識別／法的根拠／規約受諾は別レイヤー。EULA の受諾と consent を分ける。
3. **中核は契約、consent は最小**: 同意疲れ・未成年・法改正のすべてに効く。
4. **匿名にもリカバリキーと権利導線**: 匿名でも仮名＝個人情報側として扱う。本人性は鍵で、属性は問わない。
5. **継ぎ目を空け、作り込まない**: エージェントの口は予約のみ。加算で足す。
6. **世界観は固く、法規は殻で吸収**: 版図は無傷に保ち、揺れる法規対応は実務層で吸収する。
