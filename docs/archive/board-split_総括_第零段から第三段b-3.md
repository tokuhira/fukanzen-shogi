# 不完全将棋 board.js 分割アーク 総括（第〇段〜第三段b-3）

*この文書は、web/board.js を「神ファイル」から「薄い殻＋純粋モジュール群」へ解いていく一連のリファクタリング／テスト整備（第〇段〜第三段b-3、配布 v0.11.2 で据え置き・web `?v=` 0.11.2→0.11.11）の総括である。バックログの「現在地」段落と D 節が抱えていた完了記録を、ここへ綴じて受け止める。時系列の完了記録の詳細は各実装指示書（`archive/implementation/board-split/`）にあり、ここは道のり・確立した設計・次アークを course-grained に俯瞰する地図。*

## 0. このアークが始まった地点と、達成したこと

**始点**: web/board.js は 1531 行の神ファイルだった——盤描画・棋譜再生・ホットシート・オンライン対戦・観戦・書庫・記録係の招待が、モジュールスコープの大域可変状態を共有しながら同居していた。三歩（淀川）＋記録係二段で肥大した、リポジトリ最大のファイル。

**終点（現在）**: board.js は **1220 行**（-311 行）。純粋な計算・遷移・描画は **10 本の ES モジュール（計 396 行）** へ抜け、**web テストは 57 件**（第〇段前は 0 件）に育った。状態は単一の `state` 器に集約され、更新は `update(patch)` の一本の経路を通り、意味のある状態遷移は純粋 `reduce` として抽出・テストされている。ビルドレス（バンドラ／TS なし・ES モジュール直読）は最後まで維持。**製品挙動は全段で完全に不変**（配布版タプルは v0.11.2 のまま据え置き、web の `?v=` キャッシュバストのみ 0.11.3→0.11.11 へ独立に前進）。Rust には一度も触れず、Wasm も再ビルドしていない。

**確立した設計パターン**（次のアーク・他の盤・移植の下地）:
- **純粋な芯を抜き、殻に薄いラッパを残す**: DOM・Wasm・可変状態に触れない計算だけを ES モジュールへ。board.js には既存の呼び出し形を保つラッパを残す。
- **Wasm は引数注入**: 純粋モジュールは Wasm を import せず、Wasm 関数を引数で受ける。board.js のラッパが実 Wasm を綴じ込む。これによりモジュールは Wasm 非依存のまま node（vitest）でテストできる。
- **状態は器へ、更新は一本の経路へ、遷移は純粋 reduce へ**: `state`（読み取りの集約）→ `update(patch)`（書き込みの経路）→ `reduce(view, action) → patch`（意味のある遷移の純粋化）。
- **北極星は tui**: `tui/src/app.rs` の `struct App`（状態を一つの器に集約し、メソッドで遷移、描画は分離、I/O は online.rs へ）を相似形の参照点にした。ただし tui は中間形態であり、web はより純粋な形（`&mut self` メソッドではなく patch を返す純粋 reduce）へ進んだ——web の純粋 reduce は、将来 tui をも純粋方向へ引き上げる先例になる。深層構造（遷移が一箇所に集約され描画と分離）は同じで、移植時 reduce は `&mut self` へ素直に翻訳できる。

## 2. 段ごとの道のり（course-grained）

**第〇段（盤面解釈を核へ返す・v0.11.2）**: 自前 `parseSfen`（JS 側の SFEN 解釈二重定義）を削除し、engine-wasm に新設した `position_view` getter へ置換。純粋アダプタ `positionViewToState` を `position-view.js` へ分離し、初の web テスト（vitest）を据えた。CI に web ジョブ追加。「割る前に、まず核へ返す」。

**第一段a（純粋の収穫・v0.11.3）**: DOM・Wasm・可変状態に触れない純粋関数・定数を 4 モジュール（`usi`/`result-view`/`geometry`/`board-view`）へ切り出し。唯一の意味的変更は `terminalMessageJa` の `maxTurns` 引数化。座標系を `geometry.js` に一本化し、描画とヒットテストの共有正本にした（第二段で入力層を割る継ぎ目を先に彫った）。`renderSvg` の golden snapshot を据えた。1496→1307 行。

**第一段b（Wasm足場と棋譜の糊・v0.11.4）**: node で Wasm を読むテスト足場を初めて据え、engine-wasm・notation-wasm を跨ぐ糊 `usiToText` を `notation-view.js`（Wasm 引数注入）へ分離。「Wasm を引数注入する純粋モジュール＋薄いラッパ」の雛形が確立。

**第二段a（棋譜コアの遷移を純粋化・v0.11.5）**: 状態変数（`cursor` だけで参照 53 箇所）は据え置き、遷移の純粋計算だけを `game-record.js`（不変・`record` を受けて新 `record` を返す）へ抜いた。`setRecord`/`currentRecord` が状態と純粋層の橋渡し。挙動保存に注意（`watchAppendTurn` の失敗握り潰し vs `branchAndAppend`/`loadPlies` の例外伝播）。※実地で `resolve_ply` が USI の構文のみ検証し合法性は検証しないことを発見し、テストの不正手例を構文的に不正な USI へ差し替えた（ドキュメントの例を鵜呑みにせず実地確認した一例）。

**第二段b（着手組み立ての計算を純粋化・案A・v0.11.6）**: 入力島の状態は据え置き、埋もれた純粋計算（合法手フィルタ・targetMap・成り不成判定 `resolveTarget`）だけを `move-input.js` へ。選択の状態機械（`handleSvgClick`）と `confirmMove` のオンライン分岐はあえて据え置き（view 分離と一緒に扱う方が自然）。8八角が3三へ入る成・不成両立局面で `promptPromotion` を実 Wasm で検証。

**第二段c（overlay計算を純粋化・v0.11.7）**: `computeInputOverlay`/`computeRevealOverlay`（モジュールスコープを暗黙に読んでいた）を、状態スナップショットを引数で受ける純粋関数 `inputOverlay`/`revealOverlay` として `board-view.js` へ移設。render() 本体は overlay を得る 2 行だけ差し替え。view 純粋化の第一歩（本丸＝状態集約の安全網）。

**第三段a（状態を単一の state 器へ集約・v0.11.8）**: 散在する 30 の状態変数を単一の `const state` へ機械的に集約（読み取りの集約、意味不変、`kifu` は据え置き）。**このアークで唯一 board.js のほぼ全行に触れた段**。指示書の 3 危険（gameOverCache のキー・`.cursor` プロパティ・`svg.style.cursor`）に加え、実装中に 3 件を自ら発見・修正（ES6 短縮記法・`onStatus` コールバック引数名が偶然 `state` と衝突・三項演算子の `:` をキーと誤認する自作スクリプトのバグ）。board.js は vitest 対象外のため本番 WS へ実接続して全機能を実地検証。

**第三段b-1（kifu吸収と状態更新経路・v0.11.9）**: `kifu` を `state.plies` へ吸収し器を完全に一つに。状態更新の唯一の経路 `update(patch)`（浅いマージ＋再描画、tui の `App` 更新→`terminal.draw` に相似）を導入。単一呼び出し元の leaf 関数を中心に `update` へ畳み、複数箇所から異なるタイミングで呼ばれる共有ヘルパはあえて直接代入のまま据え置いた——`btn-online` ハンドラが `resetToNew()` 後にあえて render() しない現物の分岐を発見し、機械的な update 化が意図せぬ再描画を生む実害を確認（b-1 が reducer 化を b-2 へ送った理由の裏付け）。render() 呼び出し 43→22。

**第三段b-2（局面ナビゲーションの遷移を純粋 reduce へ・v0.11.10）**: 書き込み集約の第一号。`goPrev`/`goNext` の純粋な局面遷移を `navReduce(view,action)→patch|null`（`nav.js`）へ抽出。`navView()` で状態の必要部分だけ渡す。副作用分岐（promotion クリア・resetInput・pending 解決）は reduce の外。全 8 分岐・往復の可逆性・各組手が reveal→position の二拍で刻まれる（同時着手の歩み）ことをテストに刻んだ。

**第三段b-3（オンラインリセットとホットシート確定を純粋 reduce へ・v0.11.11）**: 書き込み集約の完了、アークの区切り。種類1 の残り二つを `reducers.js` へ（`resetOnlineReduce`・`hotseatConfirmReduce`）。`_resetOnlineState` は元々 render() を呼ばない挙動を保つため `Object.assign` のまま。`confirmMove` のオンライン分岐（WS I/O）は無変更、ホットシート分岐のみ reduce 化。これで種類1 が `navReduce`＋`resetOnlineReduce`＋`hotseatConfirmReduce` と出揃い、状態更新が「純粋 reduce で patch → update で適用」に統一。

## 3. 次のアーク（重い・設計相談から。地面を測り直してから入る）

board.js（1220 行）にはまだ二つの本丸が残っている。いずれも設計の自由度が大きく、これまでの「純粋な芯を抜く」より重い。急がば回れ・過ぎたるは及ばざる。

- **種類2（I/O 絡みの分解）**: `handleTurnComplete`（online＋メタ＋棋譜）・`enterWatchMode`（4 島・非同期コールバック）・`endOnlineGame`（setTimeout・記録係綴じ待ち）を、tui の `online.rs`（「ゲームロジックは App を再利用する」）に倣い「純粋な状態遷移」と「I/O」に割る。**非同期コールバックの状態更新をどう純粋遷移として括り出すか**が設計の核。各コールバックの発火経路を測り直してから。
- **view 純粋化**: render() 本体（DOM 書き込み 20 箇所超の巨大関数）の phaseText／ボタン分岐を「`state` スナップショット→表示値」の純粋関数へ。`navView()` の「state から必要部分を切り出す」パターンが芽。状態集約が済んだ今、集約前後で「同じ `state` → 同じ描画」をテストで守れる地盤はある。
- **golden snapshot への局面追加**（第一段a からの申し送り）は view 段で。成り駒複数種・二桁持ち駒など字形とレイアウトの端を突く局面を足すと錠が広がる。
- `nav.js` と `reducers.js` の統合は、種類2 の reduce が増えて「遷移モジュールが散る」と感じたら検討（今は不要）。

## 4. このアークで効いた流儀（次の Opus・実装者へ）

- **記憶ではなく現物**: 毎段、リポジトリを取り直し、現在のコード・行番号・関数シグネチャに接地してから設計した。想像で書くと必ずずれる。
- **地面を測ってから指示書**: 危険箇所の洗い出し、変換スクリプトの実地試作（第三段a）、純粋関数の node 検証（各段）を、指示書を書く前に行った。「タフな変更ほど念入りな事前調査が報われる」。
- **細かく刻む**: Sonnet／Haiku へ渡すため、一段を独立に完了・検証できる小ささに割った（第一段 a/b、第三段 a/b-1/b-2/b-3）。
- **挙動保存の細部を明記**: 失敗時の握り潰し／伝播、描画の有無、分岐順——取りこぼすと静かに壊れる細部を、指示書と受け入れ条件の中心に据えた。
- **実装者の自律を信頼**: Sonnet は指示書の危険リストを超えて自ら 3 件の欠陥を発見し（第三段a）、ラッパ方式が呼び出し箇所の数え漏れを吸収し（第一段b）、実 Wasm で挙動を確かめた（各段）。指示書は地図であって地面ではない、という原則が実装側でも生きた。
- **版の目盛り**: 挙動不変のリファクタは配布版据え置き、web `?v=` のみ前進（キャッシュ更新のため）。この分離を全段で貫いた。

*board.js を、割る前に核へ返し、純粋な芯を抜き、器へ集め、経路を一本にし、遷移を純粋 reduce へ——外堀から本丸の主要部まで、11 段で綺麗に畳んだ。神ファイルは薄くなり、純粋モジュール群がテストで固まり、tui と相似の構造が web にも現れた。残るは I/O の分解と view の純粋化。急ぐものは何もない。次の畝は、実感が「これだ」と言うのを待てばいい。*

## 付録：解決済み台帳（再燃防止の記録・旧バックログ F 節より移設）

*一度閉じた問題を、同じ轍を踏まないために記録する。これらは完全に解決済みで、バックログ（開いている畝の索引）からはここへ移した。*

- **データ喪失（棋譜が残らない）** — 第一歩で保存、第二歩で再生。※サーバ層での再発は記録係の書庫で閉じた。
- **版タプルの記録への刻印・版交渉・版不一致注意** — 第一歩〜、ルール 0.6 で初めて実地に機能（v0.9.0）。
- **アーカイブ読込の DoS（組手数・制御文字）** — v0.8.2。安全網の重複 500 は v0.9.0 で `engine::terminate::MAX_TURNS` へ一本化。
- **ライブ観戦の秘匿境界** — commit-reveal が構造的に溶かす（隠すのは状態でなく「次の一手」）。観戦は読み取り専用・公開組手のみ・back-channel なしで by construction 安全、遅延不要（第三歩 §1）。観戦者の request_reset 経路も塞いだ（v0.10.1）。
- **観戦の口を開けたことで再発したデータ喪失** — 部屋の記録が再戦（`spectate_meta`）で拭かれる作りだったため、第三歩で棋譜喪失が一段上で再発。記録係一段目（書庫・内容ハッシュ）で閉じた（v0.10.2）。
- **書庫の競合修正が生んだ退行、および上限欠如** — v0.10.2 の「二重書き込みを避ける」修正が `archived` フラグを KV 書き込み成功より前に立てており、書き込み失敗時にサイレントな記録喪失を招きうる状態だった（無害な競合を、防ぐはずの致命的喪失と交換していた）。`spectate_turn`・`spectate_result.text` のサーバ側上限欠如も併せて発見。セキュリティ自己レビューで発見・修正（v0.10.3）。
- **二証人の突き合わせが接続の現存を前提にしていた** — 記録係二段目の実装直後、Playwright の実ブラウザ検証で発覚。証言収集を `getWebSockets("player")` との突き合わせで判定していたが、実クライアントは証言送信直後に切断するため二人分揃っても一致判定に進めなかった。`Map` の `size` で判定するよう修正。綴じの結果通知が非同期で届かない構造的穴も併せて発見し、結果受領か保険タイムアウトまで切断を待つよう修正（v0.11.0）。
