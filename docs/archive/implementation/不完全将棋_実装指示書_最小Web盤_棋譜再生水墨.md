# 不完全将棋 実装指示書 — 最小 Web 盤（棋譜再生・水墨様式・静的配信の第一枚）

> 対象実行者: Claude Code
> 前提: Cloudflare へドメイン移管済み。静的配信の第一枚。**この段階ではルールエンジンを繋がない**（Wasm 化・合法手判定・衝突解決の実装は次の殻）。ルール仕様は v0.5、配布バイナリは v0.6 系だが、本成果物はそれらのコードに依存しない独立した静的フロントエンドである。
> 関連文書: `不完全将棋_バージョン互換性管理_方針.md`、各ルール仕様。参考にした既存実装の作法: Kifu for JS（SVG 出力・サイトの色を継承する控えめな配色）。
> 性格: 「描かないことで描く」局面図の様式を、Web の盤面として最小限に形にする。棋譜（あらかじめ用意した一局）を再生し、**伏せ → 同時開示 → 抜き → 次局面** の所作だけを水墨の静けさで見せる。操作・入力・ルール判定は載せない。

---

## 0. 目的と範囲

- **作るもの**: 単一ページの静的 Web。内蔵した棋譜（不完全将棋の一局の進行データ）を再生し、各ターンの「両者の伏せた着手の同時開示」と「解決後の局面」を、水墨様式の SVG 盤面で見せる鑑賞用デモ。
- **設計思想（抜き）**: 衝突の解決過程（駒がぶつかる瞬間、相討ち・戦国無双等の処理アニメーション）は**描かない**。前の局面・伏せられた両者の手・解決後の局面、という三点だけを置き、その間（斬撃の一瞬）は見る者の想像に委ねる。新聞の局面図と時代劇の殺陣に通じる省略の美。ただし**不親切は避ける**——両者が何を指したか（開示）と、結果局面が読めることは、必ず残す。
- **作らないもの（非ゴール）**:
  - ルールエンジン・Wasm・合法手判定・衝突解決ロジック（次の殻）。
  - 対局操作・着手入力・秘匿対戦・通信（後の殻）。
  - 墨の滲み・かすれ等の質感フィルタの作り込み（「墨を思わせる配色と細さ」に留める。過剰な作り込みは侘びを損なう）。
  - 色の追加・装飾（墨一色＋生成りの背景から始める）。
  - ビルドツール・フレームワーク・依存パッケージ（素の HTML/CSS/JS で完結させる）。

---

## 1. 技術構成（最小・静的）

- **単一の静的成果物**: `index.html`（必要なら `style.css`・`board.js`・棋譜データファイルに分けてよいが、ビルド不要で、ファイルを置くだけで動くこと）。フレームワーク・バンドラ・npm 依存を使わない。
- **盤面は SVG**: 盤・駒・座標を SVG で描く（ベクターなので、線の細さと余白を厳密に制御でき、任意の解像度で滲まない＝局面図の様式に最適。Kifu for JS も SVG を採用）。
- **Cloudflare 配信前提**: 生成物は静的ファイルのみ。サーバ・状態・API を持たない。そのまま CDN に置ける。
- **レスポンシブ**: SVG に `viewBox` を設定し、`max-width`（例: 400px 程度）で囲む。モバイルの縦長では横幅いっぱい、デスクトップでは過大にならない（Kifu for JS の作法に倣う）。

---

## 2. 水墨様式（配色と線）

「墨を思わせる配色と細さ」だけを守り、質感の作り込みはしない。

- **配色**: 墨一色（濃いグレー〜黒、例 `#1a1a1a`）の線と文字。背景は生成り／淡い紙色（例 `#f5f3ec` のような、白すぎない淡色）。彩度のある色は使わない。
- **線**: 盤の格子は細い線（細め、例 0.5〜1px 相当の SVG ストローク）。盤の外枠はやや太く。過剰な装飾罫を入れない。
- **駒**: 塗りつぶした駒台形（将棋駒の五角形）は描いても描かなくてもよいが、**最小に振るなら、駒は漢字一文字を置くだけでも様式として成立する**（局面図は駒形を省いて文字だけのことも多い）。駒形を描く場合も、墨の細線の輪郭に留める。先手・後手は、駒の向き（上下反転）で区別する（色では区別しない＝水墨に忠実）。
- **フォント**: 明朝体系（serif）。Kifu for JS も serif を既定にしている。漢字が端正に出るものを指定し、環境依存はフォールバックで吸収。
- **質感は作らない**: 滲み・かすれ・筆圧の SVG フィルタは入れない。「墨を思わせる」のは配色と細さで足りる。質感の追求は将来の殻に開けておく。

---

## 3. 駒の表現（漢字・成駒は一文字）

- **通常の駒**: 王/玉・飛・角・金・銀・桂・香・歩（先手玉は「玉」、後手玉は「王」等の慣例は任意。区別不要なら両方「玉」でもよいが、伝統に倣うと丁寧）。
- **成駒は一文字で正しく表す**（省略しない＝何が起きたか辿れるため）:
  - 成飛 = 龍、成角 = 馬、成銀 = 全、成桂 = 圭、成香 = 杏、成歩（と金）= と。
  - これらを文字の対応表として持ち、駒種から一文字へ写す。実装をコンパクトに保つため、一文字様式を採る（合字・二文字様式は採らない）。
- **向き**: 後手の駒は 180 度回転（SVG の transform）で表す。文字の上下反転で先後を示すのが、墨一色の様式に合う。

---

## 4. 盤面と座標

- 9×9 の盤。筋は右から 9〜1、段は上から 一〜九（伝統的な将棋盤の座標）。盤の縁に筋・段の番号を控えめに添える（局面図に倣う。なくてもよいが、あると親切）。
- 先手は下、後手は上。
- 駒台（持ち駒）: 盤の左右または下に、先手・後手の持ち駒を漢字＋数で控えめに示す（例: 「歩三」）。最小に振るなら簡素でよいが、不完全将棋では取得・相討ちで持ち駒が動くので、**持ち駒の表示は省かない**（結果局面が読めるために必要）。

---

## 5. 再生する棋譜データ（内蔵）

- 再生対象の一局を、手で書ける素朴なデータとして内蔵する（外部読み込み不要。JS 内の定数や、隣に置く小さな JSON/JS ファイルでよい）。
- データの最小構成:
  - **初期局面**: SFEN 文字列（ルール仕様の正本初期局面 `lnsgkgsnl/1r5b1/...` 等）、または駒配置の素朴な表現。本デモは描画専用なので、エンジンの SFEN パーサに依存せず、フロント側で読める形でよい。
  - **各ターンの記録**: ターンごとに「先手の着手」「後手の着手」「解決後の局面（SFEN 等）」の三つ組。
    - **重要**: 本デモはルールを計算しない。だから「解決後の局面」は**データとして与える**（エンジンが算出した正しい結果を、人があらかじめ書き込んでおく）。フロントは与えられた局面を描くだけ。これにより、衝突解決ロジックを一切実装せずに、正しい進行を再生できる。
  - 着手は USI 等の表記でデータに持ち、表示用に「7六歩」のような棋譜表記へ整形してもよい（整形は任意）。
- データを差し替えれば別の一局を再生できる構造にする（ハードコードした一局に固定しない）。

> このデータ構造は、将来エンジンを Wasm で繋ぐとき、「エンジンが各ターンの解決後局面を生成し、それをこの同じ描画層へ渡す」形へ自然に発展できる。描画層（殻）とルール（核）の境界を、データの形で先に引いておく。

---

## 6. 再生の所作（このデモの心臓：伏せ → 同時開示 → 抜き → 次局面）

各ターンを、次のサイクルで見せる。**衝突の解決過程は描かず、間を抜く。**

1. **現局面**: 解決済みの現在の盤面を静かに表示。
2. **伏せ**: 両者がこのターンの手を伏せている、という状態を示す（最小でよい。例: 「両者 着手」というテキスト、または伏せを示す控えめな印。盤上に仮の駒は動かさない）。
3. **同時開示**: 両者の着手を**同時に**見せる（これがゲームの本質＝同時性）。どの駒がどこへ動いたか／何を打ったかを、テキスト（例: 「☗7六歩 ☖3四歩」）で、また必要なら開示時に一瞬ハイライトで示す。**両者の手の開示は省略しない**（不親切回避の核心）。
4. **抜き**: 衝突の解決過程（ぶつかり・相討ち・戦国無双・玉の死などの処理）は**描かない**。すれ違いざまの斬撃の一瞬を抜く。
5. **次局面**: データに与えられた解決後の局面を表示する。前局面との差分（駒が消えた・持ち駒が増えた・玉が動いた）が、起きたことを盤上に語る。

- **操作**: 再生は、進む／戻る／（任意で）自動再生の最小コントロールで十分。Kifu for JS のように前後ボタンと手数表示があれば足りる。ボタンは無彩色・控えめに（デザインの邪魔をしない）。
- **開示と次局面の「間」**: 抜きを成立させるため、開示から次局面へ移る際に、ごく短い間（プログラム的な小さな待ち、またはユーザーの「進む」操作）を置くと、すれ違いの呼吸が出る。過剰な演出はしない（淡い切り替えで足りる。派手なアニメーションは入れない）。

> 親切と省略の境界: **抜くのは「衝突の解決過程」だけ。**「両者が何を指したか（開示）」と「結果局面が前と一意に繋がること」は必ず残す。見る者が、抜かれた一瞬を自分で埋められるなら、それは美であり不親切ではない。

---

## 7. 完了基準

1. 単一の静的成果物として、ビルドなしでブラウザで開け、Cloudflare に静的ファイルとして置ける。
2. 9×9 の盤・駒（漢字、成駒は一文字で正しく）・持ち駒が、水墨様式（墨一色・細線・生成り背景・serif）で SVG 描画される。先後は駒の向きで区別。
3. 内蔵棋譜データを、伏せ → 同時開示 → 抜き → 次局面 のサイクルで再生できる。前後の操作ができる。
4. 衝突の解決過程は描かれず（抜き）、両者の着手の開示と解決後の局面は明示される（不親切回避）。
5. 棋譜データを差し替えれば別の一局を再生できる。
6. 質感フィルタ・色・装飾・フレームワーク・ルール計算を含まない（非ゴールを守る）。
7. SVG が `viewBox`＋`max-width` でモバイル/デスクトップに収まる。

## 8. 次の殻（本実装では作らない・見通し）

- **エンジンの Wasm 接続**: Rust エンジンを `wasm-bindgen` で持ち込み、§5 の「解決後の局面」をデータ埋め込みではなくエンジンが算出する形へ。描画層は無改変で、データの供給元が人からエンジンへ替わるだけ（核と殻の境界を §5 で先に引いてある）。
- **操作可能化（ホットシート）**: 一人で両者の手を選んで指せる検証卓の Web 版。合法手ハイライトを足す。
- **秘匿対戦**: WebSocket/WebRTC の殻を足し、伏せを本物の秘匿に。観戦配信では「観戦者に開示前の手を見せるか」という秘匿境界の設計が新たに必要。
- **水墨の質感**: 滲み・かすれを SVG フィルタで足す（過ぎたるは及ばざるが如し、を見極めつつ）。
- **衝突の演出**: 「最後まで描かない」を見直し、一筋の動きを足すか判断（抜きのままで足りるかを、まず無しで確かめてから）。

各殻は独立に足せる。本第一枚は「水墨の盤面で棋譜が再生でき、抜きの様式が成立している」ことだけに集中する。
