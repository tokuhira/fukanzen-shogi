# 不完全将棋 実装指示書 — Web 盤への Wasm エンジン組み込み（棋譜再生のエンジン駆動化）

> 対象実行者: Claude Code
> 前提: ワークスペース = `engine`・`cli`・`tui`・`protocol`、静的フロントエンド `web/`（棋譜再生・水墨様式・エンジン未接続）。`engine` は I/O・乱数・ネットワークを持たない純粋なライブラリで、USI（着手）・SFEN（局面）の文字列語彙と、`legal_actions`・`resolve`・`check_status`・直列化を公開。ルールは v0.5。
> 関連文書: `不完全将棋_実装指示書_最小Web盤_棋譜再生水墨.md`（現 web/ の設計。特に §5 の棋譜データ構造）、現行ルール仕様 v0.5。
> 性格: ずっと敷いてきた「engine を純粋に保ち、いずれ wasm-bindgen で Wasm 化する」前方互換の回収。**描画層（殻）は変えず、棋譜データの“供給元”を、人が書いた解決後局面から、ブラウザ上で動くエンジンの計算へ差し替える**。エンジン本体は無改変。

---

## 0. 目的と範囲（厳密に (a) に限定）

- **作るもの**: ブラウザ上で Wasm 化した `engine` を動かし、**初期局面と各ターンの着手列だけ**を与えると、エンジンが各ターンの解決後局面を計算して、既存の水墨様式の盤面で再生できるようにする。現状データに人が手で書いていた「解決後の局面」を、エンジンの `resolve` 出力に置き換える。
- **この段階でやること**: Wasm 化という配線そのものを通し、エンジンがブラウザで v0.5 ルールどおり正しく動くことを、既存の描画の上で確かめる。
- **やらないこと（非ゴール、後の殻）**:
  - 操作可能化（クリックで着手を選ぶ、合法手ハイライト、一人で両者を指すホットシート）。本段階は**再生のみ**。`legal_actions` の UI 利用は次段階。
  - 対戦・通信（WebSocket/WebRTC）。
  - Wasm ビルドの CI 自動化（まず手元の `wasm-pack` ビルドで動かす。自動化は後）。
  - 見た目・操作の変更（水墨様式・再生 UI は現状維持）。
- **不具合の切り分けのため、本段階では「Wasm 配線」だけを変える**。操作実装を混ぜない（一つ変えて測る）。

---

## 1. クレート構成：Wasm ラッパは別クレート（engine は純粋なまま）

`engine` 本体に wasm-bindgen を入れない。`engine` に依存する**別クレート**を新設し、そこに Wasm 境界の薄いラッパだけを置く。

- 新規 `engine-wasm/`（名前は任意。`wasm/` 等でもよい）。`cdylib` クレートとして、wasm-bindgen 用のエクスポートを持つ。`engine` に依存する。
- これにより `engine` は誰にも依存されない純粋な核のまま。`cli`・`tui`・`protocol` と同じく、`engine` を包む殻の一つとして wasm ラッパが並ぶ（構造の一貫性）。
- `engine` 本体は**無改変**（公開 API のみ使用。Wasm のために内部を変えない。もし不足が見つかったら、内部変更ではなく報告）。

```
fukanzen-shogi/
├─ engine/        # 不変（純粋な核）
├─ cli/  tui/  protocol/   # 既存の殻
├─ engine-wasm/  # 新規: wasm-bindgen ラッパ（cdylib、engine に依存）
└─ web/           # 既存フロント（Wasm を読み込むよう拡張）
```

---

## 2. Wasm 境界：SFEN / USI の文字列でやり取り

JS ↔ Wasm の境界は、エンジンが既に持つ**文字列語彙（SFEN・USI）**で薄く保つ。Rust の内部型（Position・Action 等）を境界に晒さない。ラッパは「文字列を受け取り、内部型へ変換し、エンジンを呼び、結果を文字列で返す」だけ。

本段階（再生のみ）で必要な境界関数は最小限。少なくとも:

- **解決関数**: 入力 = 現局面の SFEN、先手の USI 着手、後手の USI 着手。出力 = 解決後局面の SFEN（および、表示に使うなら発生事象の要約＝取得・相討ち・スワップ・戦国無双・両玉スワップ引き分け・玉の死等）。内部で `resolve` を呼ぶ。
  - 出力に「何が起きたか」を含められると、現フロントの開示テキスト（例:「(相討ち)」）をエンジンの実結果から生成でき、データに手書きしていた説明も不要になる。事象要約の表現（文字列か小さな JSON か）は実装者判断。
- **終了判定**（任意・あれば再生の終局表示が正確に）: 局面（または直近の解決結果）から、確定的詰み・玉の死・両玉相討ち引き分け等の終了種別を返す。`check_status` 等を利用。
- **（本段階では不要）** `legal_actions` の境界: 操作可能化は次段階なので、今は出さなくてよい。出しても害はないが、範囲を絞るなら省略可。

> 文字列境界にすることで、`engine` の内部型の変更が JS 側に波及しない。境界が安定する。将来 `legal_actions` を操作 UI 用に足すときも、USI のリストを文字列で返す同じ作法で拡張できる。

## 3. 棋譜データ構造の変更（解決後局面を“与える”→“計算する”）

現 web/ の棋譜データ（`最小Web盤` 指示書 §5）は、各ターンを「先手着手・後手着手・**解決後局面（人が手書き）**」の三つ組で持っていた。これを変える。

- **新データ**: 「初期局面（SFEN）」＋「各ターンの (先手 USI 着手, 後手 USI 着手) の列」。**解決後局面はデータに持たない**。
- **再生時**: フロントが、初期局面から順に、各ターンの両着手を Wasm の解決関数へ渡し、返ってきた解決後局面 SFEN を描画する。各ターンの解決後局面は、その場でエンジンが計算する。
- **利点**: 人が解決後局面を手書きする必要が消える（手書きは誤りやすく、ルール変更で陳腐化する）。エンジンが常に v0.5 の正しい結果を出す。棋譜は「初期局面＋着手列」という最小の真実だけを持つ。
- **描画層は無改変**: 盤面 SVG・水墨様式・伏せ/開示/抜き/次局面 の所作・前後操作は、現状のまま。変わるのは「解決後局面 SFEN がどこから来るか」だけ（データの定数 → Wasm の計算）。**描画は SFEN を受け取って描く、という現在の境界がそのまま活きる**。

> これは「最小 Web 盤」指示書 §5 で予告した発展そのもの: 「エンジンが各ターンの解決後局面を生成し、同じ描画層へ渡す」。描画層とルールの境界を棋譜データの形で先に引いておいたので、ここで供給元を差し替えるだけで済む。

## 4. ビルドと読み込み（まず手元で）

- **Wasm ビルド**: `wasm-pack`（または同等）で `engine-wasm` を Wasm + JS グルーへビルドする。出力（`.wasm` と JS ラッパ）を `web/` から読み込める場所へ配置する（例: `web/wasm/` 等）。
- **読み込み**: フロントの JS が、生成された JS グルー経由で Wasm を初期化（`init`）してから、解決関数を呼ぶ。Wasm 初期化は非同期なので、初期化完了後に再生を有効化する（初期化中はローディング表示か、再生ボタンを無効化）。
- **本段階は手元ビルドで足りる**: `wasm-pack build` して `web/` に置き、ローカルでブラウザ確認 → Cloudflare へ配信、の手順。CI 自動化は後の殻（最初の Actions を Windows ビルドで手を動かして学んだのと同じく、まず手で通す）。
- 生成物（`.wasm`・JS グルー）を版管理に含めるか、ビルド時生成にするかは実装者判断。当面は手元ビルド成果物を `web/` に置いてコミットするのが簡単（後で CI 化する際に gitignore へ移せる）。

---

## 5. Cloudflare 配信の注意

- `web/` は引き続き**純粋な静的配信**（サーバ不要）。`.wasm` も静的アセットとして CDN から配れる。
- **MIME タイプ**: `.wasm` は `application/wasm` で配信される必要がある（`WebAssembly.instantiateStreaming` を使う場合は特に）。Cloudflare Pages は通常 `.wasm` を正しい MIME で配るが、もし問題が出たら、ストリーミングでない初期化にフォールバックするか、配信設定を確認する。
- 公開 URL は**カスタムドメイン** `https://fukanzen-shogi.tokuhira.net`（開発は `pages.dev`）。Wasm 追加後も、静的配信である点は変わらない。

## 6. 完了基準

1. `engine-wasm`（新規 cdylib クレート）が `engine` に依存し、SFEN/USI の文字列境界で解決関数を公開する。**`engine` 本体は無改変**。
2. `wasm-pack` 等で Wasm をビルドでき、`web/` から初期化して呼べる。
3. web/ の棋譜データが「初期局面＋着手列」だけになり（解決後局面の手書きを廃止）、各ターンの解決後局面を **Wasm のエンジンが計算**して再生する。
4. 既存の水墨様式・伏せ/開示/抜き/次局面 の所作・前後操作は**見た目も挙動も変わらない**（データ供給元だけが替わる）。
5. v0.5 ルール固有の局面（戦国無双、両玉スワップ＝相討ち引き分け、玉の死、成り等）を含む棋譜を与えても、エンジンが正しい解決後局面を返し、正しく再生される（人手の解決後局面なしで）。
6. ローカルで動作し、Cloudflare（カスタムドメイン）へ配信して動く。`.wasm` が正しく配信・初期化される。
7. 操作可能化・通信・CI 自動化は含まない（範囲を守る）。

---

## 7. 確認のための再生シナリオ（推奨）

エンジン駆動が正しいことを、人手の答えなしで確かめるため、v0.5 固有の事象を含む着手列を最低一つ用意して再生する:

- 通常の取得・歩のぶつかり（相討ち）。
- 成りが起きる手（成駒一文字の表示確認も兼ねる）。
- 戦国無双（玉が後ろ盾のない駒を斬り返す）。
- できれば両玉スワップ（相討ち引き分け＝一騎打ち）や玉の死を含む終局まで。

これらの解決後局面を**データに手書きせず**、エンジンの計算結果で再生できれば、Wasm 配線とルールのブラウザ動作が確認できる。（着手列自体は、既存の cli や tui／engine のテスト用駆動で生成したものを流用してよい。）

## 8. 次の殻（本段階では作らない・見通し）

- **操作可能化（ホットシートの Web 版）**: `legal_actions` を Wasm 境界に足し、クリックで合法手を選び、両者の手を入れて `resolve`。合法手ハイライト。TUI 検証卓の Web 版。
- **秘匿対戦**: `protocol` の純粋論理を Wasm 化し、WebSocket/WebRTC の殻で繋ぐ。観戦配信では秘匿境界（観戦者に開示前の手を見せるか）の設計。
- **Wasm ビルドの CI 自動化**: GitHub Actions で `wasm-pack` ビルド → `web/` 配信成果物を生成。
- 各殻は独立に足せる。本段階は「ブラウザでエンジンが v0.5 を正しく動かし、棋譜を計算再生できる」ことだけに集中する。
