# 不完全将棋 実装指示書 — 対局の永続と身元（記録係への一段目：存在と身元）

> 対象実行者: Claude Code（Sonnet）
> 前提: 配布 v0.10.1 リリース済み（淀川第三歩＝ライブ観戦とサーバ側アーカイブ、PROTOCOL_VERSION 3、RULE_VERSION 0.6）。関連する現物:

> - `web/online.js`: `spectate_meta`/`spectate_turn`/`spectate_result` を**先手のみ**が送る（`mySide === 'sente'`）。`_sendSpectateResult(kind, outcome)` を終局時に送る。
> - `web/board.js`: 第一歩の `buildArchiveText()`（`build_archive` で版タプル付きアーカイブの**正準テキスト**を生成。保存機能で使用中）。
> - `engine::archive`: 書式 v1（`fukanzen-shogi-archive 1` ヘッダ＋本体）。`build_archive`/`parse_archive`。
> 関連文書: `不完全将棋_棋譜対局データ設計_方針.md`（淀川。「対局中のハッシュは揮発してよい、終局で確定」——本実装はその確定ハッシュを対局の身元にする）、`不完全将棋_版図_世界観と設計方針.md`（核と交換可能な殻／記録係）、`不完全将棋_実装指示書_ライブ観戦とサーバ側アーカイブ_淀川第三歩.md`（§3.3 記録・§6 取り出し・§9 基盤中立）。
> 性格: 記録係への道の**一段目＝存在と身元**。いまサーバにあるのは「拭える一枚の棋譜用紙」であって書庫ではない。次の対局が始まると `spectate_meta` が用紙を拭き、誰かがコピーを取っていなければ**その一局はサーバから消える**——第一歩で直したかったデータ喪失が、観戦の口を開けたことで一層上で再発している。これを止める。**対局を、部屋（使い回される殻）から解いて、それ自身の身元を持つ本流として、部屋とは別の書庫へ綴じる。** 部屋はいくら拭かれても、綴じ終えた記録は書庫に残る。**信頼（二証人）・封蝋（署名）・名乗り（principal）はこの一段目では扱わない**——それらは書庫が立って初めて棲む場所を得る。

---

## 0. 目的と範囲

- **作るもの**:
  1. **対局の身元**（§2）——確定した対局は**正準アーカイブ本文の内容ハッシュ**（SHA-256）で content-address する。放棄断片は暫定 ID。
  2. **書庫**（§3）——部屋 DO とは切り離した永続ストア（新 KV `ARCHIVES`）。部屋のライフサイクルに触れられない。
  3. **綴じる契機**（§4）——終局・再戦開始・全員離脱のいずれでも、**拭く／捨てる前に必ず綴じる**。不変条件「レコードは書庫へ綴じてからでなければリセットしない」。
  4. **取り出し**（§7）——`GET /archive/:id`（Worker が KV を直接読む。部屋 DO を介さない）。content-address ゆえ返ってきた本文を再ハッシュして id と照合できる。
- **位置づけ**: 記録係への二段構えの一段目。「対局が確かに存在し、身元を持つ」を据える。これは信頼・帰属・時間より前に立つ foundational な層で、以降の畝（二証人・署名・principal・時間）はすべてこの書庫の上に積める。
- **作らないもの（非ゴール。§11 に申し送り）**:
  - **書庫の一覧・検索・鑑賞 UI**（過去局ブラウザ）。最小は「id で綴じ、id で取り出す」。
  - **二証人の交差確認**（両者の本文一致検証）。§5 で片側送信を content-address するに留め、種だけ残す。
  - **記録係署名**（完全性の封蝋。予約のまま）。
  - **principal による帰属**（席は匿名 `-`）。
  - **保持ポリシー・匿名化・再識別性**（版図の名乗りの層が来るまで素朴に恒久保存）。
  - **時間制**（持ち時間なしのまま）。

## 1. 診断と、据える不変条件

いまの構造の欠陥は一点に尽きる——**対局の身元が、部屋の身元に癒着している**。部屋（落ち合い点＝盤）は本来ただの rendezvous で、使い回されて当然の殻。対局（記録）は不変で、それ自身の身元を持つべき中身。ところが本流が殻の中に閉じ込められ、`spectate_meta` が殻を拭うと本流も消える。**核と交換可能な殻**の原則が、ここだけ破れている。

据える不変条件はこれだけ:

> **部屋のレコードは、書庫へ綴じ終えてからでなければ、リセット・破棄しない。**

この一文を、拭く／捨てる三つの契機（§4）すべてで守る。守れば、部屋はいくら再利用されても何も失わない。**重い部屋管理は要らない**——対局を身元付きで書庫へ移せば、部屋の使い回しは無害になる。

---

## 2. 対局の身元（二相の身元）

- **ライブ中の身元＝落ち合い点（揮発）**: 部屋キー／観戦トークン。使い回され、失効してよい。現状のまま。
- **綴じた記録の身元＝内容ハッシュ（不変）**: 確定した対局は、**第一歩の正準アーカイブ本文（`build_archive` 出力）を SHA-256 した 16 進文字列**を id とする。淀川が「対局中のハッシュは揮発してよい、終局で確定」と置いた、あの**終局確定ハッシュがそのまま恒久のラベル**になる。
  - content-address ゆえ、誰でも取り出した本文を SHA-256 して id と照合でき、**改竄が検知できる**（将来の署名・二証人の土台を兼ねる）。Git がオブジェクトを内容ハッシュで名付けるのと同じ所作。
  - ハッシュ対象は engine 定義の正準テキスト、ハッシュ関数は標準 SHA-256。**DO は本文を解さない**（不透明ブロブを content-address して置くだけ＝書式の正本は engine のまま／第三歩 §6 と整合）。
- **放棄断片の身元＝暫定 ID**: 終局に至らず放棄された対局は確定ハッシュを持たない（確定していないのだから当然）。`crypto.randomUUID()` を id とし、envelope に `finalized: false` を立てる。「確定した対局は内容ハッシュ、未確定の断片は暫定ラベル」——身元の別が対局の状態を素直に映す。

---

## 3. 書庫（部屋から切り離す）

- 新 KV 名前空間 **`ARCHIVES`** を追加（`SPECTATE_TOKENS` とは別）。`wrangler.toml` に binding（§8）。
- 格納 envelope（id をキーに）:
  ```json
  {
    "finalized": true,
    "text": "<正準アーカイブ本文>",        // finalized=true のとき
    "record": { "version": ..., "initial_sfen": ..., "turns": [...], "result": ... },  // finalized=false のとき
    "archived_at": "<ISO8601>"
  }
  ```
  - 確定局は `text`（正準本文）、放棄断片は `record`（構造化）を持つ。
- **書庫は部屋 DO のライフサイクルから独立**。書き込みは DO（綴じる契機、§4）、読み出しは Worker（§7）。部屋の再利用・リセット・削除は `ARCHIVES` に一切触れない。これが「殻＝部屋／中身＝身元付きの本流」の分離の実体。
- サイズ: 一局 ≤ 約 14KB（500組手上限）で KV の値上限に十分収まる。恒久保存（TTL なし）。

---

## 4. 綴じる契機（拭く／捨てる前に必ず綴じる）

部屋レコードに `archived: boolean` フラグを一つ足し（storage キー）、二重綴じを防ぐ。以下いずれの契機でも、**綴じてからリセット／破棄**する。

1. **終局（`spectate_result` 受信）**: クライアントが同梱した正準本文（§5）を `id = sha256hex(text)` で `ARCHIVES` に `finalized:true` 綴じ。部屋レコードの `result` も確定（現状どおり、なお接続中の観戦者へ最終結果を映すため）。`archived = true`。綴じた id をプレイヤー・観戦者へ `archived { id }` で通知（恒久リンクになる）。
2. **再戦開始（`spectate_meta` 受信）**: `_applyToRecord` がレコードを初期化する**前に**、現レコードが `turns` を持ち かつ `archived === false` なら、**放棄断片として** `ARCHIVES` に暫定 ID で `finalized:false` 綴じ。その後リセット（`archived` を false に戻す）。
3. **全員離脱（`webSocketClose` で残プレイヤー 0）**: 現レコードが `turns` を持ち かつ `archived === false` なら、放棄断片として暫定 ID で綴じる。その後 `gameStarted` 削除（現状どおり）。

> これで、正常終局は内容ハッシュで、放棄・途中再戦は断片として、**いかなる経路でもレコードが綴じられずに消えることはない**。データ喪失の再発が構造的に閉じる。

---

## 5. クライアントの変更（online.js / board.js）

- 終局時、**先手クライアント**が正準アーカイブ本文を作って同梱する。`board.js` は既に `buildArchiveText()`（`build_archive`）で保存用の正準本文を作れるので、それを再利用する。
- `_sendSpectateResult(kind, outcome)` を `_sendSpectateResult(kind, outcome, text)` に拡張し、`{ type: "spectate_result", kind, outcome, text }` を送る（`text` は最終結果まで含んだ完成アーカイブ）。
- **後方互換**: `text` は追加フィールドで、対局チャネル（peer 間の commit/reveal）には無関係。DO は `text` があれば `finalized:true` で content-address、無ければ現レコードから `finalized:false` 断片として綴じる（§4-2/4-3 と同じ経路）。ゆえに **PROTOCOL_VERSION は 3 のまま**で済む（ワイヤは加算的・任意）。web は一括デプロイなので、配備後は全終局が内容ハッシュ id を得る。

---

## 6. DO の変更（room.ts）

- `SpectateRecord` に `archived: boolean` を追加（`_loadRecord` の既定 false）。
- `webSocketMessage` の `spectate_result` 分岐（`_applyToRecord` 経由）で:
  - `msg.text`（string）があれば `id = sha256hex(msg.text)`（Web Crypto: `crypto.subtle.digest("SHA-256", ...)` を hex 化）、`ARCHIVES.put(id, {finalized:true, text: msg.text, archived_at})`。無ければ現レコードで `finalized:false` 断片綴じ（暫定 ID）。
  - `result` を確定、`archived = true` を storage へ。プレイヤー・観戦者へ `archived { id }` を送る。
- `spectate_meta` 分岐（`_applyToRecord` 初期化の**前**）に、§4-2 の断片綴じを挿入。
- `webSocketClose` の全員離脱枝に、§4-3 の断片綴じを挿入。
- ヘルパ `_archiveFinalized(text)` / `_archiveFragment(record)` を用意して三契機から呼ぶ。DO は `ARCHIVES` binding を `env` に持つ（§8）。**DO は `text` を解さない**（ハッシュして置くだけ）。

---

## 7. 取り出し（部屋を介さない）

- Worker（`index.ts`）に **`GET /archive/:id`** を追加し、`env.ARCHIVES.get(id)` を返す（部屋 DO を経由しない。書庫は部屋から独立）。
- 返す envelope の `finalized:true` なら `text`（正準本文＝第二歩の `parse_archive`／`loadPlies` がそのまま消費、または第一歩で保存可能）。`false` なら `record`（構造化）。
- content-address の検証: `finalized:true` の場合、取り出した `text` を SHA-256 して `id` と一致すること（誰でも改竄検知できる）。
- 既存の `GET /room/:key/archive`（部屋のいまの一局）は現状維持（ライブ・診断用）。恒久の記録は `/archive/:id`。

---

## 8. 設定（wrangler）

- `ARCHIVES` KV 名前空間を作成し、Worker と DO の両方から使える binding を `wrangler.toml` に追加（`SPECTATE_TOKENS` と同様）。`GameRoom` の `Env` に `ARCHIVES: KVNamespace` を足す。

## 9. テストと受け入れ基準

- **データ喪失の閉塞（最重要）**: 一局を終局させ `archived { id }` を受ける → **続けて同じ部屋で再戦を始めて `spectate_meta` が部屋レコードを拭いても**、`GET /archive/:id` で前局が取り出せる（消えない）。
- **放棄の取りこぼし防止**: 数手指して全員離脱 → 断片が暫定 ID で綴じられ、`/archive/:id`（暫定 ID）で `finalized:false` の `record` が取れる。
- **途中再戦**: 未終局のまま `spectate_meta`（再戦）→ 直前の未確定局が断片として綴じられてから部屋がリセットされる（前局が消えない）。
- **content-address 検証**: `finalized:true` の `text` を SHA-256 して `id` に一致する。
- **回帰**: 既存のライブ観戦・`/room/:key/archive`・対局（commit/reveal・再接続）が壊れない。`text` を送らない旧クライアントでも、断片として綴じられデータは失われない。

- **配布版 v0.10.1 → v0.10.2（パッチ）** を既定とする。理由: これは観戦の口を開けたことで再発したデータ喪失を塞ぐ**修理**であり、対戦互換・ルール・書式・プロトコルを変えない加算的な直し。

## 11. 非ゴールと次段への申し送り（記録係への残りの道）

- **書庫の一覧・鑑賞**: 過去局のブラウズ・検索。KV の prefix 列挙で素直に載る（当面は id 直取り）。
- **二証人の交差確認**: 両プレイヤーの正準本文（＝同一のはず）の一致を確認して初めて綴じる（審判なしの改竄検知）。§5 で片側の本文を content-address する形にしてあるので、もう一方の本文を受けて hash 一致を要求するだけで育つ。版図「審判なし・侘びで競う」の芯。
- **記録係署名**: content-address した id に署名（完全性の封蝋）。内容ハッシュが既に土台。
- **principal による帰属**: 席が埋まれば、正準本文の `sente`/`gote` に principal が載り、書庫の記録が「誰の対局か」を宿す。
- **保持ポリシー・匿名化・再識別性**: 名乗りの層（版図）と絡む。当面恒久保存、方針は後日。
- **時間制・人 vs AI の速度差**: 持ち時間なしを超える層。

## 12. 不変の原則（本実装が守るもの）

1. **綴じてから拭く**: 部屋レコードは書庫へ綴じ終えてからでなければリセット・破棄しない。いかなる契機（終局・再戦・離脱）でも守る。
2. **殻と中身を分ける**: 部屋（落ち合い点）は使い回される殻、対局（記録）はそれ自身の身元を持つ中身。書庫は部屋のライフサイクルから独立。
3. **身元は内容から**: 確定局の身元は正準アーカイブ本文の内容ハッシュ。誰でも再ハッシュで検証でき、署名・二証人の土台を兼ねる。放棄断片は暫定 ID。
4. **DO は書式を知らない**: 正準テキストの生成は engine（クライアント）、DO は不透明ブロブを content-address して置くだけ。書式の正本は engine。
5. **加算的・後方互換**: 対局チャネルは不変。`spectate_result` に本文を加える任意フィールドと、書庫・契機を足すのみ。ルール・書式・プロトコルは据え置き。
6. **設計は全体、実装は最小から**: 一段目は存在と身元だけ。書庫 UI・二証人・署名・帰属・時間は据え置き、この書庫の上に後から積む。

*記録係への一段目——対局の永続と身元。観戦の口を開けたことで再発したデータ喪失を、対局を部屋（使い回される殻）から解いて、それ自身の身元（正準アーカイブ本文の内容ハッシュ）を持つ本流として、部屋とは別の書庫（KV `ARCHIVES`）へ綴じることで閉じる。据える不変条件は「綴じてから拭く」。終局・再戦・離脱のいずれでも取りこぼさない。取り出しは `/archive/:id`（部屋を介さず、content-address ゆえ検証可能）。信頼・封蝋・名乗り・時間は、この書庫の上に後から積む。ルール・書式・プロトコルは不変、配布版はパッチ v0.10.2（記念として minor の別案も）。*
