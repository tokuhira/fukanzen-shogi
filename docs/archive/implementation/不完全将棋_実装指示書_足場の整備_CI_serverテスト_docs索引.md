## 0. 目的と範囲

- **作るもの**:
  1. **CI 門番**（§1）——Rust に `cargo test` + `cargo clippy -D warnings` + `cargo fmt --check`、server に `tsc --noEmit`（＋ §3 のテスト）を、push / pull_request で回す。既存の Windows リリースビルドは残す。
  2. **server テスト**（§3）——0 から、DO の最も危うい要所（秘匿境界・二証人・綴じてから拭く・招待）に絞って張る。純粋ロジックの抽出でテスト可能にする。
  3. **docs 索引**（§4）——`docs/README.md` を一枚。生きた設計文書／ルール変遷（歴史）／実装指示書（完了記録）を仕分ける dashboard。
  4. **取り残しの一掃**（§2）——`cli` の「暫定」drift を v0.6 に整合、同種の取り残しを掃く。
- **位置づけ**: board.js に手を入れる前に足場を固める。CI が門番として立てば、後の分割で何かを壊しても即座に分かる。整備が次の相談の下ごしらえになる。
- **作らないもの（非ゴール）**:
  - **board.js の分割**——別途方針相談（§7 に申し送り）。
  - **境界を跨ぐ重複定数の codegen**（`MAX_TURNS`・`MAX_ARCHIVE_TEXT_BYTES` の Rust↔TS 二重）——DO が Wasm を持てない構造上のもの。comment-sync のまま据え置き（codegen は過剰）。ただし §2 の掃除ついでに、両者にクロス参照コメント（「もう一方と一致させること／出典は engine::terminate::MAX_TURNS」）が両側に揃っているか確認するだけはしてよい。
  - ルール・プロトコル・棋譜書式・製品挙動の変更（整備は挙動を変えない）。

## 1. CI 門番

- 新規ワークフロー `.github/workflows/ci.yml` を追加。トリガは `push` と `pull_request`。
- **Rust ジョブ**（`ubuntu-latest`）:
  - `cargo fmt --all -- --check`
  - `cargo clippy --workspace --all-targets -- -D warnings`
  - `cargo test --workspace`
  - ビルドキャッシュ（`Swatinem/rust-cache` 等）で高速化してよい（任意）。
- **server ジョブ**（`ubuntu-latest`, Node）:
  - `cd server && npm ci && npm run typecheck`
  - §3 のテストを追加したら `npm test` も。
- **既存の `windows-build.yml` / `release.yml`（Windows TUI ビルド）はそのまま残す**（役割が違う——門番ではなく成果物ビルド）。
- rustfmt/clippy 設定: 既定で十分。スタイルを固定したいなら最小の `rustfmt.toml` を置いてよいが必須ではない。**設定を足すより、まず既定で緑にすることを優先**。
- **重要な手順は §5（baseline を先に）**。門番を厳格化する前に、木を綺麗にする。

## 2. 取り残しの一掃（ルール確定が残した「暫定」drift）

- `cli/src/main.rs:115` の TODO コメントを **v0.6 に整合**。千日手は「暫定引き分け・要再検討」ではなく**引き分けで確定**（ルール v0.6 §5.6）。コメントを確定内容へ書き換える（「§7 未確定」への参照を除去し、v0.6 §5.6 を指すか、単に確定事実を述べる）。挙動は変えない（既に引き分けを返しているはず）。
- 同種の取り残しを一掃: リポジトリ全体で `暫定`・`要再検討`・`§7（未確定）`・`TODO`・`FIXME` を grep し、**v0.6 で決着した項目への参照が残っていないか**確認。engine の `check_sennichite` doc は更新済みのはずだが、他クレート・web・server・docs にも取り残しがないか掃く。まだ未決の項目（ルール v0.6 §7 のプレイテスト待ち等）への言及は**残してよい**——それは正しく未決だから。消すのは「もう決まったのに未決と書いてある」ものだけ。

## 3. server テスト（危うい要所に絞って）

- **足場**: Cloudflare Workers/DO の公式テスト経路 `vitest` + `@cloudflare/vitest-pool-workers` を `server/` に導入（devDependencies、`vitest.config`、`npm test` スクリプト）。
- **テスト可能化のための最小リファクタ**: `room.ts` の**判定ロジックを純粋関数へ抽出**してから単体テストする。WebSocket ハイバネーション API 越しの統合テストは扱いが重いので、まず純粋関数を直接叩く。抽出候補:
  - routing 判定（送信者の役割＋メッセージ型 → 転送先が「相手プレイヤーのみ／観戦者へ fan-out／破棄」のどれか）。
  - 二証人の評決（二つの証言テキスト → `witnesses:2 一致`／`不一致（disputed）`／`witnesses:1` のどれか）。
  - content-address（正準本文 → `sha256Hex`）。
  - envelope 構築（確定局／disputed／断片）。
  - 「綴じてから拭く」ゲート（`recording` 真偽と契機 → 綴じる／揮発）。
- **守るべき不変条件（＝実行される仕様。記録係二段目・第三歩・一段目より）**:
  1. **秘匿境界**: プレイヤーの commit/reveal は観戦者へ届かない。観戦者の入力は破棄され、プレイヤーへ届かない。**観戦者の `request_reset` は対局を壊さない**（v0.10.1 の退行防止を固定）。
  2. **二証人**: 一致する二証言 → `witnesses:2` で `sha256` id 綴じ。不一致 → `record_disagreement`＋disputed（権威ある確定局を作らない）。片方のみ → `witnesses:1`。
  3. **綴じてから拭く（招かれた対局のみ）**: 招待済みは終局・再戦・離脱で綴じられる。**未招待は綴じられず揮発**するが、ライブ観戦は動く。`archived` フラグは KV 書き込み成功後にのみ立つ。
  4. **招待の相互同意**: `record_invite`+`record_accept` → `recording=true`＋`record_confirmed`。`record_decline` → 未記録。
  5. **content-address**: `/archive/:id` の `finalized` 本文を `sha256` すると `id` に一致。
- **範囲**: 網羅ではなく**最も危うく、最もテストの無い面**に絞る。上の 5 群が入れば、board.js を触るときの回帰検知にもなる。統合テスト（実 WebSocket フロー）は可能なら足すが、純粋関数の単体テストを優先（高価値・低摩擦）。
- server ジョブ（§1）に `npm test` を追加。

## 4. docs 索引

- `docs/README.md` を新設。バックログと同じ **dashboard** 発想で、30 文書を三群に仕分ける（詳細は各文書、ここは索引）:
  - **生きた設計文書（現行の権威）**: 版図・淀川（棋譜対局データ設計）・プリンシパル設計・**ルール仕様 v0.6（現行）**・バージョン互換性管理方針・バックログ。
  - **ルール変遷（歴史）**: ルール仕様 v0.1〜v0.5（**v0.6 に置換済み＝superseded** と明記）＋各実装変更指示（戦国無双・両玉スワップ）。
  - **実装指示書（完了記録・時系列）**: Phase1/2/3 から淀川三歩・記録係二段まで。可能なら各文書に**着地した配布版（git タグ）**を併記すると、どの一歩がどの版で結晶したかの地図になる。
- 目的: 三者（Opus・あなた・Sonnet）と将来の来訪者の可読性。**歴史は消さず索引する**（棋譜と同じ——過去局は残す）。v0.6 が現行と一目で分かるように。

## 5. baseline を先に（門番を立てる前に木を綺麗に）

門番を厳格化した瞬間に赤信号だらけ、を避けるため、**この順で**:

1. `cargo fmt --all` を一度実行し、木全体を正規化（差分をこの整備のコミットに含める）。その後 CI に `--check` を入れる。
2. `cargo clippy --workspace --all-targets` を実行し、警告を潰す（真に無害なものだけ理由コメント付き `#[allow(...)]`）。緑にしてから CI に `-D warnings` を入れる。
3. `cargo test --workspace` で 124 本が緑を確認。落ちるものがあれば、それは整備で拾うべき既存の綻び——直すか、正しく `#[ignore]` して理由を残す。
4. `server`: `tsc --noEmit` を緑にし、§3 のテストを緑にしてから CI ジョブへ。

> 本書の実行者（Sonnet）は cargo が使える環境で作業する前提。**この baseline 手順の結果（fmt 差分・clippy 警告・テスト結果）を、実際に走らせて確認してから門番を厳格化すること。**

## 8. 不変の原則（本整備が守るもの）

1. **足場を固めてから前進**: リスクのある分割（board.js）の前に、門番とテストを立てる。整備は次の一歩の下ごしらえ。
2. **テストは実行される仕様**: 124 の Rust テストと、新たな server テストを CI が守る。とりわけ手作業レビューで綻びの出た server の要所を、実行される不変条件にする。
3. **歴史は消さず索引する**: 旧ルール仕様・完了した実装指示書は消さない。索引で「生きている／歴史」を仕分ける（棋譜と同じ思想）。
4. **挙動は変えない**: ルール・プロトコル・棋譜書式・製品の振る舞いは不変。変わるのは足場だけ。
5. **整備も最小から**: 網羅ではなく、最も危うく最も守られていない面（CI 不在・server 0 テスト・取り残しの暫定）に絞る。board.js・重複定数 codegen は別（過ぎたるは及ばざる）。
