# 不完全将棋 終局判定の単一正本化アーク — 概観と段組（設計方針／各段指示書の錨）

> 対象読者: このアークを実装する Claude Code（Sonnet 5）、各段の指示書、次の Opus セッション。
> 位置づけ: **設計方針の錨**。終局判定（盤面終局＋投了）を一つの窓口へ寄せる決定と、その層の置き方・段組・版の物語をここに綴じる。
> 前提の現在地: 通信核の一本化アーク＋延長の一部が着地（配布 v0.12.1）。**4b（TUI 先手の観戦配信）に入る前に**、TUI の最長手数（500 組手）未対応が発覚し、その根に「終局判定が単一正本でない」問題があると判明した。
> 関連文書: `design/不完全将棋_ルール仕様_v0.6`（§5 終局・5.4 両者投了・5.7 最長手数500組手）、`design/不完全将棋_版図_世界観と設計方針`（核と交換可能な殻）、`不完全将棋_バックログ_伏線と未決`。

---

## 0. なぜこのアークか（発覚した綻び）

4b の指示書を書く際、TUI の `GameOverKind` に `MaxTurns` があると**仮定**して結果語彙の対応表を書いたが、現物にそれは無かった。測ると:

- engine は 500 組手を実装している（`terminate.rs`: `MAX_TURNS=500`・`check_max_turns`・統合判定 `evaluate(kifu)->Terminal` が `DrawKind::MaxTurns` を含む）。
- **web は既に `evaluate` を使っている**（board.js `currentResult()` → engine-wasm `evaluate_terminal`）。盤面終局については単一正本の上にいる。
- **TUI だけが終局判定を手組みで並べている**（`app.rs::resolve_turn` が `check_king_death`→`check_sennichite`→`check_status` を個別に呼び、`evaluate` を使わない）。ゆえに v0.6 が `evaluate` に足した最長手数を**静かに取りこぼした**。
- **投了はどのクライアントでも `evaluate` の外**で捌かれている（TUI online.rs `resolve_completed_turn`・TUI local `app.rs`・web `resultOverride`）。理由は `resolve()` が `Action::Resign` を `unreachable!` で弾くから——engine は投了を「盤の出来事ではない」と扱っている。

これは通信核の一本化アークと同じ構図の、終局判定版：**単一の正本から離れた殻（TUI の手組み）が静かにずれる**。クラウドで TUI がブラウザと同席する今、500 組手に達すると片側だけ終局して desync する実害もある。

## 1. 設計判断（投了は engine でなく protocol 層へ）

「投了も engine に組み込むのが正統では」という直感を検討し、**決定的優位があるときだけ engine へ**という基準で比較した結果、**投了は protocol 層で扱う**と決めた。

- **本将棋/USI でも投了は着手でなく宣言＝プロトコルの範疇。** engine 自身が `Action::Resign` を `resolve()` で弾いている＝「盤の出来事ではない」と既に表明している。`evaluate` に投了を入れるには「resolve を呼ぶ前に特別扱いする」ワートが要り、engine の純粋さに逆らう。
- **勝敗の共通通貨 `(ResultKind, Outcome)` は `engine::archive` にあり、既に `ResultKind::Resign` を含む。** 投了と盤面終局は `Terminal` enum ではなく **result のレベルで合流**する。だから engine の `Terminal` に投了を足す必要はない。
- **単一正本は protocol の合成窓口で達成する。** engine 案に決定的優位はない（単一窓口も簡潔さも protocol 案が同等以上・層が澄む）。

## 2. 目標構造（層の置き方）

```
┌───────────────────────────────────────────────────────────┐
│ クライアント（TUI online.rs/app.rs・web board.js）          │
│   終局を知りたい → protocol::game_result(kifu) を呼ぶだけ     │
├───────────────────────────────────────────────────────────┤
│ protocol::game_result(kifu) -> (ResultKind, Outcome)        │
│   1. 最後の組手が投了か（protocol の領分）                    │
│        → 投了なら (Resign, 勝者/引き分け) を返す              │
│   2. でなければ engine の盤面終局へ委譲:                      │
│        terminal_to_result(evaluate(kifu))                   │
├───────────────────────────────────────────────────────────┤
│ engine（盤面ルールの純粋核・投了を知らない）                  │
│   evaluate(kifu) -> Terminal         （盤面終局・無変更）     │
│   terminal_to_result(&Terminal) -> Option<(ResultKind,Outcome)> （移設）│
└───────────────────────────────────────────────────────────┘
```

- **engine は盤面終局だけ**（玉の死・詰み・千日手・最長手数）。投了を知らないまま。`Terminal`/`LossKind`/`DrawKind` は無変更。
- **protocol が投了を合成する**（投了はプロトコルの範疇）。`game_result` が単一の窓口。TUI（ネイティブ）も web（protocol-wasm）もこれを呼ぶ。
- `evaluate` は「最後の組手が投了の kifu を渡されない」前提で動く。`game_result` が投了を先に捌くのでこの前提が守られる（`evaluate` は投了組手で `resolve` を呼ぶと panic するため）。

## 3. 段組（依存 A→B→(C・D 並行可)→4b）

- **Step A — engine: `terminal_to_result` を engine へ移設**
  engine-wasm にインライン展開されている `Terminal→(ResultKind,Outcome)` マッピングを、`engine::terminate::terminal_to_result(&Terminal)->Option<(ResultKind,Outcome)>` として engine へ移す（Ongoing は None）。盤面終局限定・純粋。`evaluate` は無変更、前提を doc コメントで明記。`cargo test -p engine` で完結。**要石**。
- **Step B — protocol: `game_result` を新設**
  `protocol` に `game_result(kifu: &Kifu) -> (ResultKind, Outcome)`：最後の組手の投了を判定（`is_resign`）→ 投了なら `(Resign, 勝者/引き分け)`、でなければ `terminal_to_result(evaluate(kifu))`。投了と盤面終局を合流させる**単一の正本**。`cargo test -p protocol` で完結。
- **Step C — engine-wasm/protocol-wasm＋web: web を `game_result` 一本へ**
  `protocol-wasm` に `game_result` を公開。web の `currentResult()` を、`resultOverride`（投了別扱い）＋`evaluate_terminal`（盤面）の二経路から `game_result` 一本へ寄せ、投了組手を plies に統一する。engine-wasm のインラインマッピングは Step A の関数呼び出しに置換。wasm 再ビルド・`?v=` 前進。**web の挙動は保存**（同じ結果・経路が統一されるだけ）。
- **Step D — TUI: 終局判定を `game_result` へ委譲**
  `app.rs::resolve_turn` の手組みの並びと、投了検出（online.rs `resolve_completed_turn`・app.rs local）を `protocol::game_result` へ委譲。**max_turns も投了も統一取得**、手組みの乖離が消える。将来のルール追加も取りこぼさない。LAN・クラウド・ローカルで検証。
- **その後 4b**: `spectate_result` は `game_result` を呼ぶだけ（私が綻びさせた対応表は不要になる）。TUI が単一正本に乗るので max_turns も正しく配信される。

## 4. 版の物語

- **Step A・B**: 純粋核への**追加のみ**（挙動不変）。配布 bump なし。
- **Step C**: web は挙動保存（結果同一・経路統一）。**web `?v=` 前進**（wasm 再ビルド）。配布据え置き。
- **Step D**: TUI が 500 組手を正しく終局させる**バグ修正**＋投了統一。**配布パッチ bump**（v0.12.2）。`--version` を揃える。
- **4b**: 観戦配信の新能力。Step D と同じ v0.12.2 に束ねるか別建てかは作り手判断。

## 不変の原則（このアークで守る）

- **層の純粋さ**: engine は盤面ルールだけ（投了を知らない）。投了は protocol の合成が扱う。本将棋/USI どおり。
- **単一正本**: 終局を知りたい者は `protocol::game_result` を呼ぶ。TUI も web も同じ窓口。手組みの並びを残さない。
- **共通通貨は `(ResultKind, Outcome)`**: 投了と盤面終局は result のレベルで合流。`Terminal` は盤面限定のまま。
- **evaluate の前提**: 「最後の組手が投了の kifu を渡さない」。`game_result` が投了を先に捌いてこれを守る。
- **挙動保存**: Step C は web の結果を保存（経路統一のみ）。Step D は投了・盤面終局の判定結果を保存しつつ max_turns を正す。
- **過ぎたるは及ばざる**: engine に投了を入れない（決定的優位がないため）。防御コードは前提の doc 明記で足りる。
