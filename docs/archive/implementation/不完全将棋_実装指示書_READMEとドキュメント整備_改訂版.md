## 3. リポジトリ構成の記載（Rust と Web を別グループに）

実態に合わせ、二つのグループで示す。

**Rust ワークスペース（核と殻）:**
- `engine/` — 純粋なルールエンジン（I/O・乱数・ネットワークなし）。`legal_actions`・`resolve`・`check_status`・直列化。**核**。
- `cli/` — 機械可読の検証用 CLI（標準入出力でテキスト駆動、スクリプト・差分テストの土台）。
- `tui/` — ratatui による対話的検証卓。さらに**ネットワーク対戦モード**を持つ（コマンドライン引数で対戦モードに入る）。TCP の殻は `tui/net.rs`。
- `protocol/` — 通信プロトコルの**純粋論理**（commit-reveal・盤面ハッシュ相互検証・Ack・本人認証・中断救済・バージョン交渉）。ソケットから切り離してテスト可能。

**Web フロントエンド（静的）:**
- `web/` — 静的フロントエンド（HTML/CSS/JS、ビルド不要）。現状は**棋譜再生のみ・水墨様式・Rust エンジン未接続**。Cloudflare Pages へ配信。いずれ Wasm 化したエンジンを接続予定（描画層とルールの境界は棋譜データの形で既に分離）。

**ドキュメント:**
- `docs/` — ルール仕様・実装指示書・変更指示・方針文書（§5）。

> 設計哲学「共通の核と交換可能な殻」を一言添える: engine が核、cli・tui・protocol・web はすべて核を包む交換可能な殻。エンジン本体には I/O を一切加えない。

### 3.1 公開 URL と本番／開発の区別（方針）

- **公開（本番）URL = `https://fukanzen-shogi.tokuhira.net`**（自己管理のカスタムドメイン）。README・対外発信・SNS で示す URL は**こちらに統一**する。理由: 自分の所有ドメインなので、将来ホスティング先を変えても URL を保てる／撒いた住所が長く生き続ける。
- **開発 URL = `https://fukanzen-shogi.pages.dev`**（Cloudflare 割り当てサブドメイン）。手元での確認・開発用に残す。README には**載せない**（または「開発用」と明示して脇に置く）。
- 現状は本番・開発で**中身に区別はない**が、URL を二系統持っておくこと自体に価値がある（後から役割を割り当てられる）。これは「核と殻」をデプロイ面へ広げたもの——**世に見せる本番と、自分が試す開発を分けられる土台**を、必要が生じる前に用意しておく。
- **将来の布石（今は作らない）**: 本番／開発の分離が整っていれば、いずれ AB テスト（盤面様式の濃淡、ルールのバリエーション等を異なる利用者へ出し分け、実際の振る舞いから比較）へ発展できる。ただし AB テストは**一定数の訪問者が集まってから**意味を持つため、現段階では URL 二系統の土台までで十分。README には AB テストの記載は不要。

## 7. 各クレート／web の README（優先度づき）

「ルートは地図、各部屋は各部屋で説明」の標準に倣い、各ディレクトリに短い README を置く。ただし一度に全部は不要——**優先度を付ける**。

- **優先（今すぐ）**: `web/README.md` — 何で、どう動かし（ブラウザで開くだけ）、現状どこまで（棋譜再生のみ・エンジン未接続）、公開先（カスタムドメイン）と開発先（pages.dev）。web/ は新しく宙に浮いているので、最初に説明を与える。
- **次点**: `protocol/README.md` — プロトコルの純粋論理という位置づけと、守っている性質（commit-reveal の binding/hiding、ハッシュ相互検証、バージョン交渉等）。外部の同好の士が最も興味を持ちやすい部分。
- **後回しで可**: `engine/`・`cli/`・`tui/` の README。ルート README の構成説明で当面は足りる。気が向いたら一段詳しくする。

各クレート README は短くてよい（数行〜十数行）。ルート README と重複しすぎないよう、「そのクレート固有の事情」に絞る。
