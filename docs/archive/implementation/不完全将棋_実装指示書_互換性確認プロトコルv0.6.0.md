# 不完全将棋 実装指示書 — 互換性確認プロトコル（v0.6.0・接続時のバージョン交渉）

> 対象実行者: Claude Code
> 前提: v0.5.0 リリース済み。ワークスペース members = `engine`・`cli`・`tui`・`protocol`（TCP の殻は `tui/net.rs`）。Phase3 で、接続確立・commit-reveal・盤面ハッシュ相互検証・Ack・本人認証（共有秘密のハッシュ交換）・中断救済が実装済み。プロトコル論理はソケットから切り離した純粋な `protocol` クレートにあり、五性質が実通信なしのテストで守られている。
> 関連文書: `不完全将棋_バージョン互換性管理_方針.md`（本実装は方針の「第二段」に対応）、`不完全将棋_実装指示書_Phase3_TCP通信秘匿対戦.md`。
> 性格: 接続時に (ルール版, プロトコル版) を名乗り合い、互換性を確認してから対戦を始める。**これはプロトコルの変更であり、v0.5.0 とは対戦非互換**となる。ゆえに版を v0.6.0 へ上げる（方針文書 §3: プロトコルは本ゲームの核であり、その変更は対戦互換性を壊す＝メジャー級の刻み）。

---

## 0. 目的と範囲

- **作るもの**: 接続確立の最初の関門として、両者が (ルール版, プロトコル版) のタプルを名乗り合い、**完全一致**で互換性を判定する版交渉。一致すれば対戦へ進み、不一致なら**対戦開始前に**両者の版を示して拒否する。
- **位置づけ**: 方針文書の第二段。対戦が始まる前に弾く**事前検出**。既存の盤面ハッシュ相互検証（対戦中に局面のずれで気づく**事後検出**）と重層的に効く。
- **作らないもの（非ゴール）**: 範囲許容の互換性判定（完全一致のみ。緩和は第三段で再評価）。「最新版はこれ」「どちらに寄せよ」といった更新案内・価値判断（プロトコルは事実の照合に徹する）。配布元への問い合わせ。タプルの体系的な対応表管理（第三段）。

---

## 1. 版の自己申告（タプルの出どころ）

対戦互換性は (ルール版, プロトコル版) のタプルで決まる（方針文書 §3）。各版を、実行体がコンパイル時定数として自己申告できるようにする。

- **ルール版**: `engine` クレートが、実装しているルール仕様の版を定数として持つ（例: `engine::RULE_VERSION` のような定数。値は仕様版に対応、例 `"0.6"` または構造化した数値）。本リリースのルールは v0.5 から変更なしだが、ルール版という独立した識別子を**ここで初めて明示的に持たせる**（これまでは配布バイナリ版に暗黙に含まれていた）。
- **プロトコル版**: `protocol` クレートが、プロトコル自身の版を定数として持つ（例: `protocol::PROTOCOL_VERSION`）。版交渉の導入により、プロトコルは v0.5.0 時点のものから変わる。これを区別できる版にする。
- これらは**核（engine・protocol）が自己申告する値**であり、表示・交渉といった I/O は殻が担う（方針文書 §6 の分離原則）。配布バイナリ版（`workspace.package.version`、Step1 で導入）とは別の、機能的な版識別子である。

> ルール版とプロトコル版の値の表現（文字列か、整数か、major/minor の構造か）は実装者判断でよいが、**完全一致判定が決定的に行え、かつ将来の比較に耐える**形にする。当面は完全一致しか使わないが、第三段で範囲比較に拡張しうるため、順序比較可能な表現だと将来楽（必須ではない）。

---

## 2. 交渉の位置：握手の最初の関門（認証より前）

版交渉は、接続確立の握手の**最も早い段階**——接続が確立した直後、本人認証（共有秘密のハッシュ交換）よりも**前**——に置く。

- 理由: 版が食い違っていれば、その先の認証もゲームも一切意味をなさない。互換性は握手の最初の関門で確かめ、通った者だけが認証へ進む。
- 握手の順序（v0.6.0）:
  1. 接続確立（TCP）。
  2. **版交渉**（本実装）— (ルール版, プロトコル版) を名乗り合い、完全一致を確認。不一致なら拒否して終了。
  3. 本人認証（既存、共有秘密のハッシュ交換）。
  4. 先後（陣営）の取り決め（既存）。
  5. 対局開始（commit-reveal の各ターン）。

> 物理盤の比喩: 入室の最初に「同じ版の規則書を持っているか」を確かめ、持っていない者はそもそも対局室に入れない。立会人は事実（版が違う）を告げるが、どちらに揃えよとは指図しない。

---

## 3. 交渉の手順と判定

- 両者が、自分の (ルール版, プロトコル版) を相手へ送る。
- 各自、相手のタプルと自分のタプルを**完全一致**で比較する。両方の版が一致したときのみ「互換」と判定し、次（認証）へ進む。
- いずれか一方でも不一致なら「非互換」と判定し、**対戦を開始せずに**拒否する（§4）。

---

## 4. 不一致時の振る舞い（親切に、ただし踏み込まない）

- 不一致を検出したら、対戦を開始せず、**両者の版を示したメッセージ**を利用者に提示して終了する。
  - 例（文言は簡潔に）: 「対戦できません。あなた: ルール 0.6 / プロトコル X、相手: ルール 0.5 / プロトコル Y。版が異なります。」
  - どの版が不一致なのか（ルールか、プロトコルか、両方か）が分かるとなお親切。
- **言及しないこと**: どちらの版に寄せるべきか、どちらが新しいか、最新版が何か、更新方法。プロトコルは「一致しているか否か」という事実の照合に徹する。価値判断（どちらに揃えよ）は人間の領分であり、プロトコルの責務外（ユーザー方針）。
- 拒否は対称に行う（両者がそれぞれ相手の版を見て、それぞれ拒否メッセージを得る）。

---

## 5. 異常系：交渉に応答しない相手（過渡期の扱い）

版交渉を導入する**最初の版だからこそ**起きる過渡期の問題を扱う。

- v0.5.0 は版交渉を知らない。v0.6.0 が交渉メッセージを送っても、v0.5.0 はそれを理解できず、別の反応をするか沈黙する。つまり v0.6.0 から見ると、v0.5.0 は「版を名乗らない／交渉プロトコルを話さない相手」に見える。
- これを「互換性なし」へ倒す。具体的には:
  - 交渉メッセージへの応答が**タイムアウト**したら、非互換（または接続不成立）として扱い、利用者に通知して終了する（黙り込む相手を無限に待たない）。
  - 応答が**版タプルとして解釈できない**（フォーマット不正）なら、非互換として扱う。
- これは v0.6.0 と「交渉以前の版・別物」との間でだけ起きる過渡的事象。版交渉が両者に行き渡れば以後は起きない。だが最初の一回のために、タイムアウトと不正応答の異常系を実装・テストしておく。

> 実際、本実装の最初のテストケースは「v0.6.0 と v0.5.0 の非互換」になる。v0.5.0 が交渉に応じないことを、v0.6.0 がタイムアウト／不正応答として正しく「非互換」に倒せるかが、過渡期の正しさの確認になる。

---

## 6. 実装箇所と分離

- **`protocol` クレート（純粋論理）**: 版交渉の状態機械（タプルを受け取り、完全一致を判定し、互換／非互換／不正応答を返す純粋関数群）。ここに版交渉のロジックを置く。ソケット・タイムアウトの「待つ」具体は持たず、「応答が来た／来ない／不正」を入力として受け取り判定する形にする（タイムアウトの計時自体は殻側、判定は純粋側）。
- **`engine`・`protocol` の版定数**: §1 のとおり自己申告の値を持つ。
- **`net`（`tui/net.rs`）**: 交渉メッセージの送受信、タイムアウトの計時という I/O の殻。`protocol` の版交渉判定を駆動する。
- **`tui`**: 不一致時のメッセージ表示、接続フローの中での版交渉の位置づけ。
- **`engine` のルール挙動は無改変**（ルール版定数の追加のみ。ルール v0.5 の挙動は変えない）。

---

## 7. テスト（実通信を伴わない純粋論理）

Phase3 と同じ方針: `protocol` の版交渉を**ソケットなしで駆動できる純粋論理**として実装し、単体テストで守る。

1. **一致 → 互換**: 同一タプル同士で「互換」と判定され、次段（認証）へ進める状態になる。
2. **ルール版のみ不一致 → 非互換**: プロトコル版が同じでもルール版が違えば非互換。
3. **プロトコル版のみ不一致 → 非互換**: ルール版が同じでもプロトコル版が違えば非互換。
4. **両方不一致 → 非互換**。
5. **不一致メッセージの内容**: 非互換判定が、両者の版（とどの版が食い違うか）を提示できる情報を返す。更新案内・最新版・寄せ先を含まない。
6. **不正応答 → 非互換**: 版タプルとして解釈できない入力を、非互換（不正）として扱う。
7. **応答なし（タイムアウト相当）→ 非互換**: 「応答が来なかった」という入力に対し、非互換（不成立）として倒す（計時は殻、判定は純粋側で「タイムアウトが起きた」を入力として受ける形）。
8. **順序**: 版交渉が握手の最初に位置し、互換が確認されるまで認証・対局の段階へ進めないこと（状態機械として、版交渉未完了で先の段階を受け付けない）。

加えて、既存の Phase3 の五性質テスト・本人認証テスト等が**引き続き全て通る**こと（版交渉の追加が既存プロトコルを壊していないこと）。

---

## 8. 完了基準

1. `engine` がルール版定数、`protocol` がプロトコル版定数を自己申告し、配布バイナリ版（Step1）とは別の機能的版識別子として保持する。
2. 接続握手の最初（認証より前）に版交渉が入り、(ルール版, プロトコル版) の完全一致でのみ対戦へ進む。
3. 不一致時、対戦開始前に両者の版を示すメッセージで拒否する。更新案内・寄せ先・最新版には言及しない。
4. 交渉に応答しない／不正応答の相手（v0.5.0 を含む過渡期）を、タイムアウト・不正応答として「非互換」に倒せる。
5. §7 の版交渉テストが実通信なしで全て通り、Phase3 の既存テストも全て通る。
6. ルール（`engine`）の挙動は無改変。配布バイナリ版は v0.6.0 に上がる（`workspace.package.version`）。
7. リリース時、v0.6.0 が v0.5.0 と対戦非互換であることをリリースノートに明記する（番号の刻みと非互換の事実を記録）。
