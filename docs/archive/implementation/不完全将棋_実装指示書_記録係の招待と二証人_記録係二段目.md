# 不完全将棋 実装指示書 — 記録係の招待と二証人（記録係二段目：見る目）

> 対象実行者: Claude Code（Sonnet）
> 前提: 配布 v0.10.3 リリース済み（記録係一段目＝書庫。対局を内容ハッシュの身元で `ARCHIVES` KV へ綴じる。PROTOCOL_VERSION 3、RULE_VERSION 0.6）。関連する現物:
> - `server/src/room.ts`（`GameRoom` DO）: `spectate_meta`/`spectate_turn`/`spectate_result` を観測し記録へ反映（`_applyToRecord`）。**現状は全オンライン対局が自動綴じ**——`spectate_result{kind,outcome,text}`（**先手のみ送信**）で `text`（`build_archive` の正準本文）を `_archiveFinalized`（`sha256Hex`）で確定綴じ、`archived{id}` を放送。`_archiveCurrentIfNeeded`（再戦・全員離脱の断片綴じ）は**招待に関わらず無条件**。`archived:boolean` フラグ、envelope `{finalized, text|record, archived_at}`。routing は (役割,型)——commit/reveal は相手プレイヤーのみ、`spectate_*` は観戦者へ fan-out、観戦者送信は破棄。`webSocketClose` で全員離脱時 `_archiveCurrentIfNeeded`→`gameStarted` 削除。定数 `MAX_TURNS=500`・`MAX_ARCHIVE_TEXT_BYTES=512KB`。
> - `server/src/index.ts`: `GET /archive/:id`（書庫を部屋 DO を介さず直接）、`/watch/:token`（観戦、KV でトークン→roomKey 解決）、`/room/:key(/status|/archive|/spectate)`。
> - `web/online.js`: `spectate_*` を**先手のみ**送信（`mySide==='sente'`）。`_sendSpectateResult(kind, outcome, text)`。`connectOnline`・`hello` 版交渉。
> - `web/board.js`: `buildArchiveText()`（`build_archive` で正準アーカイブ本文を生成。両クライアントが同じ棋譜から作れば**バイト一致**＝同一 SHA-256）。終局判定 `evaluate_terminal`。
> - `protocol::negotiate`: 対戦互換は (ルール版, プロトコル版) 完全一致。
> 関連文書: `不完全将棋_版図_世界観と設計方針.md`（記録係＝権威なき観察者・忍者／審判なし／サードパーティとは侘びで競う）、`不完全将棋_プリンシパル設計_対局主体スキーマ.md`（consent は最小・席の予約）、`不完全将棋_実装指示書_対局の永続と身元_記録係一段目`（書庫・content-address・「綴じてから拭く」）、`不完全将棋_実装指示書_ライブ観戦とサーバ側アーカイブ_淀川第三歩`（秘匿境界・基盤中立）、`不完全将棋_バックログ_伏線と未決`。
> 性格: 記録係の道の**二段目＝見る目**。一段目で書庫（存在と身元）を据えた上に、(1) 散って暗黙だった記録係の役を、**明示的に招かれて対局に寄り添う一人の参加者**として立て（相互同意のオプトイン）、(2) その記録を**二証人の交差確認**で改竄検知可能にする（両者の証言＝正準本文の内容ハッシュが一致して初めて確定綴じ、不一致は**裁定せず surface**＝審判なし）。**サードパーティ層には何も足さない**——観戦口が既に対等な公開情報を開いているから。**ルールは不変**（RULE 0.6）、記録系メッセージを加える（PROTOCOL 3→4）。

## 0. 目的と範囲

- **作るもの**:
  1. **記録係（DO）の明示的招待**（§2）——相互同意のオプトイン。`record_invite`/`record_accept`/`record_decline`、確定 `record_confirmed`。
  2. **二証人の交差確認**（§3）——招かれた対局は、両プレイヤーが正準本文を証言として送り、DO が内容ハッシュ一致を確認して初めて確定綴じ（witnesses=2）。**不一致は裁定せず surface**（審判なし）、証拠は失わない。片方のみは witnesses=1 で綴じ。
  3. **招かれた対局のみ綴じる**（§4）——未招待は揮発（意図的）。「綴じてから拭く」は**招かれた対局のみ**に読み替え。
  4. **サードパーティ層はそのまま**（§6）——観戦口（`/watch`）が既に対等な公開情報を開く。専用 API を足さない。
  - PROTOCOL 3→4、配布 v0.10.3 → v0.11.0。
- **位置づけ**: 記録係を「部屋（使い回す殻）とは別の、招かれて寄り添う権威なき観察者」として立て、その審判なき記録を二証人で改竄検知可能にする。一段目（存在と身元）の上に、二段目（見る目）を積む。
- **作らないもの（非ゴール。§13 に申し送り）**:
  - **サードパーティ記録係の専用 API**——観戦口で足りる。内容ハッシュで第三者ログも公式 id と照合可能（特権は綴じ込みのみ）。
  - **記録係署名**（完全性の封蝋。予約のまま。二証人が第一の防波堤）。
  - **principal による帰属**——席は匿名 `-`。招待は「記録への軽い consent」を宿すが、識別・consent receipt は載せない（席だけ予約）。
  - **記録係の中立性・複数記録係**（新しい伏線。§13）。
  - **書庫の閲覧 UI**——`archived{id}` は最小限 surface するに留める。
  - **時間制**（持ち時間なし維持）。

## 1. 二層の記録係と、変わる不変条件

- **公式記録係＝DO**: 招かれ、両者の証言を突き合わせ、一致した記録を書庫へ綴じる、システムに組み込まれた特権的な役。両プレイヤーの間に座り確実に常在するのは DO だけ。**「部屋（中継役）」と「記録係（綴じる役）」は同じ DO 内でも概念として別の役**として扱う（殻／中身の分離を実装にも通す）。
- **サードパーティ記録係＝観戦口経由のログ**: 対局者・将来のマッチメーカー・第三者が `/watch/:token` で入り、流れる公開組手を自分で記録する。得られるのは公式と**寸分違わぬ公開情報**（commit-reveal が秘匿境界を守るため）。ゆえに独立に同じ棋譜を作れ、正準本文にすれば**同じ SHA-256 id** を生み、後日「公式書庫の id と同じ対局だ」と検証つきで照合できる。だが**書庫への綴じ込みという特権は持たない**。版図「侘びで競う」——特権にただ乗りせず、同じ公開情報から自分の器で。

**変わる不変条件**: 一段目の「綴じてから拭く」を、**「綴じてから拭く——ただし招かれた対局のみ」**へ読み替える。未招待の対局は揮発する（次局で拭かれる／終局で綴じない）。これは一段目で塞いだ**事故のデータ喪失（残したいのに消えた）はそのまま塞いだまま**、**選択による非記録（残さないと選んだから残らない）を許す**もの。事故と選択は別物。ただし「招き忘れて名局を失う」新しい保存し損ねが生じ得るので、招待は対局開始時に**明示的に prompt する**（§5）。

## 2. 記録係の招待（相互同意のオプトイン）

- **メッセージ**（対局チャネル。DO は観測しつつ相手へ中継する＝`spectate_*` と同じく型を解する）:
  - `record_invite`（提案プレイヤー → DO → 相手プレイヤー）: 「この対局を記録係に綴じてもらう」提案。
  - `record_accept` / `record_decline`（相手 → DO → 提案者）。
  - DO は `record_invite` を相手へ中継し pending を記録、`record_accept` を受けたら **`recording=true`** を storage へ置き、**`record_confirmed` を両プレイヤー＋観戦者へ放送**（観戦者にも「この対局は綴じられる」と透明に示す）。`record_decline` は `record_declined` を提案者へ。
- **タイミング**: 主に対局開始（handshake 完了）時に招く（§5 の prompt）。機構としては**終局前ならいつでも**受け付けてよい（綴じは終局の証言で行うため、招待時点の履歴蓄積を要しない）。
- **DO storage**: `recording:boolean`（既定 false）。`spectate_meta`（新局）で false にリセット。

## 3. 二証人の交差確認（終局）

- 招かれた対局（`recording=true`）では、終局時に**両プレイヤーが各自 `buildArchiveText()` で正準本文を作り、証言として送る**:
  - `record_testimony { text, kind, outcome }`（**両プレイヤーが送信**。`spectate_*` の先手のみとは異なる）。
- DO は両証言を収集し（`witness_a`/`witness_b`）、突き合わせる:
  - **一致**（`sha256Hex(text_a) === sha256Hex(text_b)`）→ その内容ハッシュを id に **`_archiveFinalized`（envelope に `witnesses: 2`）**、`archived{id}` を放送。二人の独立した目が合致した、確かな記録。
  - **不一致** → **裁定しない**（審判なし＝版図）。`record_disagreement { id_a, id_b }` を両プレイヤー（＋観戦者）へ surface し、**両証言を disputed envelope**（`{finalized:false, disputed:true, texts:[...], archived_at}`、暫定 id）で保存して**証拠を失わない**。どちらが正しいかは決めない。
  - **片方のみ**（相手が終局前に離脱等）→ その一証言を **`witnesses: 1`** で確定綴じ（データ喪失を避け、証人数を正直に記す）。
- **`spectate_result{kind,outcome}`（先手・ライブ）は分離して残す**——観戦者への終局表示のため。**綴じ（binding）は `spectate_result` から外し、`record_testimony` の二証人経路へ移す**。未招待の対局では `spectate_result` が終局のライブ表示だけを担い、綴じは起きない。
- **envelope 拡張**: `{ finalized, text, archived_at, witnesses: 2|1 }`（確定局）／`{ finalized:false, disputed:true, texts, archived_at }`（不一致）／`{ finalized:false, record, archived_at }`（放棄断片、従来）。

> **二証人が成り立つ根拠**: 両クライアントは commit-reveal で同一の着手列を共有し、版交渉で同一の (ルール版,プロトコル版) を持ち、同一デプロイゆえ app 版も一致する。ゆえに正直な両者の `buildArchiveText()` は**バイト一致→同一 id**。食い違いは、片方の改竄・バグ・分岐でのみ起き、それを surface するのが二証人の目的。

---

## 4. 招かれた対局のみ綴じる（未招待は揮発）

- **`_archiveFinalized`（終局・§3）・`_archiveCurrentIfNeeded`（放棄/再戦の断片）を `recording=true` で gate する**。
- **未招待**（`recording=false`）: 綴じない。ただし**ライブ観戦は従来どおり動く**（`spectate_turn` 蓄積・fan-out・遅れ参加の catch-up は招待に関わらず機能＝全局を観られる）。終局・次局・離脱で揮発する。
- **招かれた対局の断片**（招待済みで終局前に放棄）: `_archiveCurrentIfNeeded` が DO の `turns` から断片綴じ（従来どおり、ただし `recording=true` のときのみ）。二証人は完局でのみ成立するため、断片は単一源・best-effort（`finalized:false`）で正しい。
- 三契機の不変条件はこう読み替わる: **`recording=true` のとき、綴じてから拭く**。未招待は綴じずに拭いてよい（意図的な揮発）。

---

## 5. 招待の UX（招き忘れ対策・最小 surface）

- 対局開始（handshake 完了）時、両者へ**明示の招待 affordance**を出す（例:「この対局を記録係に綴じてもらいますか?」）。**オプトインだが prompt される**ので、招き忘れを抑止する。未 accept は**未記録（安全な既定）**。
- 一方が提案（`record_invite`）、他方が承諾（`record_accept`）で成立。`record_confirmed` を受けたら、両クライアントは「この対局は記録される」状態を持つ（終局時に証言を送る条件）。
- `archived{id}` を受けたら**最小限 surface**——「記録されました（リンク: …）」。これは**バックログの `archived{id}` UI 導線ゼロ状態を最小回収**する一歩（完全な書庫 UI は据え置き）。
- `record_disagreement` を受けたら「記録が食い違いました」を両者に見せる（**裁定しない**）。

---

## 6. サードパーティ層（何も足さない）

- サードパーティ記録係は、既存の**観戦口 `/watch/:token`** をそのまま使う。**専用 API・エンドポイント・メッセージを一切足さない。**
- 彼らが得るのは公式記録係と同じ公開情報（両者 reveal 後の確定組手）。独立に取ったログを正準本文にすれば同一 SHA-256 id を生み、公式書庫の id と照合できる（特権は綴じ込みのみ、真実は共有）。
- 「作らないことによって、サードパーティに場所を空ける」。版図「侘びで競う」の実装的表現。

---

## 7. consent と席（軽く、予約）

- `record_invite`＋`record_accept` は「**両者が、この対局が記録されることに同意した**」という**軽い consent** を宿す。今はそれを `recording=true` という真偽値として持つのみ。
- **席は匿名 `-` のまま**（principal 未実装）。principal＋consent receipt が来たら、招待/承諾が principal と consent 参照を運ぶ**受け皿**になる。ここでは継ぎ目だけ空ける（プリンシパル §5）。

---

## 8. プロトコル版と交渉

- **PROTOCOL_VERSION 3 → 4**。`record_invite`/`record_accept`/`record_decline`/`record_confirmed`/`record_declined`/`record_testimony`/`record_disagreement` を加え、綴じの契機を `spectate_result` から `record_testimony` の二証人へ移すため、ワイヤと綴じ挙動が変わる。
- **交渉**: 完全一致ゆえ**両プレイヤーとも v4** でなければ対戦不可（v3 は弾かれる）。web は一括デプロイ・CLI/TUI 凍結中なので同時に上がる前提でよい。
- **RULE 0.6・アーカイブ書式版 1 は据え置き**（ルールも棋譜書式も不変）。

---

## 9. クライアントの変更（online.js / board.js）

- **招待 UI**（§5）: 対局開始時の招待 affordance、`record_invite`/`record_accept`/`record_decline` の送受、ローカル `recording` 状態。`record_confirmed`/`record_declined` の反映。
- **二証人の証言**: 終局時、`recording` なら**両プレイヤー**が `buildArchiveText()` で正準本文を作り `record_testimony{text,kind,outcome}` を送る（現状 `spectate_result` の text を先手のみ送っていたのを、**両者の証言**へ移す）。`spectate_result{kind,outcome}` は先手がライブ終局表示のため引き続き送る（text は不要に）。
- **受信 handling**: `archived{id}`（記録された・リンクを最小 surface）、`record_disagreement{id_a,id_b}`（食い違いを surface・裁定しない）、`record_confirmed`/`record_declined`。

## 10. DO の変更（room.ts）

- `recording:boolean` state（storage、既定 false、`spectate_meta` で false リセット）。
- `webSocketMessage`（player 送信）に record 系を追加:
  - `record_invite` → 相手へ中継＋pending 記録。`record_accept` → `recording=true`＋`record_confirmed` を両者＋観戦者へ放送。`record_decline` → `record_declined` を提案者へ。
  - `record_testimony` → `recording` のとき収集。両証言が揃ったら `sha256Hex` 突き合わせ（§3）: 一致→`_archiveFinalized(text)`（`witnesses:2`）、不一致→`record_disagreement` surface＋disputed 保存、片方のみ→`witnesses:1` で綴じ。**綴じ後に `archived=true`**（KV 書き込み成功後にのみ立てる＝一段目の失敗安全 を踏襲）。
  - `record_*` は観戦者へは中継しない（対局チャネル。ただし `record_confirmed`/`archived`/`record_disagreement` の結果通知は観戦者にも放送＝透明性）。
- **`spectate_result` から綴じ（`_archiveFinalized`）を除去**し、`result` のライブ反映だけ残す。綴じは `record_testimony` 経路へ。
- **`_archiveFinalized`（§3 経由）・`_archiveCurrentIfNeeded` を `recording=true` で gate**（§4）。
- envelope に `witnesses`（確定局）・`disputed`/`texts`（不一致）を追加。
- 二証人の証言収集は**単一終局分の一時状態**でよい（`witness_a`/`witness_b` を storage or in-memory で保持し、綴じたらクリア）。DO は本文を解さない（ハッシュ突き合わせと綴じのみ）。

## 11. テストと受け入れ基準

- **招待**: `record_invite`→`record_accept` で `recording=true`、`record_confirmed` が両者＋観戦者へ。`record_decline` で未記録のまま。
- **二証人・一致**: 招待済み対局を終局→両者が同一 `record_testimony` を送る→`witnesses:2` の確定局が `sha256` id で書庫に、`archived{id}` 放送。`GET /archive/:id` で取れる。
- **二証人・不一致**: 片方の証言 text を改竄→`record_disagreement{id_a,id_b}` が surface され、**権威ある確定局は作られない**、disputed envelope に両証言が残る（証拠を失わない）。裁定は起きない。
- **片方離脱**: 招待済みで終局時に一証言のみ→`witnesses:1` で綴じられる。
- **未招待の揮発**: 招かない対局は、終局しても綴じられず、次局で拭かれる。ただし**ライブ観戦（`/watch`）は動く**。
- **未招待の放棄**: 招かない対局を放棄→断片は綴じ**られない**。**招待済み**の放棄→断片は綴じ**られる**。
- **サードパーティ不変**: `/watch` 観戦口は変わらず、専用 API は増えていない。第三者が観戦ログから作った正準本文の SHA-256 が、同一対局の公式 id と一致する。
- **版・回帰**: `protocol` が 4、v3 は弾かれる。commit-reveal 対局・再接続・`/archive/:id`・秘匿境界（commit/reveal が観戦者へ漏れない）が回帰しない。

## 12. 版の刻み

- **配布版 v0.10.3 → v0.11.0（マイナー）**。理由: 記録係が「暗黙・自動」から「招かれる・二証人で裏づく参加者」になる——記録の性質そのものが変わる出来事であり、綴じ挙動（オプトイン化）と対戦互換の境界（v4）も動く。地続きの前進を超える。
- **PROTOCOL_VERSION 3 → 4**。**RULE 0.6・アーカイブ書式版 1 は据え置き**。

---

## 13. 非ゴールと次段への申し送り

- **記録係署名**: content-address した id・二証人一致に、記録係の署名（完全性の封蝋）。二証人が第一の防波堤、署名はその上。
- **principal による帰属**: 席が埋まれば、招待/承諾に principal＋consent receipt が載り、書庫の記録が「誰が指し、誰が記録に同意したか」を宿す。§7 の継ぎ目に嵌める。
- **記録係の中立性・複数記録係（新しい伏線）**: 公式記録係（DO）が嘘を綴じないと誰が保証するか。二証人が第一の防波堤だが、究極的には複数記録係の突き合わせ・記録係署名・検証可能な記録が要る。版図「正体を問わない世界」の遠い地平。**バックログへ一行追加**。
- **書庫の閲覧 UI**: `archived{id}` の完全な導線（過去局一覧・リンク共有）。本書は最小 surface のみ。
- **時間制・人 vs AI の速度差**: 別層。

## 14. 不変の原則（本実装が守るもの）

1. **招かれて、寄り添い、記す**: 記録係は特定の一局に相互同意で招かれ、干渉せず立ち会い、後で棚に綴じる。招かれない対局は揮発する（選択による非記録）。
2. **審判なし**: 記録係は勝敗を決めない（それは peer-to-peer の commit-reveal が決める）。両者の証言が一致した記録だけを綴じ、食い違えば**裁定せず surface する**。証拠は失わない。
3. **綴じてから拭く——招かれた対局のみ**: 一段目の不変条件を、招待でゲートする。事故の喪失は塞いだまま、意図的な揮発を許す。
4. **公開情報は対等、綴じ込みだけが特権**: サードパーティは観戦口で同じ公開情報を得て、独立に同じ内容ハッシュを作れる。専用層を足さず、侘びで競う場所を空ける。
5. **DO は本文を解さない**: 二証人はハッシュの突き合わせ、綴じは不透明ブロブの content-address。書式の正本は engine。
6. **軽い consent、席は予約**: 招待は記録への軽い同意を宿すのみ。principal と consent receipt の継ぎ目だけ空ける。
7. **加算的**: commit-reveal 対局チャネル・ルール・棋譜書式は不変。記録系メッセージと二証人の綴じ経路を足し、綴じを招待でゲートするのみ。プロトコル版のみ 4 へ。

*記録係の二段目——招待と二証人。散って暗黙だった記録係の役を、相互同意で招かれ対局に寄り添う権威なき観察者として立て、その記録を二証人の交差確認（両者の正準本文の内容ハッシュ一致で確定綴じ、不一致は裁定せず surface）で改竄検知可能にする。招かれた対局のみ綴じ（未招待は揮発）、サードパーティ層は観戦口のまま何も足さない（侘びで競う場所を空ける）。席は匿名・consent は軽く予約。ルール・棋譜書式は不変、プロトコル版を 4、配布版を v0.11.0 へ。見る目が、書庫に宿る。*
