# 不完全将棋 実装指示書 — Phase2（ratatui による TUI 検証卓）

> 対象実行者: Claude Code
> 前提: Phase1（Phase 1）が実装済み。ワークスペースは `engine/`（純粋ライブラリ）と `cli/`（検証用バイナリ）からなり、`cargo test` が全て通る（v0.4 ルール、戦国無双特則を含む）。
> 関連文書: `不完全将棋_ルール仕様_v0.4.md`（ルールの正本）、`不完全将棋_実装指示書_Phase1.md`（Phase 1 の全体設計）。
> 性格: 本書は**新しいゲーム体験ではなく、既存の検証モードの操作性改善**である。エンジンの核には一切手を触れない。TUI は `engine` を叩く新しい殻にすぎない。

---

## 0. この段階の位置づけ（重要な範囲限定）

第二弾は「マウスでサクサク操作できる検証卓」である。当初ロードマップのPhase2（Wasm・ブラウザ・秘匿通信）の**手前に置く、ターミナル内の検証卓**であり、以下を明確に区別する。

- **本段階で作る**: ratatui ベースの全画面 TUI。一人が先手・後手の両方の着手を、カーソルとマウスで組み立てて同時解決する検証モード（秘匿なし）。合法手のインタラクティブな提示を目玉機能とする。
- **本段階で作らない（非ゴール）**: Wasm・ブラウザ UI・ネットワーク・コミットメント方式・秘匿同時着手・二プロセス対戦・CPU 対戦。これらはPhase3以降。本段階は単一プロセス・単一画面・一人操作に限る。

> 設計哲学「共通の核と交換可能な殻」をそのまま適用する。エンジンは I/O・乱数・ネットワークを持たない純粋なまま。TUI は `engine` の公開 API（`legal_actions`・`resolve`・`check_status`・直列化）を叩くだけの殻である。**エンジンクレートを変更してはならない**（もし TUI 実装の都合で公開 API に不足が見つかったら、勝手に内部ロジックを変えず、必要な追加を報告すること。理想は既存 API のみで完結する）。

---

## 1. 二つの殻の棲み分け（明示）

「残しつつ追加」が本段階の基本方針である。既存の逐次 CLI は廃止・改変せず、TUI を第三のクレートとして**追加**する。

- **逐次 CLI（既存 `cli/`、無傷で温存）** — 標準入出力でテキストの着手列を受け、局面・結果をテキストで返す**機械可読の口**。スクリプト・パイプ・自動テストから決定的に駆動できる。将来の多言語実装との差分テストの土台。**この性格を損なう変更を加えないこと。**
- **TUI 検証卓（新規 `tui/`）** — 人間が対話的にカーソル・マウスで指して目視検証する**快適な卓**。全画面・常駐表示・合法手ハイライト。

両者は同じ `engine` クレートを叩く別の殻として並立する。どちらかの都合でエンジンや相手の殻をいじらない。

---

## 2. 技術スタックとクレート構成

- **クレート追加**: ワークスペースに `tui/`（バイナリクレート）を追加する。`engine` に依存し、`cli` には依存しない。
- **UI フレームワーク**: **ratatui** を採用する（バックエンドは crossterm）。盤面・駒台・ステータス・テキストパネルを領域分割して常駐表示し、状態が変わるたびに宣言的に再描画する、というモデルが本卓の構造に合う。マウス／キーのイベントは crossterm 由来のものを受け取る。
- **依存方針**: TUI クレートは ratatui・crossterm に依存してよい。`engine` の依存方針（依存薄・I/O なし）は不変。
- **既存資産の流用**: 既存逐次 CLI が持つ機能（合法手列挙、undo、save/load、resign、SFEN 表示）は、同じエンジン API を叩けば TUI でも実現できる。CLI のロジックを参考にしてよいが、共有が自然なヘルパー（USI 整形、盤面の ASCII 化など）があれば `engine` ではなく小さな共有箇所へ切り出すか、TUI 側に持つ。**エンジンを肥大化させない。**

推奨構成:

```
fukanzen-shogi/
├─ Cargo.toml          # [workspace] に tui を追加
├─ engine/             # 不変（触らない）
├─ cli/                # 不変（機械可読の口として温存）
└─ tui/                # 新規: ratatui 検証卓
   ├─ Cargo.toml       # depends on engine, ratatui, crossterm
   └─ src/
      ├─ main.rs       # 端末初期化・イベントループ・終了処理
      ├─ app.rs        # アプリ状態（現局面・棋譜・選択状態・入力中の着手・直前の解決結果）
      ├─ ui.rs         # ratatui による描画（盤面・駒台・ステータス・テキストパネル）
      └─ input.rs      # キー／マウスイベント → 操作への変換
```

実行は `cargo run --bin <tui のバイナリ名>`（名前は適宜。例: `fukanzen-shogi-tui`）。

## 3. 画面レイアウト

全画面を ratatui の領域分割で構成する。最小構成は次の領域:

- **盤面**: 9×9 の盤。筋（1–9、右から左）と段（一〜九、上から下）の座標を縁に表示。先手・後手の駒を大文字小文字や色で区別。各マスは ASCII/Unicode の駒文字で描く（盤面は静的描画でよい。唯一の動的要素は後述の合法手ハイライト）。
- **駒台（左右または上下）**: 先手・後手それぞれの持ち駒と枚数。
- **ステータス行**: 手数、現在どちらの着手を入力中か（先手入力中／後手入力中／解決待ち）、直前のターンの解決結果の要約。
- **情報パネル（テキスト）**: 次の二つをテキストで表示する。
  - **入力中の着手**: 先手・後手それぞれの「仮確定した着手」を USI＋簡単な説明で表示（例: `先手: 7g7f (歩)`、`後手: 未入力`）。**盤面上に仮の駒を動かして見せる必要はない**——テキスト表示で足りる（ユーザー方針）。
  - **直前の解決結果**: 直前ターンで何が起きたかをテキストで列挙（後述 §6）。

> レイアウトの凝りすぎは不要。検証に必要な情報が一望でき、合法手ハイライトが見やすいことを優先する。色やアニメーションの作り込みは最小限。

## 4. 操作のモデル（同時着手の三拍子）

一ターンは「先手の着手を組む → 後手の着手を組む → 同時解決」の三拍子で進む。一人が両方を順に入力する（検証モード、秘匿なし。先手の仮着手は後手入力中も画面に見えていてよい＝ユーザー方針 (a)）。

### 4.1 着手の組み立て（移動）

1. 盤上の自陣の駒を、カーソル移動（方向キー）＋決定キー、またはマウスクリックで**選択**する。
2. 選択した瞬間、その駒の**合法な移動先を盤上にハイライト**する（§5。本卓の目玉）。
3. ハイライトされた移動先を選ぶと、その着手が**仮確定**する。
   - 移動先が成り可能な場合は、**成る／成らず**を問う（キーの二択、またはポップアップ）。成れない駒・成れない位置では問わない。行き所のない駒になる不成は、そもそも合法手に現れない（エンジンが弾く）。
4. 仮確定した着手は情報パネルにテキスト表示する。

### 4.2 着手の組み立て（打ち）

1. 駒台の持ち駒を選択する（キーまたはマウス）。
2. 選択した瞬間、その駒を**打てる合法なマスを盤上にハイライト**する（二歩・行き所のない駒・打ち歩詰めなどはエンジンが除くので、合法な着地先のみが光る）。
3. 着地先を選ぶと仮確定。打ちは成り選択なし。

### 4.3 先手→後手→解決

- まず先手の着手を上記で仮確定し、続いて後手の着手を同様に仮確定する。
- 両者が仮確定したら、明示的な**解決キー**（例: Enter、または専用キー）で `resolve(position, sente_action, gote_action)` を呼び、同時解決する。
- 解決前なら、仮確定の**取り消し**（選択やり直し）ができる。
- 解決後、新しい局面を描画し、直前の解決結果をテキスト表示し、次のターンへ。

### 4.4 補助コマンド（既存 CLI と等価の機能を TUI 操作に）

既存逐次 CLI のコマンドに相当する操作を、キーバインドとして用意する:

- **合法手一覧**: 選択中の駒のハイライトに加え、必要なら先手／後手の全合法手をパネルに列挙するキー（`:moves` 相当）。
- **undo**: 直前の Ply を取り消す（棋譜末尾を削って再 replay。`:undo` 相当）。
- **save / load**: 棋譜の保存・読み込み（既存形式と互換。`:save`/`:load` 相当）。
- **resign**: 先手／後手の投了（`:resign` 相当）。
- **SFEN 表示**: 現局面の SFEN をパネルに表示（`:sfen` 相当）。
- **quit**: 端末状態を必ず復元してから終了する（raw mode 解除・画面復帰。パニック時も復元すること）。

キー割り当ては実装者判断でよいが、画面内にヘルプ（キー一覧）を出せること。

## 5. 合法手のインタラクティブ提示（本段階の目玉）

これは TUI 版の中核機能であり、最も丁寧に作る。逐次 CLI の `:moves` が、操作に溶け込んで常時使える形へ進化したもの。

- 駒（盤上の駒、または駒台の持ち駒）を選択した瞬間、`legal_actions(position, side)` の結果から**その駒に関する合法手だけ**を抽出し、移動先（または打ち先）のマスを盤上で強調表示する。
- これにより、Phase1のテストで目視確認したかった事柄が、対話的に確認できる:
  - 玉を選ぶと、相手の利きに侵入する手は**ハイライトされない**（玉の侵入禁止）。
  - 自玉が王手されているとき、王手を解除しない手は**現れない**（合法手が王手回避手に絞られる）。
  - 戦国無双が支えのある駒には働かないこと（玉でその駒を取りに行く手が合法手に**現れない**＝ハイライトされない）。逆に後ろ盾のない鉄砲玉へ向かう玉の手はハイライトされる。
  - ピンされた駒、二歩になる歩打ち、行き所のない駒打ちなどが除かれていること。
- ハイライトは「合法な移動先」を示すだけでよく、解決結果の予測（取得が起きるか等）まで盤上に描く必要はない。検証者は合法手集合そのものを見たいのであって、結果予測は解決して確かめる。

> この機能があることで、TUI は単なる入力の楽な盤ではなく、「合法手生成の正しさをインタラクティブに検証する道具」になる。Phase1で最も目視確認したかった部分を、最も触りやすくする。

## 6. 解決結果のテキスト表示

解決後、直前ターンで何が起きたかを**テキストで**情報パネルに示す（盤面上の凝った視覚表現は不要＝ユーザー方針）。`resolve` の結果から、少なくとも次を文言で列挙する:

- 各陣営の着手（USI＋駒種）。
- 発生した事象: 取得（どの駒がどの駒を取ったか）、逃げた駒、空きマスの相討ち、スワップの相討ち、経路すり抜け、**戦国無双の発動**（玉が相手駒を斬り返した）、玉の死。
- 取得・相討ちによる持ち駒の変化。
- 終了した場合はその種別（確定的詰み・玉の死・投了・引き分け）。

> 文言は簡潔でよい。例:「先手 7g7f、後手 8h2b+：後手の角が成って 2b の銀を取得（先手の銀→後手駒台）」「先手 玉 5i5h、後手 P*5h：玉が打ち歩を斬り返す（戦国無双）」。盤面の差分は新旧の盤を描き直すことで十分伝わる。

直前ターンに加えて、ステータス行には現在の入力フェーズ（先手入力中／後手入力中／解決待ち）を常に示すこと。

## 7. 状態管理（TUI 側）

アプリ状態は概ね次を保持する（`engine` の型を流用し、TUI 固有の選択状態だけを足す）:

- 現局面（`engine` の Position）と棋譜（Kifu）。これらは `engine` の型をそのまま使う。
- 入力フェーズ（先手入力中 / 後手入力中 / 解決待ち）。
- 先手・後手それぞれの仮確定した着手（未入力 / 確定済みの Action）。
- 現在の選択状態（盤上カーソル位置、選択中の駒、その駒の合法手ハイライト集合）。
- 直前の解決結果（テキスト表示用）。

盤面の描画は「現局面 ＋（選択中の駒があれば）その合法手ハイライト」という純粋な関数として書ける。仮着手や解決途中状態を盤面に重ねないので、描画はシンプルに保てる。

## 8. 完了基準（Definition of Done）

1. ワークスペースに `tui/` クレートが追加され、`cargo run` で TUI 検証卓が起動する。`engine` と `cli` は**無改変**（少なくともロジック・公開 API は不変。`cli` は機械可読の口として従来どおり動く）。
2. ratatui で、盤面・両駒台・手数・入力中着手・直前の解決結果が常駐表示される。
3. カーソルとマウスの**両方**で、移動・打ち・成りを含む着手を組み立てられる。
4. 駒（盤上・駒台）を選ぶと、その駒の合法手が盤上にハイライトされる（§5）。玉の侵入禁止・王手回避絞り込み・戦国無双の非発動などが、ハイライトの有無として目視確認できる。
5. 先手・後手の着手を組んで同時解決でき、解決結果（取得・逃げ・相討ち・スワップ・戦国無双・玉の死）がテキストで分かる。
6. 確定的詰み・玉の死・投了で一局を最後まで指せる。undo・save/load・SFEN 表示・resign の補助操作が使える。
7. 端末状態が常に正しく復元される（正常終了・quit・パニックいずれでも raw mode 解除と画面復帰）。
8. `cargo test`（既存の 23 本＋ v0.4 分）が引き続き全て通る（TUI 追加でエンジン挙動が変わっていないこと）。

## 9. 進め方の指針

1. `tui/` クレートの骨格（端末初期化・イベントループ・終了時の状態復元）をまず安全に作る。パニック時も端末を復元するガードを最初に入れる。
2. 静的描画から: 現局面（engine の初期局面）を盤面・駒台・ステータスとして描く。
3. カーソル移動と駒選択 → 合法手ハイライト（§5）を実装。**ここが目玉なので丁寧に。**
4. 着手の組み立て（移動・打ち・成り）→ 仮確定 → 先手後手の両方 → 解決キーで `resolve`。
5. 解決結果のテキスト表示（§6）と次ターンへの遷移。
6. マウス対応（クリックでマス・駒台選択）を追加。
7. 補助操作（undo・save/load・resign・SFEN・ヘルプ）。
8. 端末復元・エラー処理の総点検。

横断的注意:

- **エンジンを変更しない。** 公開 API のみで実装する。不足が出たら報告（勝手に内部を変えない）。
- 既存 `cli` の機械可読性を損なわない。
- ルール変更による手戻りは、エンジンクレート側で吸収される設計なので、TUI 側は basically 無傷で済む。UI に多少凝ってもルール調整で壊れない（核と殻の分離の効用）。
