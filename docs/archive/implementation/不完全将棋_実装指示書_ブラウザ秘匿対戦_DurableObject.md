# 不完全将棋 実装指示書 — ブラウザ秘匿対戦（WebSocket + Cloudflare Durable Object 中継・ルームキー方式）

> 対象実行者: Claude Code（Sonnet）。**本書は段階を追って進める構成。各ステップを完了し動作を確かめてから次へ進むこと。一度に全体を実装しない。**
> 前提: `web/` は Wasm 化した `engine` で操作可能（ホットシート・水墨様式・合法手の墨点提示）。`protocol` クレートに、ソケットから切り離した純粋な秘匿対戦論理が実装済み（commit-reveal、盤面ハッシュ相互検証、Ack、本人認証＝共有秘密のハッシュ交換、中断救済、バージョン交渉 v0.6）。Phase3 でこれらは TCP 上で動作し、五性質＋本人認証＋バージョン交渉が実通信なしのテストで守られている。ルールは v0.5。
> 関連文書: `不完全将棋_実装指示書_Phase3_TCP通信秘匿対戦.md`、`不完全将棋_実装指示書_互換性確認プロトコルv0.6.0.md`、`不完全将棋_実装指示書_Web盤Wasm組み込み.md`、`不完全将棋_実装指示書_Web盤操作可能化.md`。
> 性格: Phase3 の秘匿対戦を、ブラウザ二つの間で成立させる。三つの流れ（テスト済み protocol 論理・Wasm で動く Web 盤・トランスポート）が合流する。**WebRTC・観戦は今回やらない**（後の殻）。

---

## 0. 全体方針（最初に読む）

### 0.1 何を作るか
二人が、それぞれのブラウザで、**ルームキー（合言葉）を共有して同じ部屋に入り**、互いの着手を知らないまま秘匿同時対戦できるようにする。トランスポートは **WebSocket**、二人を束ねる中継は **Cloudflare Durable Object**。

### 0.2 設計の背骨（崩してはならない原則）
- **審判はいない。** Durable Object は二人の WebSocket を同じ部屋に集め、メッセージを右から左へ転送する**郵便配達兼待合所**にすぎない。**ゲームの判定（合法性・解決・勝敗・ハッシュ照合）には一切関与しない。** どちらの手が正しいか・勝ったかは、両者の手元の Wasm protocol が commit-reveal とハッシュ相互検証で互いに確かめ合う。中央に器は置くが、審きは両岸にある。
- **論理は純粋、I/O は殻。** `protocol` の論理は Wasm 化してブラウザの手元で動かす（Rust のまま。JS で書き直さない）。WebSocket の送受信という I/O は JS の殻が担う。`protocol` は「届いたメッセージを食わせると状態が進み、次に送るべきメッセージや判定結果が返る」純粋な状態機械に徹する。
- **核は無改変。** `engine`・`protocol` の Rust 本体は変えない。Wasm 化のラッパと、JS/Durable Object の殻を**足す**だけ。
- **Phase3 の論理をそのまま使う。** 握手（バージョン交渉 → 本人認証 → 先後決め）、各ターンの commit-reveal、ハッシュ相互検証、Ack、中断救済は、Phase3 で設計・テスト済みのものを WebSocket の上で動かす。プロトコル論理を作り直さない。

### 0.3 今回やらないこと（非ゴール・後の殻）
- WebRTC ピア直結（今回は Durable Object 中継のみ。WebRTC は次の殻）。
- 観戦・観戦配信（対戦者二人に絞る。観戦の秘匿境界の設計は後）。
- ロビー・マッチメイキング・アカウント（ルームキーで二人が合流するだけ）。
- CPU 対戦。

### 0.4 進め方の鉄則（Sonnet 向け）
- **各ステップを完了し、動作確認してから次へ進む。** 前のステップが緑になっていない状態で次を積まない。
- **一度に全部を書かない。** 下のステップ順に、小さく作って小さく確かめる。
- 詰まったら、その時点で止めて報告する（勝手に大きく作り込まない）。
- 各ステップの「確認」を必ず実施する。

## 1. クレート／ディレクトリ構成（全体像）

足すもの:
- `protocol-wasm/`（新規・任意名）: `protocol`（と必要なら `engine`）を wasm-bindgen でラップする cdylib クレート。秘匿対戦の状態機械を、SFEN/USI/メッセージ文字列の境界でブラウザへ出す。`protocol` 本体は無改変。
  - 既存の `engine-wasm` に相乗りさせてもよい（実装者判断）。ただし責務が分かるように。
- `web/` 内: WebSocket 接続・ルーム入室・秘匿対戦 UI（JS）。
- `server/`（新規・任意名）: Cloudflare Worker + Durable Object の中継（TypeScript/JavaScript）。二人を部屋に束ねてメッセージを転送するだけ。ゲーム論理なし。

> `protocol` の純粋論理は Rust のまま Wasm で動く（核と殻の一貫）。Durable Object は論理を持たない中継に徹する。

## 2. ステップ構成（この順に、各ステップで確認しながら進む）

各ステップは独立して確認可能なまとまり。**前のステップの確認が通ってから次へ。**

---

### ステップ A — Durable Object 中継の骨格（ゲーム論理なしの郵便配達）

**目的**: WebSocket でつながった二人を、ルームキーで同じ部屋に束ね、一方のメッセージをもう一方へ転送するだけの中継を立てる。ゲームのことは何も知らない。

**作業**:
1. Cloudflare Worker + Durable Object のプロジェクトを `server/` に作る（`wrangler` 構成）。
2. Durable Object を「部屋」とする。**ルームキー**ごとに一つの Durable Object インスタンスが対応する（ルームキーを Durable Object の名前に使う）。
3. クライアントは `wss://.../room/<ROOMKEY>` のように WebSocket 接続する。Durable Object は、同じルームキーに来た接続を**最大2つ**保持する。
4. 一方の接続から受け取ったメッセージを、**もう一方の接続へそのまま転送**する（中身を解釈しない＝不透明な中継）。
5. 3人目の接続は拒否する（満室）。一方が切断したら、その旨をもう一方へ通知できるようにしておく（中断救済の土台。詳細はステップ F）。

**確認**:
- 二つのブラウザタブ（または `wscat` 等）で同じルームキーに接続し、片方で送った文字列がもう片方に届く。違うルームキーなら届かない。3つ目の接続が拒否される。
- **この時点でゲームは一切関与しない。ただのチャット転送ができればよい。**

> 注意: Durable Object はメッセージ内容を**解釈しない**。commit も reveal も、ただのバイト列／文字列として右から左へ流す。これが「審判なし」を構造で保証する。

---

### ステップ B — protocol の Wasm 化（手元で動く秘匿対戦の頭脳）

**目的**: テスト済みの `protocol` 純粋論理を、ブラウザの手元で動かせるようにする。WebSocket はまだ繋がない。Wasm として論理が呼べることだけ確認。

**作業**:
1. `protocol-wasm/` を作り、`protocol` を wasm-bindgen でラップする。
2. 境界は**文字列メッセージ**で薄く保つ。状態機械を駆動する最小の口を出す。例（名前は実装に合わせる）:
   - 対局セッションを生成する（自分の陣営、ルール版・プロトコル版、本人認証用の共有秘密などを与えて初期化）。
   - 「相手から届いたメッセージ（文字列）を食わせる」と、状態が進み、**次に送るべきメッセージ（あれば）**と**この時点の状態／判定結果**（例: 互換 OK、認証 OK、相手の commit 受領、reveal 検証 OK/NG、解決後 SFEN、Ack 完了、終局、エラー等）を返す。
   - 自分の着手（USI）を投入して commit を作る、reveal を出す、等の、こちら起点のアクション。
3. これらは Phase3 の状態機械をそのまま呼ぶ。**新しいプロトコル論理を書かない。** Wasm 境界の変換（文字列⇔内部メッセージ型）だけを書く。

**確認**:
- Wasm をビルドし、ブラウザの JS から protocol セッションを初期化して、ローカルだけで（WebSocket なしで）状態機械を一手ぶん駆動できる（例: 自分の commit メッセージ文字列が生成される）。
- 既存の `protocol` の Rust テスト（五性質＋認証＋バージョン交渉）が**引き続き全て通る**（Wasm 化がロジックを変えていない）。

---

### ステップ C — JS の WebSocket 殻と protocol の接続（ローカル疎通）

**目的**: ステップ A の中継と、ステップ B の Wasm protocol を繋ぐ。JS が「WebSocket で届いたメッセージを Wasm に食わせ、Wasm が返したメッセージを WebSocket で送る」というループを作る。まだ握手だけ。

**作業**:
1. JS に、WebSocket 受信 → Wasm protocol に投入 → Wasm が返した送信メッセージを WebSocket 送信、というディスパッチループを書く。
2. ルームキーを入力して入室 → WebSocket 接続 → 握手の最初（**バージョン交渉**: (ルール版, プロトコル版) を名乗り合い完全一致を確認）を、二つのブラウザ間で通す。
3. 続いて**本人認証**（共有秘密のハッシュ交換。Phase3 の論理）まで通す。

**確認**:
- 二つのブラウザで同じルームキーに入室すると、バージョン交渉が成立し（一致すれば次へ、不一致なら両者の版を示して拒否＝v0.6 の挙動）、本人認証まで完了する。
- 故意に違うバージョン／違う共有秘密にすると、正しく拒否される。
- **Durable Object はこれらのメッセージを解釈せず転送しているだけ**であることを意識（中継は不透明）。

---

### ステップ D — 一手の同時着手（commit → reveal → ハッシュ照合 → 解決 → Ack）

**目的**: 握手の済んだ二人が、**一手だけ**、秘匿同時着手を完走する。ここが秘匿対戦の心臓。

**作業**:
1. 先後が決まった状態（握手で確定）で、各自が自分の着手を選ぶ。**この段階の着手選択 UI は、操作可能化で作ったホットシートの「自分側だけ」を流用**（合法手の墨点提示も使える。`legal_actions` は engine-wasm 経由）。
2. 各自 commit を作って送る（相手の手は見えない）。**両者の commit が揃うまで reveal しない**（順序の正しさ）。
3. 両者の commit が揃ったら reveal（着手・ノンス・着手開始時の盤面ハッシュ）を送る。
4. 受領した reveal を検証（commit との一致＝拘束性、盤面ハッシュの相互一致）。不一致なら中断（ステップ F）。
5. 両者の着手が確定したら、各自手元で解決（engine-wasm の解決関数で次局面 SFEN を得る。両者同じ入力なので同じ結果）。
6. Ack を交換し、両者の Ack が揃ったら次局面を確定。盤面を更新表示（水墨様式・発生事象テキスト＝取得/相討ち/スワップ/戦国無双/両玉スワップ引き分け/玉の死等）。

**確認**:
- 二つのブラウザで、互いの手を伏せたまま一手を指し、同時開示され、同じ次局面が両画面に出る。
- 自分の手を入力中、相手の手は reveal まで**見えない**（秘匿が成立）。相手が考え中である気配だけ分かる。
- 故意に commit と違う手を reveal させる細工をしたら、相手側で検証が失敗し中断する（拘束性が効く）。

---

### ステップ E — 対局の継続（終局まで）

**目的**: ステップ D の一手を繰り返し、一局を最後まで指せるようにする。

**作業**:
1. ステップ D のターンサイクルをループさせ、ターンを重ねられるようにする。手番はない（同時着手）。
2. 終局条件（確定的詰み・玉の死・両玉スワップ引き分け・投了）を検出して対局を終える。投了の送受も用意。
3. 対局中、自分側の着手選択は墨点提示つきホットシート、相手側は秘匿、という非対称な画面を保つ。
4. 棋譜（初期局面＋着手列）を各自ローカルに積む（中断救済とハッシュ照合の土台。操作可能化で作った棋譜構造を流用）。

**確認**:
- 二人で一局を終局まで指せる。戦国無双・両玉スワップ・玉の死などが起きても、両画面で同じ結果になる（盤面ハッシュ相互検証が毎ターン一致）。
- 投了で正しく終わる。

---

### ステップ F — 切断と中断救済

**目的**: 通信が切れても、再接続してハッシュ照合＋本人認証で正しい再開点から続けられる。

**作業**:
1. 一方が切断したら、もう一方に通知し、再接続待ち状態にする（ステップ A の切断通知を使う）。
2. 再接続時、**本人認証**（共有秘密）で正当性を確認し、**ハッシュ照合**（相手の申告ハッシュを自分の棋譜から再現される各局面の正準直列化ハッシュと照合）で再開点を特定する（Phase3 の中断救済論理）。一致が無ければリジェクト。
3. Ack による進行同期で、両者が必ず同じ局面で止まる設計を活かし、再開点のズレを吸収する。

**確認**:
- 対局中に一方のタブを閉じる／リロードする等で切断し、同じルームキー・同じ共有秘密で再接続すると、ハッシュ照合で正しい局面から再開できる。
- 誤った共有秘密や、棋譜が食い違う相手の再接続はリジェクトされる。

---

## 3. 横断的な注意

- **審判なしを構造で守る**: Durable Object は全ステップを通じてメッセージを**解釈しない不透明な中継**に保つ。判定は両者の手元の Wasm protocol のみ。Durable Object にゲーム状態やルールを持たせない。
- **核は無改変**: `engine`・`protocol` の Rust 本体を変えない。Wasm ラッパと JS／Durable Object の殻を足すだけ。不足が出たら報告。
- **秘匿の正しさは protocol が担保**: binding/hiding/順序/ハッシュ/中断救済は Phase3 のテスト済み論理が守る。Wasm 化・WebSocket 化はその論理を運ぶだけで、秘匿の正しさを新たに JS 側で再実装しない。
- **Cloudflare 配信**: 静的フロント（`web/`）はカスタムドメイン `https://fukanzen-shogi.tokuhira.net`、開発は `pages.dev`。Durable Object/Worker は別途デプロイ（`wrangler`）。WebSocket の接続先（wss URL）を設定で切り替えられるようにする。
- **ローカル確認**: Worker/Durable Object はローカルエミュレート（`wrangler dev`）で二タブ疎通を確認してからデプロイ。

---

## 4. 完了基準

1. ステップ A〜F が順に完成し、各ステップの確認が通っている。
2. 二人が同じルームキーで入室し、バージョン交渉・本人認証を経て、秘匿同時着手で一局を終局まで対戦できる。自分の手は選べ、相手の手は reveal まで秘匿される。
3. Durable Object はメッセージを解釈しない中継に徹し、判定は両者の手元の Wasm protocol が行う（審判なし）。
4. 盤面ハッシュ相互検証が毎ターン一致し、v0.5 固有局面（戦国無双・両玉スワップ・玉の死）でも両画面が一致する。
5. 切断後、本人認証＋ハッシュ照合で正しい再開点から続行できる。
6. `engine`・`protocol` は無改変。既存の Rust テストが全て通る。既存のホットシート・再生・水墨様式は壊れていない。
7. WebRTC・観戦・ロビーは含まない（範囲を守る）。

## 5. 次の殻（本段階では作らない・見通し）

- **WebRTC ピア直結**: シグナリングを WebSocket（Durable Object）で行い、繋がったら手のやり取りをピア間直接へ。トランスポートの殻だけ差し替え（protocol は無改変）。トランスポート抽象をここで本格的に切り出す。
- **観戦・観戦配信**: Durable Object のブロードキャストで観戦者へ。**秘匿境界の設計**（観戦者に開示前の手を見せるか）が新たに必要。
- **ロビー／マッチメイキング**: ルームキー手動共有から、相手探しの仕組みへ。
- **CPU 対戦**: ルールが固まってから。
- 各殻は独立に足せる。本段階は「二人がブラウザで、ルームキーで合流し、秘匿同時対戦を一局完走できる」ことに集中する。
