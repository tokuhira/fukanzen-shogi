# 不完全将棋 実装指示書 — バージョン管理 Step1（バイナリ版表示とリリースタグ連動）

> 対象実行者: Claude Code
> 前提: ワークスペース members = `engine`・`cli`・`tui`・`protocol`（TCP の殻は `tui/net.rs`）。GitHub Actions に、push トリガーで Intel Windows（msvc）向け TUI バイナリをビルドし Artifacts に上げる最小ワークフローが既にある。
> 関連文書: `不完全将棋_バージョン互換性管理_方針.md`（本 Step は方針の「第一段」に対応）。
> 性格: 配布の背骨を作る最小の一歩。配布バイナリに版を埋め込んで表示し、Git のリリースタグとバイナリ版を一致させ、タグ発火で GitHub Releases にバイナリを添付する。**ルール版・プロトコル版の分離管理はまだ行わない**（それは第二段以降。本 Step は単一の配布バイナリ版で足りる）。

---

## 0. 目的と範囲

- **作るもの**:
  1. ワークスペースの version を一元管理する（`workspace.package`）。
  2. バイナリに version を埋め込み、利用者が確認できるよう表示する（`--version` と TUI 内表示）。
  3. リリースタグとバイナリ版を一致させる仕組み（Cargo.toml を正本とし、タグとの不一致をワークフローで検出）。
  4. 既存ワークフローを拡張し、タグ push 時に GitHub Releases を作成してバイナリを添付する（日常の push 時の Artifacts ビルドは温存）。
- **作らないもの（本 Step の非ゴール）**: ルール版・プロトコル版の分離・自己申告・交渉（第二段）、タプルの体系管理（第三段）、複数ターゲットのマトリクス、データ（棋譜）の版管理。

---

## 1. ワークスペースの version 一元管理

配布バイナリ版を単一の番号で扱うため、各クレートの version をばらばらに持たず、ワークスペースで集約する。

- ルート `Cargo.toml` の `[workspace.package]` に `version` を置く（例: `version = "0.5.0"`）。
- 各メンバークレート（`engine`・`cli`・`tui`・`protocol`）の `Cargo.toml` で `version.workspace = true` として継承する。
- これにより、版を上げるときルート一箇所を直すだけで全クレートが揃う。

> 版番号は当面、ルール仕様の版に合わせて `0.5.0` から始めるのが分かりやすい（方針文書 §1 のとおり、本 Step ではルール版・プロトコル版を分離せず、配布バイナリ版という単一番号で扱う。ルール v0.5 に対応する初リリースを 0.5.0 とする、という素直な対応）。セマンティックバージョニング的な刻み方（互換を壊す核の変更でメジャー級、殻の改善でパッチ級）は第三段で本格化する。本 Step では単調に上げていければよい。

---

## 2. バイナリへの版埋め込みと表示

Rust では、コンパイル時に環境変数 `CARGO_PKG_VERSION` がそのクレートの version として利用可能（`env!("CARGO_PKG_VERSION")`）。これをバイナリに取り込む。

- **`--version` 引数**: TUI バイナリを `--version`（および慣例として `-V`）で起動したら、版を標準出力に表示して終了する。形式は簡潔に（例: `fukanzen-shogi-tui 0.5.0`）。
  - 注意: 既存の対戦モード等のコマンドライン引数解釈と衝突しないこと。`--version` は最優先で処理し、版を出して即終了する（TUI の端末初期化より前に処理する＝非対話で版だけ取れる）。
- **TUI 内表示**: 対話起動時、画面のどこか（タイトル/ヘッダ、またはステータス）に版を常時表示する。利用者が対局相手と「お互い何版か」を口頭確認できるようにするのが目的。
- **対戦相手との突き合わせ（任意・軽量）**: 本 Step ではプロトコルへの版交渉組み込みは行わない（第二段）。ただし、対戦接続時に相手へ自分の版文字列を一度送って画面に併記する程度の「参考表示」は、軽量なら入れてよい（強制力のある互換性判定はしない。あくまで人間の目視確認の補助）。判断は実装者に委ねるが、入れる場合もプロトコルの正式な交渉とは区別すること。

> 版の表示は CLI バイナリにも同様に付けてよい（`cli` も `--version` 対応）。ただし本 Step の配布対象は TUI なので、TUI を優先する。

---

## 3. タグとバイナリ版の一致（Cargo.toml を正本に）

リリースタグ（例 `v0.5.0`）と、バイナリが名乗る版（`workspace.package` の version、例 `0.5.0`）を一致させる。手で二箇所を直すと必ずいつか忘れるため、**Cargo.toml の version を正本**とし、タグとの不一致を**ワークフローで検出して失敗させる**番人を置く。

- 運用: 版を上げるときは、まずルート `Cargo.toml` の `workspace.package.version` を更新してコミットし、その版に対応するタグ（`v` + 版、例 `v0.5.0`）を打つ。
- 番人: タグ発火のワークフロー（§4）の中で、タグ名から先頭の `v` を除いた文字列と、`Cargo.toml` の version が一致するかを検証する。不一致ならワークフローを**失敗させ**、リリースを作らない。これにより「タグと中身の版がずれたリリース」を防ぐ。
- タグ名の規約: `v<major>.<minor>.<patch>`（例 `v0.5.0`）。先頭 `v` 付き。

---

## 4. GitHub Actions の拡張（タグ発火でリリース添付）

既存の最小ワークフロー（push トリガー → Windows ビルド → Artifacts）を土台に、タグ発火のリリース経路を足す。**二系統**とする。

- **日常系統（既存、温存）**: 通常の push で Windows ビルドし Artifacts に上げる。日常の動作確認はこちらで従来どおり。
- **リリース系統（新規）**: タグ（`v*`）の push で発火し、(1) §3 のタグ⇔version 一致検証、(2) Windows TUI バイナリのリリースビルド、(3) GitHub Releases の作成とバイナリ添付、を行う。

### 4.1 実装方針

- 既存ワークフローに、`on:` のトリガーとして `push: tags: ['v*']` を加える（または、リリース専用のワークフローファイルを別に作る。学習上は別ファイルの方が二系統の区別が明確で読みやすい——実装者判断でよい）。
- リリースジョブのステップ概略:
  1. チェックアウト。
  2. Rust セットアップ。
  3. **タグ⇔version 一致検証**（§3 の番人）。不一致なら失敗。
  4. TUI バイナリのリリースビルド（`-p <TUI_PKG> --bin <TUI_BIN> --target x86_64-pc-windows-msvc`）。
  5. GitHub Releases の作成と `.exe` の添付。リリースの作成は、GitHub が用意する権限（`GITHUB_TOKEN`）と定番のリリースアクション、または `gh` CLI を用いる。リリース名・タグ名・添付ファイル名を明示する。
- 権限: リリース作成にはワークフローに `contents: write` 権限が要る（`permissions:` で明示）。
- 添付ファイル名: 版とターゲットが分かる名前にする（例: `fukanzen-shogi-tui-<version>-windows-x64.exe`）。利用者がどの版か一目で分かるように。

### 4.2 確認事項（決め打ちにしない）

- TUI のパッケージ名 `<TUI_PKG>`・バイナリ名 `<TUI_BIN>` は `tui/Cargo.toml` の実値を使う（前回同様）。
- リリースアクションのメジャーバージョンは、現時点の安定版を確認して指定する。

---

## 5. 完了基準

1. ルート `Cargo.toml` の `workspace.package.version` が版の正本となり、各クレートが継承している。版を上げるのが一箇所の編集で済む。
2. TUI バイナリが `--version` で版を表示し、対話起動時にも画面に版が見える。
3. タグ（`v0.5.0` 等）を打つと、リリース系統のワークフローが発火し、タグ⇔version の一致を検証したうえで、Windows TUI バイナリを添付した GitHub Release が作成される。
4. タグと `Cargo.toml` の version が食い違う場合、ワークフローが失敗し、リリースが作られない（番人が機能する）。
5. 日常の push では従来どおり Artifacts へのビルドが続く（二系統が両立）。
6. `engine`・`protocol` のロジックは無改変（version 表示の追加と Cargo.toml の版集約のみ。ルール・プロトコルの挙動は変えない）。

---

## 6. 動作確認

1. ルート `Cargo.toml` の version を `0.5.0` に設定し、`cargo build` が通ることを確認。
2. ローカルで `<TUI_BIN> --version` が `... 0.5.0` を表示することを確認。
3. タグ `v0.5.0` を push し、リリース系統のワークフローが緑になり、GitHub Releases に `.exe` が添付されることを確認。
4. （番人テスト）試しに `Cargo.toml` の version を `0.5.0` のまま、誤ったタグ `v0.5.1` を打って push し、ワークフローが一致検証で**失敗**することを確認（確認後、その誤タグは削除）。
5. 添付された `.exe` をダウンロードし、ARM Windows 上（x64 エミュレーション）で起動して版表示と対戦モードを確認（前回のビルド確認と同じ要領）。

## 7. 次の段（本 Step では作らない・見通し）

- **第二段（Web 手前）**: プロトコルにバージョン交渉を組み込む。接続時に (ルール版, プロトコル版) を名乗り合い、完全一致で互換性を確認してから対戦を始める。本 Step の「参考表示」を、強制力のある交渉へ格上げする。ここで初めて、配布バイナリ版とは別にルール版・プロトコル版を実行体が自己申告する形が要る。
- **第三段（Web 初版）**: (ルール版, プロトコル版) のタプルを体系的に管理。配布バイナリ版とタプルの対応表、セマンティックバージョニングの確立、完全一致規約の再評価。
- これらはいずれも本 Step の版管理基盤の上に積む。
