# 不完全将棋 実装指示書 — 通信秘匿対戦（家庭内 LAN・P2P・TUI 延長）

> 対象実行者: Claude Code
> 前提: v0.5 まで実装済み。ワークスペースは `engine/`（純粋ライブラリ）・`cli/`（機械可読の口）・`tui/`（ratatui 検証卓）の三クレート。エンジンの公開 API は `legal_actions`・`resolve`・`check_status` と直列化関数群。USI 着手・SFEN 局面、正準直列化（盤面＋持ち駒＋手数、SHA-256 前方互換）と千日手用の内容直列化（手数除く）が用意済み。
> 関連文書: `不完全将棋_ルール仕様_v0.5.md`、`不完全将棋_実装指示書_Phase2_TUI検証卓.md`。
> 性格: 二人の人間が、別々の PC（家庭内 LAN）で、互いの着手を知らないまま秘匿同時対戦できるようにする。設計済みの通信プロトコル（commit-reveal・盤面ハッシュ相互検証・Ack・中断救済・本人認証）を実装する。**エンジンの核には一切触れない。**

---

## 0. この段階の目的と範囲

一人で両者を操作する検証では、隠された一手の読み合い——相手の裏をかく戦略——の妥当性を検証できない。独立した意思を持つ対戦相手（別の部屋の人間）と、秘匿同時対戦することが目的。

- **本段階で作る**: 二台の PC を家庭内 LAN で P2P 接続し、commit-reveal による秘匿同時着手で対戦する機能。TUI の延長として、ターミナルで動く。通信プロトコル層は**ソケットから切り離した純粋論理**として実装し、五つの正しさをテストで守る。
- **本段階で作らない（非ゴール）**: 中継サーバ、NAT 越え／インターネット越え対戦（家庭内 LAN・直接 IP 接続に限る。将来の拡張余地）。Wasm・ブラウザ UI（TUI で行う。ブラウザは将来）。CPU 対戦。

> 設計哲学は不変: 「共通の核と交換可能な殻」。エンジンは I/O・乱数・ネットワーク・暗号を持たない純粋なまま。通信プロトコルも、**論理は純粋・ソケットは殻**の二層に分ける（後述 §3）。

---

## 1. クレート構成

通信プロトコルの**純粋論理**と、それを運ぶ**ソケットの殻**を分離する。

- **新規 `protocol/`（ライブラリクレート）** — commit-reveal・ハッシュ照合・状態機械・中断救済・本人認証の**純粋論理**。I/O・ソケット・乱数を**直接持たない**（乱数=ノンス生成は呼び出し側から注入するか、トレイト境界で抽象化する。ハッシュ計算はここで行う＝SHA-256）。`engine` に依存する。**この層が五つの性質のテスト対象。**
- **新規 `net/`（または `tui` 内のモジュール）** — TCP ソケットの確立・送受信という**殻**。`protocol` の純粋論理を運ぶだけ。接続待ち受け／接続、メッセージのフレーミング（長さ前置等）、`protocol` の状態機械を駆動する。
- **`tui/`（既存、拡張）** — 対戦モードを追加。秘匿入力（自分の手だけを入力し、相手の手は reveal まで見えない）と、接続の確立（相手の IP 指定／待ち受け）。
- **`engine`・`cli`（不変）** — 触らない。

```
fukanzen-shogi/
├─ engine/        # 不変
├─ cli/           # 不変
├─ tui/           # 拡張: 対戦モード・秘匿入力・接続UI
├─ protocol/      # 新規: 純粋なプロトコル論理（テスト対象）
└─ net/           # 新規: TCP ソケットの殻（または tui 内モジュール）
```

> 乱数（ノンス生成）と時刻は殻側に置き、`protocol` には値として渡す。これにより `protocol` は決定的にテストできる（同じノンスを与えれば同じ commit になる）。

## 2. プロトコルの全体像（一ターンの状態機械）

審判はいない。二人のプレイヤーが対等に、互いを検証し合う。一ターンは次の状態を遷移する。両者対称。

1. **着手選択** — 各プレイヤーが現局面で自分の合法着手を一つ選ぶ（`legal_actions` を引く。TUI で秘匿入力）。
2. **commit 送受** — 自分の着手とノンス（乱数）からコミットメントを作り、相手へ送る。同時に相手の commit を受信する。**両者の commit が揃うまで、次へ進まない**（順序の正しさ）。
3. **reveal 送受** — 自分の着手・ノンス・**着手開始時点の盤面ハッシュ**を相手へ送る。同時に相手のそれを受信する。
4. **検証** — 受信した相手の (着手, ノンス) が、先に受け取った commit と一致するか検証（拘束性）。さらに相手が添えた盤面ハッシュが自分の現局面ハッシュと一致するか検証（相互検証）。いずれか不一致なら対局中断（§5）。
5. **解決** — 両者の着手が確定したので、各自手元で `resolve(現局面, 先手着手, 後手着手)` を呼び、次局面を得る（両者同じ入力なので決定的に同じ次局面になる）。
6. **Ack 送受** — 「reveal を受け取った」ことを相手へ通知し、相手の Ack を待つ。両者の Ack が揃ってから次局面へ進む（進行同期）。
7. 次ターンへ。

> commit は「着手とノンスのハッシュ」。reveal で (着手, ノンス) を開示し、受け手がハッシュし直して commit と照合する。ノンスは hiding（commit から手を推測されない）と binding（後から別の手で開けない）の両方に必要。

---

## 3. 五つの正しさ（純粋論理として実装し、テストで守る）

通信境界の正しさは目視で守れない。`protocol` クレートを**ソケットなしでテストできる純粋論理**として実装し、次の五性質を単体テストで守る（**実通信は伴わない**——関数呼び出しとして状態機械を駆動して検証する）。

1. **拘束性（binding）** — commit した着手を、後から別の着手に対応する (着手, ノンス) で開けない。テスト: commit を作り、異なる着手で reveal の検証を試みると**失敗**する。正しい (着手, ノンス) なら**成功**する。
2. **秘匿性（hiding）** — commit 単体から元の着手を復元できない。テスト: 同じ着手でもノンスが異なれば commit が異なること、commit から着手を逆算する手段がないこと（少なくとも、異なる着手が同じ commit を生まないことと、ノンス空間が推測困難なこと）を確認。
3. **順序の正しさ** — 両者の commit が揃う前に reveal を受理しない。テスト: 状態機械に対し、commit 完了前に reveal を投入すると**拒否／エラー**になることを確認。これが破れると後出しが可能になる。
4. **盤面ハッシュ相互検証** — 両者の局面認識が一致していることを毎ターン確認。テスト: 同一局面なら両者のハッシュが一致し、局面が食い違えば不一致が**検出**される。正準直列化（盤面＋持ち駒＋手数）に対する SHA-256 を用いる。
5. **中断救済** — 切断後、ハッシュ照合で正しい再開点を見つけ、棋譜から局面を復元して再開できる。テスト: ある局面で中断し、再接続時に相手の申告ハッシュを自分の棋譜（の各局面ハッシュ）と照合して、一致すればその地点から再開、完全不一致ならリジェクト。

加えて、設計上の同期と異常系を実装・テストする:

- **Ack による進行同期** — 両者が reveal を受け取ったことを Ack で確認してから次局面へ進む。これにより、切断時に両者が必ず同じ局面で止まる（一手ズレを防ぐ）。テスト: 状態機械が Ack 往復の完了をもって次ターンへ遷移すること。
- **二人の将軍問題の実務的打ち切り** — Ack の Ack…は理論上無限なので、有限回で打ち切り、残るごく稀なズレは中断救済（ハッシュ照合）で吸収する。完全合意の理論的不可能性を踏まえ、「予防（Ack）でズレをほぼ無くし、漏れは救済で拾う」二段構え。テスト: Ack 欠落時に救済へ落ちること。
- **ハッシュ不一致での即終了** — reveal 時の盤面ハッシュが食い違えば、自分のバグ・相手のバグ・改竄のいずれかなので、対局を中断する。テスト: 不一致を注入すると中断状態へ遷移すること。
- **本人認証（再接続）** — 対局前に共有秘密の**ハッシュ**を交換し、再接続時に秘密の本体を提示してハッシュ照合する（commit-reveal と同型の漏洩耐性）。テスト: 正しい秘密でのみ再開が受理され、誤った秘密はリジェクトされること。

## 4. 棋譜と局面復元（中断救済の土台）

- 各プレイヤーは対局開始からの全 Ply 履歴（棋譜）をローカルに保持する（既存の Kifu）。棋譜＋初期局面から任意局面を決定的に再現できる（既存の replay）。
- 中断救済は、相手の申告ハッシュを、自分の棋譜から再現される各局面の**正準直列化ハッシュ**と照合し、一致する局面を再開点とする。一致が無ければリジェクト。
- 盤面ハッシュの対象は**盤面＋双方の持ち駒＋手数**（既存の正準直列化）。千日手検出用の内容直列化（手数除く）とは別物——中断救済では手数を含む位置ハッシュを使い、局面を時間軸上の一点として一意に指す。

---

## 5. 異常系の扱い（プロトコルの一部として明示）

審判がいないので、異常系の解釈を両者で一致させておく。

- **ハッシュ不一致** → 即終了（続行不能）。どちらのバグか改竄かは問わず中断。
- **再開申告のハッシュが自分の棋譜のどの局面とも一致しない** → リジェクト。
- **相手の切断** → 待機し、再接続を受け付ける。再接続時はハッシュ照合（§4）＋本人認証（§3 共有秘密）で正当性を確認してから再開。
- **reveal の検証失敗（commit と不一致）** → 相手の不正（後出し）として中断。
- 一手ズレは Ack で原則防ぐ（§3）。Ack の漏れによる稀なズレのみ、ハッシュ照合の救済で吸収。差分送付による一手ズレ救済は設けない（Ack で代替済み）。

## 6. TUI 側（対戦モードの操作）

既存の ratatui 検証卓に、対戦モードを追加する。

- **接続確立**: 家庭内 LAN 前提。一方が待ち受け（ポート指定）、他方が相手の IP:ポートへ接続する、というシンプルな P2P。中継サーバ・NAT 越えは扱わない。接続前に、対局前の共有秘密（本人認証用）のハッシュ交換も行う。
- **先後（陣営）の取り決め**: 手番の優劣は無いが、盤の向きと駒の所有のため二陣営を区別する必要がある。接続時にどちらがどちらの陣営かを決める（待ち受け側＝先手、等の単純な規約でよい）。
- **秘匿入力**: 自分の着手だけを入力する。相手の着手は reveal まで画面に出さない（検証卓の「両方見える」モードとは別。対戦モードでは秘匿が本質）。自分の合法手ハイライト（検証卓の目玉機能）はそのまま使える。
- **解決結果の表示**: reveal 後、両者の着手と解決結果（取得・相討ち・戦国無双・両玉スワップ・玉の死等）をテキストで表示（検証卓と同じ表示を流用）。
- **状態表示**: 現在のプロトコル状態（相手の commit 待ち／reveal 待ち／Ack 待ち／解決済み）と、接続状態（接続中／切断・再接続待ち）を常に示す。
- **中断・再開**: 切断時の待機と再接続のフローを操作できる。

> 凝った視覚表現は不要（検証卓の方針を踏襲）。対戦モードで新たに本質的なのは「秘匿」と「接続状態の可視化」。

## 7. 完了基準（Definition of Done）

1. `protocol/` クレートが、ソケットなしで駆動できる純粋なプロトコル論理として存在する。`engine` に依存し、I/O・ソケットを直接持たない（乱数・時刻は注入）。
2. §3 の五つの性質（拘束性・秘匿性・順序の正しさ・盤面ハッシュ相互検証・中断救済）が、**実通信を伴わない単体テスト**として実装され、全て通る。加えて Ack 同期・ハッシュ不一致終了・本人認証のテストがある。
3. 二台の PC（または同一マシンの二プロセス）を家庭内 LAN（または localhost）で P2P 接続し、二人が秘匿同時着手で一局を最後まで（確定的詰み・玉の死・両玉スワップ引き分け・投了）対戦できる。
4. 一方を意図的に切断し、再接続してハッシュ照合＋本人認証で正しい再開点から続行できる。
5. TUI 対戦モードで、自分の手のみ入力でき、相手の手は reveal まで秘匿される。プロトコル状態と接続状態が可視化される。
6. `engine`・`cli` は無改変。`tui` の既存検証卓モードも従来どおり動く。
7. 既存のエンジンテスト（v0.5 まで）が引き続き全て通る。

## 8. 進め方の指針

1. **`protocol/` の純粋論理を最初に作り、§3 の五性質のテストを先に書く**（テストファースト）。ソケットを立てる前に、状態機械・commit/reveal・ハッシュ照合・中断救済を関数として完成させ、テストで固める。ここが本段階の心臓。
2. 乱数（ノンス）と時刻の注入境界を決める（トレイトか引数）。`protocol` を決定的にtestできる形に。
3. `net/`（TCP の殻）を実装し、`protocol` の状態機械を実通信で駆動する。フレーミング（メッセージ境界）を決める。まず localhost の二プロセスで疎通。
4. `tui/` に対戦モードを追加（接続確立・秘匿入力・状態表示）。
5. 切断・再接続・中断救済のフローを実装し、実際に切断して再開できることを確認。
6. 家庭内 LAN の実機二台で対戦。

横断的注意:

- **エンジンを変更しない。** プロトコルもエンジンも純粋論理として保ち、I/O は殻に押し出す。
- **審判なしの本質を守る**: P2P でも、ハッシュ相互検証と commit-reveal により、両者が互いを検証し合う。どちらか一方を信頼する設計にしない。
- 暗号強度（ノンスの質、ハッシュ関数）は家庭内・趣味の文脈に見合う実用水準でよいが、binding/hiding を実際に満たすこと（ノンスは予測困難な乱数、commit は暗号学的ハッシュ）。
- ルール変更による手戻りは、引き続きエンジンクレート側で吸収される。プロトコル・通信・TUI はエンジンの公開 API を叩く殻なので、ルールが動いても基本無傷。

## 9. 将来の拡張（本段階では作らない）

- 中継サーバ経由のインターネット越え対戦（NAT 越え）。家庭外の相手と指したくなったときに、`net/` の殻だけを差し替える形で追加できる（`protocol` は不変）。
- ブラウザ（Wasm）版。エンジンと `protocol` をそのまま Wasm 化し、`net/` を WebSocket/WebRTC の殻に差し替える。
- これらはいずれも `protocol` 純粋論理を再利用でき、本段階の投資が無駄にならない。
